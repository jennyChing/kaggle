         <p>
          If someone says "
          <em>
           just
           <strong>
            bin
           </strong>
           Deploy it
          </em>
          " they mean "deploy the application with the dependencies copied into the application's /bin folder, rather than running an MSI that installs the dependencies into the Global Assembly Cache (GAC)."
         </p>
         <p>
          You may not have administrative control over your Web Server and your host may not want you running installers when new stuff like ASP.NET MVC 3 and Razor comes out. You'll want to "
          <strong>
           bin deploy
          </strong>
          " these new technologies.
         </p>
         <p>
          Here's two ways. The simple way and The Awesome Way.
         </p>
         <h3>
          The Manual Way to BIN Deploy ASP.NET MVC 3 with Razor
         </h3>
         <p>
          On your development machine that has ASP.NET MVC 3 installed you have a C:\Program Files\Microsoft ASP.NET\ASP.NET MVC 3 folder and a C:\Program Files (x86)\Microsoft ASP.NET\ASP.NET Web Pages folder. In those folders are "Assemblies" folders.
         </p>
         <p>
          <a href="http://www.hanselman.com/blog/content/binary/WindowsLiveWriter/BIN.NETMVC3withRazortoaWindowsServerwith_E61F/Assemblies_2.png">
           <img alt="Windows Explorer showing the Assemblies that ASP.NET MVC and Web Pages needs for BIN Deployment" border="0" height="344" src="http://www.hanselman.com/blog/content/binary/WindowsLiveWriter/BIN.NETMVC3withRazortoaWindowsServerwith_E61F/Assemblies_thumb.png" style="border-right-width: 0px; margin: 0px; display: inline; border-top-width: 0px; border-bottom-width: 0px; border-left-width: 0px" title="Windows Explorer showing the Assemblies that ASP.NET MVC and Web Pages needs for BIN Deployment" width="640"/>
          </a>
         </p>
         <p>
          ASP.NET MVC itself references these additional assemblies.
         </p>
         <ul>
          <li>
           System.Web.Mvc (well, we ARE this assembly, actually)
          </li>
          <li>
           Microsoft.Web.Infrastructure
          </li>
          <li>
           System.Web.Razor
          </li>
          <li>
           System.Web.WebPages
          </li>
          <li>
           System.Web.WebPages.Razor
          </li>
         </ul>
         <p>
          And the default Web.config for projects also expects:
         </p>
         <ul>
          <li>
           System.Web.Helpers
          </li>
         </ul>
         <p>
          After you deploy your ASP.NET MVC 3 Razor application, you'll need to manually copy these five assemblies to the \bin folder of your deployed application.
         </p>
         <blockquote>
          <p>
           <em>
            <strong>
             NOTE:
            </strong>
            It's possible to just reference these assemblies directly from your application, then click properties on each one and set copyLocal=true. Then they'd get automatically copied to the bin folder. However, I have a problem with that philosophically. Your app doesn't need a reference to these assemblies. It's assemblies your app depends on. A depends on B that depends on C. Why should A manually set a dependency on C just to get better deployment? More on this, well, now.
           </em>
          </p>
         </blockquote>
         <p>
          Now, you likely remember that "
          <a href="http://www.hanselman.com/blog/WebDeploymentMadeAwesomeIfYoureUsingXCopyYoureDoingItWrong.aspx">
           If you're using XCOPY for deployment, you're doing it wrong
          </a>
          " so you're likely wondering how to make this deployment process less manual and more awesome.
         </p>
         <h3>
          The Alpha-Geek Show-Off Fancy-Pants Future-Proofed Way to BIN Deploy ASP.NET MVC 3 with Razor
         </h3>
         <p>
          I created a custom Windows 7 Virtual Machine that had only IIS7 and .NET 4 installed, plus
          <a href="http://www.iis.net/download/WebDeploy">
           <strong>
            Web Deploy
           </strong>
          </a>
          , and nothing else in order to test this scenario. This machine has never seen ASP.NET MVC 3 or Razor and we're not installing anything on it. It's our pretend "shared host."
         </p>
         <p>
          <a href="http://www.hanselman.com/blog/content/binary/WindowsLiveWriter/BIN.NETMVC3withRazortoaWindowsServerwith_E61F/ASPNETDeployment%20-%20Windows%20Virtual%20PC_2.png">
           <img alt="My custom Virtual Machine without ASP.NET MVC installed" border="0" height="389" src="http://www.hanselman.com/blog/content/binary/WindowsLiveWriter/BIN.NETMVC3withRazortoaWindowsServerwith_E61F/ASPNETDeployment%20-%20Windows%20Virtual%20PC_thumb.png" style="border-right-width: 0px; margin: 0px; display: inline; border-top-width: 0px; border-bottom-width: 0px; border-left-width: 0px" title="My custom Virtual Machine without ASP.NET MVC installed" width="458"/>
          </a>
         </p>
         <p>
          In my ASP.NET MVC Application, I'm going to create a folder called "_bin_deployableAssemblies."
          <strike>
           Yes, that's a underscore in the front. IIS will not serve from folders whose first character is an underscore. You can thanks FrontPage for that feature. In fact, this guarantees that even if you accidentally copy this folder up (don't) it won't be served.
          </strike>
          <em>
           Apparently I got bad info on this from the IIS team. I'm looking into this statement as it's incorrect.
          </em>
         </p>
         <ul>
          <li>
           Create a _bin_deployableAssemblies folder.
          </li>
          <li>
           Copy the assemblies we want bin deployed into this _bin_deployableAssemblies folder.
          </li>
          <li>
           Select them all, right click and hit properties. Set their Build Action to "None."
          </li>
          <li>
           You are expected to check these into source control as well.
          </li>
         </ul>
         <p>
          Here's my Solution so far:
         </p>
         <p>
          <a href="http://www.hanselman.com/blog/content/binary/WindowsLiveWriter/BIN.NETMVC3withRazortoaWindowsServerwith_E61F/image_2.png">
           <img alt="My Solution Folder View" border="0" height="401" src="http://www.hanselman.com/blog/content/binary/WindowsLiveWriter/BIN.NETMVC3withRazortoaWindowsServerwith_E61F/image_thumb.png" style="border-right-width: 0px; margin: 0px; display: inline; border-top-width: 0px; border-bottom-width: 0px; border-left-width: 0px" title="My Solution Folder View" width="286"/>
          </a>
         </p>
         <p>
          Here comes the secret awesome sauce.
         </p>
         <blockquote>
          <p>
           <strong>
            <font color="#ff0000">
             <em>
              NOTE: You're reading this on some random dude's blog, so lower your expectation of support now. No, lower. Bit lower. There. Yes, that's zero support. If this accidentally formats your harddrive or deletes your source code, kick yourself hard so you'll wake up to the previous level of Inception, specifically the one where the skinny kid from Third Rock From The Sun is walking on the ceiling. Be careful. Make backups.
             </em>
            </font>
           </strong>
          </p>
         </blockquote>
         <p>
          Get ready, because we are going into the belly of the beast. This is for VS2010, to be clear. Also, your Program Files folder will vary depending on your x86/x64 version of Windows.
         </p>
         <ul>
          <li>
           Go into
           <strong>
            C:\Program Files (x86)\MSBuild\Microsoft\VisualStudio\v10.0\WebApplications
           </strong>
           and make a copy of this folder. Put it somewhere safe.
          </li>
          <li>
           Go into
           <strong>
            C:\Program Files (x86)\MSBuild\Microsoft\VisualStudio\v10.0\Web
           </strong>
           and make a copy of this folder. Put it somewhere safe.
          </li>
          <li>
           Download this file and unzip it somewhere:
           <a href="http://www.hanselman.com/blog/content/binary/FancyAndTotallyUnsupportedMSBuildFilesForBinDeployableAssemblies.zip" title="http://www.hanselman.com/blog/content/binary/FancyAndTotallyUnsupportedMSBuildFilesForBinDeployableAssemblies.zip">
            <strong>
             FancyAndTotallyUnsupportedMSBuildFilesForBinDeployableAssemblies.zip
            </strong>
           </a>
          </li>
          <li>
           Inside is:
           <ul>
            <li>
             "C:\Program Files (x86)\MSBuild\Microsoft\VisualStudio\v10.0
             <strong>
              \WebApplications\Microsoft.WebApplication.targets"
             </strong>
            </li>
            <li>
             "C:\Program Files (x86)\MSBuild\Microsoft\VisualStudio\v10.0
             <strong>
              \Web\Microsoft.Web.Publishing.OnlyFilesToRunTheApp.targets
             </strong>
             "
            </li>
           </ul>
          </li>
          <li>
           Copy these new files OVER your existing ones (remember, the ones you backed up?)
          </li>
         </ul>
         <p>
          Now when you build your Web Application, all the files that are in _bin_deployableAssemblies will automatically get copied into your /bin folder.
         </p>
         <p>
          This is better than a simple Post Build Step Batch File because it works specifically with publishing. Now, you can right click on your Web Project and "Publish..." an the bin folder contain your project's build output PLUS the _bin_deployableAssemblies folder.
         </p>
         <p>
          <img alt="Build Deployment Package" border="0" height="199" src="http://www.hanselman.com/blog/content/binary/WindowsLiveWriter/BIN.NETMVC3withRazortoaWindowsServerwith_E61F/image_7.png" style="border-right-width: 0px; margin: 0px; display: inline; border-top-width: 0px; border-bottom-width: 0px; border-left-width: 0px" title="Build Deployment Package" width="414"/>
         </p>
         <p>
          I could publish directly to my Shared Host, or I can Build Deployment Package and get a ZIP file with all the things my app needs.
         </p>
         <p>
          For me, as I have a VM (remember that I put
          <a href="http://www.iis.net/download/WebDeploy">
           <strong>
            Web Deploy
           </strong>
          </a>
          on it?) and from IIS Manager just "Import Application." Some hosters have a control panel for this, but
          <strong>
           you'll likely use Web Deploy or Publish via FTP.
          </strong>
         </p>
         <p>
          <a href="http://www.hanselman.com/blog/content/binary/WindowsLiveWriter/BIN.NETMVC3withRazortoaWindowsServerwith_E61F/ASPNETDeployment%20-%20Windows%20Virtual%20PC%20(3)_2.png">
           <img alt="Import Application from inside IIS7 with WebDeploy" border="0" height="467" src="http://www.hanselman.com/blog/content/binary/WindowsLiveWriter/BIN.NETMVC3withRazortoaWindowsServerwith_E61F/ASPNETDeployment%20-%20Windows%20Virtual%20PC%20(3)_thumb.png" style="border-right-width: 0px; margin: 0px; display: inline; border-top-width: 0px; border-bottom-width: 0px; border-left-width: 0px" title="Import Application from inside IIS7 with WebDeploy" width="550"/>
          </a>
         </p>
         <p>
          This is a nice, clean, self-contained way to bin deploy an ASP.NET MVC 3 application with Razor views to a host that doesn't have any of these bits installed.
         </p>

           </font>

It's a "future-proofed hack."
         </p>
