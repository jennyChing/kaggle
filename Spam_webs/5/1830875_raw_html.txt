      <h1 class="article_title">
       New-Window Links in a Standards-Compliant World Article
      </h1>
      <div class="contributor article_contributor">
       <figure class="contributor_avatar">
        <a href="http://www.sitepoint.com/author/kevin-yank/">
         <img alt="Kevin Yank" class="avatar avatar-96 wp-user-avatar wp-user-avatar-96 alignnone photo" height="96" src="http://dab1nmslvvntp.cloudfront.net/wp-content/uploads/2015/04/1429112558kevinyank-96x96.jpg" width="96"/>
        </a>
       </figure>
       <div class="contributor_details">
        <div class="contributor_name">

     <section class="article_body">
      <p>
       <strong>
        Much to the chagrin of Web designers everywhere, the HTML 4.0 Strict and XHTML 1.0 Strict recommendations of the W3C no longer include the
        <code>
         target
        </code>
        attribute of the
        <code>
         &lt;a&gt;
        </code>
        tag. The Transitional versions of the specifications still include it, but by definition, these specs are on the way out.
       </strong>
      </p>
      <p>
       Whatever your personal feelings on the practice, the most common application for this attribute  opening a link in a new browser window with
       <code>
        target="_blank"
       </code>
        is still useful on todays Internet. So if the standards say we shouldnt use it, how should we go about creating new-window links, while following the latest Web standards?
      </p>
      <h5>
       Idealism Applied
      </h5>
      <p>
       The Web standards are written with a lot of ideals in mind. In many cases, established practices go by the wayside in favour of the right thing to do. One of the ideals that is expressed by the removal of the
       <code>
        target
       </code>
       attribute from the Strict standards is that (X)HTML should only be concerned with the information thats displayed within a browser window.
      </p>
      <p>
       Consequently, as soon as we start talking about opening new browser windows, the idealistic notion is that we have exceeded the responsibilities of (X)HTML and entered the world of client-side scripting (i.e. JavaScript).
      </p>
      <p>
       But before we dive into the JavaScript code necessary to open a link in a new browser window, we still have a fundamental problem to solve. We still need a way to mark links that should be opened in a new window, and if we cant use the
       <code>
        target
       </code>
       attribute to do it, we need to find some other way  a way that fits with the ideals behind the latest (X)HTML standards.
      </p>
      <div class="vertical" id="adz">
       <script type="text/javascript">
        // &lt; ![CDATA[
GA_googleFillSlot("Articles_6_300x250");
// ]]&gt;
       </script>
      </div>
      <h5>
       A New Relationship
      </h5>
      <p>
       The HTML 4.0 specification took away the
       <code>
        target
       </code>
       attribute, but it added another attribute:
       <code>
        rel
       </code>
       . This attribute is
       <em>
        intended
       </em>
       to specify the relationship between the document that contains the link, and the target of the link. The specification defines a bunch of standard values for this attribute (e.g.
       <code>
        next
       </code>
       ,
       <code>
        previous
       </code>
       ,
       <code>
        chapter
       </code>
       ,
       <code>
        section
       </code>
       ), most of which have to do with relationships between small sections of a larger document. However, the spec leaves the developer free to use nonstandard values for site-specific purposes.
      </p>
      <p>
       Here at SitePoint, we have adopted a custom value for the
       <code>
        rel
       </code>
       attribute to mark links leading to other Websites. These are the very same links that we want to open in a new browser window. For these links, we set the
       <code>
        rel
       </code>
       attribute to
       <code>
        external
       </code>
       .
      </p>
      <p>
       <strong>
        Before:
       </strong>
      </p>
      <p>
       <code>
        &lt;a href="document.html"
        <strong>
         target="_blank"
        </strong>
        &gt;external link&lt;/a&gt;
       </code>
      </p>
      <p>
       <strong>
        After:
       </strong>
      </p>
      <p>
       <code>
        &lt;a href="document.html"
        <strong>
         rel="external"
        </strong>
        &gt;external link&lt;/a&gt;
       </code>
      </p>
      <p>
       So, now that we have our new-window links marked up in a standards-compliant way, we need to write the JavaScript that will cause them to actually open in a new window.
      </p>
      <h5>
       Deconstructing the Script
      </h5>
      <p>
       The job of our script, which will be run once the document is loaded, will be to find all the hyperlinks in the document and modify those that have our special (but standards-compliant!)
       <code>
        rel="external"
       </code>
       attribute, so that they open in a new window.
      </p>
      <p>
       Sound daunting? Dont worry  the codes not too bad. The first step is to make sure the browser is up to the task:
      </p>
      <p>
       <code>
        if (!document.getElementsByTagName) return;
       </code>
      </p>
      <p>
       <code>
        getElementsByTagName
       </code>
       is a handy method of the Document Object Model 1.0 (DOM1) standard supported by modern browsers. Since a few of the older browsers that are still hanging around, such as Netscape 4 and Internet Explorer 4, dont support DOM1, we need to weed them out by checking for the presence of this function. On those browsers, we simply return and let external links open in the same browser window  no great loss, in most cases.
      </p>
      <p>
       Next, we use the
       <code>
        getElementsByTagName
       </code>
       method to get a list of all the
       <code>
        &lt;a&gt;
       </code>
       tags in the document:
      </p>
      <p>
       <code>
        var anchors = document.getElementsByTagName("a");
       </code>
      </p>
      <div class="vertical" id="adz">
       <script type="text/javascript">
        // &lt; ![CDATA[
GA_googleFillSlot("Articles_6_300x250");
// ]]&gt;
       </script>
      </div>
      <p>
       <code>
        anchors
       </code>
       will contain a JavaScript array after the method does its job. Each element in the array will represent an
       <code>
        &lt;a&gt;
       </code>
       tag in the document.
      </p>
      <p>
       Now we need to go through all of the
       <code>
        &lt;a&gt;
       </code>
       tags we just found and modify those that represent new-window links. You can use a JavaScript
       <code>
        for
       </code>
       loop to work on each of the tags in turn:
      </p>
      <p>
       <code>
        for (var i=0; i &lt; anchors.length; i++) {
        <br/>
        var anchor = anchors[i];
       </code>
      </p>
      <p>
       For each element in the
       <code>
        anchors
       </code>
       array, we create a variable inside the loop called
       <code>
        anchor
       </code>
       to isolate the tag with which we are currently concerned.
      </p>
      <p>
       Now we need to check if
       <code>
        anchor
       </code>
       represents a new-window link. As Im sure you know, the
       <code>
        &lt;a&gt;
       </code>
       tag isnt just used to create hyperlinks. With a
       <code>
        name
       </code>
       attribute, you can create a target for links that go to a particular location in the page. Only
       <code>
        &lt;a&gt;
       </code>
       tags with an
       <code>
        href
       </code>
       attribute qualify as hyperlinks. Our code must therefore check that
       <code>
        anchor
       </code>
       has an
       <code>
        href
       </code>
       attribute as well as a
       <code>
        rel
       </code>
       attribute that is set to
       <code>
        external
       </code>
       .
      </p>
      <p>
       <code>
        if (anchor.getAttribute("href") &amp;&amp;
        <br/>
        anchor.getAttribute("rel") == "external")
       </code>
      </p>
      <p>
       Now that weve confirmed that were dealing with a new-window link, we can set its
       <code>
        target
       </code>
       attribute to
       <code>
        "_blank"
       </code>
       :
      </p>
      <p>
       <code>
        anchor.target = "_blank";
       </code>
      </p>
      <p>
       And that should do it! Read on for the complete code, and for a few more words on why this solution is acceptable
      </p>
      <h5>
       A Fine Line
      </h5>
      <p>
       As a bit of a perfectionist, my first impulse when I learned of this solution was to ask, Is there really a difference? And yes, I often talk to myself while doing research.
      </p>
      <p>
       I mean, if the
       <code>
        target
       </code>
       attribute of the
       <code>
        &lt;a&gt;
       </code>
       tag is being phased out, does it really make a difference whether were setting it with JavaScript instead of HTML? Sure, the page will validate against the HTML 4.0 Strict and XHTML 1.0 Strict Document Type Definitions, but arent we technically cheating?
      </p>
      <div class="vertical" id="adz">
       <script type="text/javascript">
        // &lt; ![CDATA[
GA_googleFillSlot("Articles_6_300x250");
// ]]&gt;
       </script>
      </div>
      <p>
       The answer is no. The Document Object Model (DOM), which governs the document objects and attributes that are available to JavaScript code, is a totally separate standard from (X)HTML. Also consider that the DOM 2.0 standard that was published in January 2003 (well after XHTML 1.0, let alone HTML 4.0) still includes this attribute. It seems clear that while this attribute is slated to be phased out of (X)HTML, it will be available to JavaScript through the DOM for the foreseeable future.
      </p>
      <p>
       A number of other standards-compliant new-window link scripts out there propose using
       <code>
        window.open()
       </code>
       to load the document in a new window. While this approach generally works well on the surface, its downfall is that most browsers do not correctly report the referring URL in the request for the new page, which can be a serious issue, especially in inter-site links.
      </p>
      <h5>
       The Complete Script
      </h5>
      <p>
       As promised, here is the complete script. Notice the last line, which assigns the
       <code>
        externalLinks
       </code>
       function to the windows
       <code>
        onload
       </code>
       event handler. This triggers the function when the document has finished loading.
      </p>
      <p>
       <code>
        function externalLinks() {
        <br/>
        if (!document.getElementsByTagName) return;
        <br/>
        var anchors = document.getElementsByTagName("a");
        <br/>
        for (var i=0; i&lt;anchors.length; i++) {
        <br/>
        var anchor = anchors[i];
        <br/>
        if (anchor.getAttribute("href") &amp;&amp;
        <br/>
        anchor.getAttribute("rel") == "external")
        <br/>
        anchor.target = "_blank";
        <br/>
        }
        <br/>
        }
        <br/>
        window.onload = externalLinks;
       </code>
      </p>
      <p>
       As this is the kind of script youll want to deploy across your entire site, you should copy this code into a separate file (e.g.
       <code>
        external.js
       </code>
       ), and then load it in every page on your site with the following code, which should appear in the
       <code>
        &lt;head&gt;
       </code>
       tag of each document:
      </p>
      <p>
       <code>
        &lt;script type="text/javascript" src="/external.js"&gt;
        <br/>
        &lt;/script&gt;
       </code>
      </p>
      <p>
       Problem solved!
      </p>
