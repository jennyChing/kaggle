      Basic tasks for new Linux users
     </h1>
     <p>
      <em>
       <span>
        Help! I got my Linux installed. Now what?
       </span>
      </em>
     </p>
    </div>

    <div class="dw-summary-columns" id="dw-summary-area">
     <div class="ibm-col-6-4">
      <p>
       If you're a Windows user, you may find your new Linux system to be a bit different
            from what you're used to. One of your first tasks will probably
            be installing additional software, including some of IBM's
            extensive software for Linux. In many cases, installing software on a
            Linux system requires you to log in, become the super user (or root) for some
            tasks, open a terminal or shell window, and mount a CD-ROM, DVD, or USB drive.
            This article guides you through these basic first-time user tasks and
            more.
      </p>

         <h2 id="h0">
          Introduction
         </h2>
         <p>
          If you're a Windows user who wants to try Linux, and you are new to
                    Linux, this tutorial is for you. We'll cover a few basic tasks that
                    may be different from what you are used to. We'll cover these
                    tasks:
         </p>

          <dd>
           When you really need to work at the command line

         <p>
          These instructions and examples focus on three popular distributions
                    and their default graphical desktops:
         </p>
         <ul class="ibm-bullet-list">
          <li>
           Fedora 14 using a GNOME desktop
          </li>
          <li>
           OpenSUSE 11.3 using a KDE 4 desktop
          </li>
          <li>
           Ubuntu 10.10 using a GNOME desktop
          </li>
         </ul>
         <p>
          Other distributions, desktops, and window managers are available, and
                    we encourage you to explore further on your own. Even if some of the
                    information here is specific to a particular distribution and even a
                    particular version of a distribution, what you learn here should help
                    you navigate a strange distribution. Because this tutorial simply aims
                    to help you get started, you won't find information on more advanced
                    tasks like recompiling your kernel or installing software. We
                    recommend our no-charge
          <a href="http://www.ibm.com/developerworks/linux/library/l-lpic1-v3-map/">
           Linux (LPI) certification self-study guides
          </a>
          for deeper
                    information.
         </p>

         <h2 id="h1">
          Login
         </h2>
         <p>
          Once you have installed your Linux distribution and booted your system,
                    either you will be automatically
                    logged in or you will see a login screen. The next section shows how
                    to switch between these two startup methods and discusses the security
                    implications of automatic login. For this section, we'll assume you
                    are logging in. The three systems we consider in this tutorial                    
                     implement login a little
                    differently from each other, but all will prompt with the name of one
                    or more users. Once a user is selected, you need to enter a
                    password.
         </p>
         <h4 id="N100A4">
          The login screen
         </h4>
         <p>
          The appearance of the screen varies according to your Linux
                    distribution, and it will probably come as no surprise that you can
                    customize it further, although we won't cover that in this short
                    tutorial. A typical login screen for an Ubuntu 10.10 system is shown
                    in Figure 1, and a typical one for
                    Fedora 14 is shown in Figure 2.
                    These both illustrate the
                    default GNOME desktop used on these systems.
         </p>
         <h5 id="u-f1">
          Figure 1. Ubuntu
                        10.10 login screen
         </h5>
         <img alt="Ubuntu 10.10 login screen" src="u-login.jpg" width="550"/>
         <h5 id="f-f1">
          Figure 2. Fedora
                        14 login screen
         </h5>
         <img alt="Fedora 14 login screen" src="f-login.jpg" width="550"/>
         <p>
          A typical login screen for an OpenSUSE 11.3 system with the default
                    KDE 4 desktop is shown in Figure 3
                    or
         </p>
         <h5 id="s-f1">
          Figure 3. OpenSUSE
                        Linux 11.3 login screen
         </h5>
         <img alt="OpenSUSE Linux 11.3 login screen" src="s-login.jpg" width="550"/>
         <p>
          When you enter or select an id and press
          <strong>
           Enter
          </strong>
          , you
                    will be prompted for your password. If a password field is showing on
                    the login screen, as in the OpenSUSE example here, you can tab to the
                    password field and enter your password. If there is no entry field for
                    an id showing and you need to enter an id that is not in the list of
                    available ids, you will usually find an entry for something like
                    "other..." as shown in Figure 2.
                    Select that and you should see an entry field for the id. We'll talk
                    more about this in the section
          <a href="#h8">
           Becoming superuser (or
                        root)
          </a>
          .
         </p>
         <p>
          Login screens may have other items on them, including a clock, perhaps
                    the name of the system, and icons or named menus that allow you to
                    shut down or restart the system.
         </p>

                    dialog for accessibility options such as an on-screen keyboard or
                    larger font. An example from our Fedora system is shown in Figure 4. We used the option to make
                    text larger and easier to read for the login screen shown in
          <a href="#f-f1">
           Figure 2
          </a>

         </p>

         </h4>
         <p>

          <strong>

          </em>

                    them.

         <h2 id="h2">
          Navigation and settings
         </h2>
         <p>
          Let's spend a moment exploring the Linux panels on the desktop and then
                    look at navigation and an example of how you can customize your
                    desktop by switching to left-handed mouse usage.
         </p>
         <h4 id="N1015D">
          Desktop panels
         </h4>
         <p>
          Panels give you a starting place for interacting with your desktop and
                    provide information about your system. You will usually find one or
                    two panels on your desktop. Typically these will default to being on
                    the top, the bottom, or both the top and bottom edges of the screen.
                    You can move them to the left or right edges if you wish, typically by
                    right clicking and changing the properties.
         </p>
         <p>
          Different distributions and different desktops often differ in panel
                    layout, so expect differences here. Our Ubuntu GNOME panels are shown
                    in Figure 8. We've shown the ends
                    of the panels for clarity.
         </p>
         <h5 id="u-panel">
          Figure 8. GNOME
                        panel features on Ubuntu
         </h5>
         <img alt="GNOME panel features on Ubuntu" src="u-panel.gif" width="550"/>
         <ul class="ibm-bullet-list">
          <li>
           The left part of the top panel provides a launching place for
                        accessing programs, folders (directories), or system settings and
                        information.
          </li>
          <li>
           The right part of the top panel provides information such as time
                        and date, along with several quick-access buttons for functions
                        like setting volume control, opening chat windows, and logging
                        out.
          </li>
          <li>
           The left part of the bottom panel has a button to hide all windows
                        and show the desktop, along with buttons for active windows.
          </li>
          <li>
           The right part of the bottom panel contains a trash bin and four
                        rectangles that allow you to navigate between your virtual
                        desktops. Most Linux systems set up multiple distinct desktops, so
                        you can keep your email and web browsing on one desktop, while
                        doing program development on another and testing on yet another,
                        for example. You switch between them by clicking the appropriate
                        smaller square or by using a key combination. For GNOME,
           <strong>
            Ctrl-Alt-left arrow
           </strong>
           or
           <strong>
            Ctrl-Alt-right
                            arrow
           </strong>
           usually cycles through them in the same way
                        that
           <strong>
            Alt-tab
           </strong>
           will cycle through application
                        windows on a given desktop. For OpenSUSE,
           <strong>
            ctrl-F1
           </strong>
           through
           <strong>
            ctrl-F4
           </strong>
           will directly select desktops 1
                        through 4. When exploring, make sure you log in as a non-root user
                        to avoid accidents; such mistakes may be more serious when you
                        have unlimited authority.
          </li>
         </ul>
         <p>
          The panels for our Fedora system are shown in Figure 9. The System item in the upper left is
                    selected as we used the image from Figure 6 to create this image. Notice that we do
                    not have quick access to the logout function in the upper right, the
                    hide all windows function in the lower left, or the trash icon that we
                    saw in the lower right of the Ubuntu panels. Otherwise, the panel
                    layout is reasonably similar.
         </p>
         <h5 id="f-panel">
          Figure 9. GNOME
                        panel features on Fedora
         </h5>
         <img alt="GNOME panel features on Fedora" src="f-panel.gif" width="550"/>
         <p>
          The OpenSUSE system has a panel across the bottom only, as shown in
                        Figure 10. Access to programs
                    as well as folders and system functions starts with the large button
                    at the far left, which we will refer to as the
          <strong>
           Start
          </strong>
          button. Also on the left are quick-access buttons to a browser and
                    desktop navigator, followed by buttons for each of the five virtual
                    desktops. On the right end of the panel, you find a clock and several
                    convenience buttons similar to those at the right end of the upper
                    Ubuntu GNOME panel.
         </p>
         <h5 id="s-panel">
          Figure 10. KDE
                        4 panel features on OpenSUSE
         </h5>
         <img alt="KDE 4 panel features on OpenSUSE" src="s-panel.gif" width="550"/>
         <h3 id="N101B5">
          Navigating
         </h3>
         <p>
          The GNOME 2.3x desktop uses the cascading menus that have now become
                    familiar. Figure 11 illustrates
                    how to access the mouse settings from the Fedora System icon.
                    Different distributions may arrange these menus differently. For
                    example, you will find the mouse settings on Ubuntu in the same
                    location, but if you are looking for the Desktop Effects preferences,
                    which you find under
          <strong>
           System-&gt;Preferences-&gt;DesktopEffects
          </strong>
          on our Fedora system, you find it as the
          <strong>
           VisualEffects
          </strong>
          tab under
          <strong>
           System-&gt;Preferences-&gt;Appearance
          </strong>
          . Exploring
                    graphical applications is often like turning over different rocks to
                    see what is hiding underneath them.
         </p>
         <h5 id="f-menu1">
          Figure 11. Accessing mouse settings in Fedora
         </h5>
         <img alt="Accessing mouse settings in Fedora" src="f-menu1.gif" width="326"/>
         <p>
          In contrast, the KDE 4 desktop uses a different metaphor for the
          <strong>
           Start
          </strong>
          menu. Menu panels replace each other, and
                    you navigate by clicking on items in the menu or by mousing over the
                    icons at the bottom of the menu. Figure 12 illustrates the
          <strong>
           Favorites
          </strong>
          and
          <strong>
           Applications
          </strong>
          menus.
         </p>
         <h5 id="s-start1">
          Figure 12. Changing KDE 4 menus by mousing over
                        icons
         </h5>
         <img alt="Changing KDE 4 menus by mousing over icons" src="s-start1.gif" width="550"/>
         <p>
          When submenus are selected, such as from
          <strong>
           Start-&gt;Applications-&gt;Utilities
          </strong>
          , a back
                    button opens along the left of the menu so you can return to the
                    previous menu level. Figure 12
                    illustrates this.
         </p>
         <h5 id="s-start2">
          Figure 13. KDE
                        4 menu back button
         </h5>
         <img alt="KDE 4 menu back button" src="s-start2.gif" width="265"/>
         <p>
          Some
          <strong>
           Start
          </strong>
          menu items on OpenSUSE open a dialog box,
                    possibly containing further selections. An example is the
          <strong>
           Start-&gt;Applications-&gt;ConfigureDesktop
          </strong>
          menu, which opens a window like that in Figure 14, where we show the hover help for the
          <strong>
           Keyboard&amp;Mouse
          </strong>
          settings menu
                    item.
         </p>
         <h5 id="s-menu1">
          Figure 14. KDE
                        4 Configure Desktop menu
         </h5>
         <img alt="KDE 4 Configure Desktop menu" src="s-menu1.gif" width="364"/>
         <h3 id="N10226">
          Switching to left-handed mouse
                    usage
         </h3>
         <p>
          A right-handed user is generally assumed, but you can change your
                    mouse configuration for left-handed use, along with many other desktop
                    settings. Refer back to Figure 11
                    or Figure 14 to navigate to the
                    mouse settings dialog.
         </p>
         <p>
          On an Ubuntu system, you should see a window similar to Figure 15 where you can change your
                    mouse settings. In addition to basic left-handed or right-handed use,
                    there are several other settings you can change and a tab of
                    additional settings for accessibility. Settings take effect
                    immediately, so once you click the left-handed choice, your mouse is
                    set for left-handed use and you'll have to use the right mouse button
                    as button 1 to close the dialog. The dialog on a Fedora system is
                    similar.
         </p>
         <h5 id="u-mouse1">
          Figure 15. GNOME dialog to change mouse settings
         </h5>
         <img alt="GNOME dialog to change mouse settings" src="u-mouse1.gif" width="318"/>
         <p>
          On an OpenSUSE system with KDE, you should see a window similar to
                        Figure 16 where you can
                    change your mouse settings. Note the mouse image in the dialog. If you
                    switch to left-handed use, the right button in the mouse illustration
                    will be highlighted to indicate that it is your primary selection
                    button. Switch back to right-handed use, and the left button will be
                    highlighted. As with the Ubuntu dialog, there are several other
                    options that you can set. Unlike the Ubuntu dialog, the changes only
                    take effect when you click the
          <strong>
           Apply
          </strong>
          button.
         </p>
         <h5 id="s-mouse1">
          Figure 16. KDE
                        dialog to change mouse settings
         </h5>

         <h2 id="h8">
          Becoming superuser (or root)
         </h2>
         <p>
          For many tasks on Linux, you need
          <em>
           root
          </em>
          or
          <em>
           superuser
          </em>
          authority. The
          <em>
           root
          </em>
          user, sometimes called the
          <em>
           superuser
          </em>
          , is the user that is normally used for
                    administrative tasks like configuring the system or installing
                    software.
          <strong>
           Use
           <code>
            root
           </code>
           only when you
                        need to do administrative tasks; avoid using
           <code>
            root
           </code>
           for your normal work.
          </strong>
          The
                    root user can do anything, including accidentally destroying your
                    system, which is usually not a good thing. Normal users have fewer
                    privileges, and the system is much more protected from being
                    inadvertently damaged by normal users.
         </p>
         <p>
          Most administrative applications that have a graphical interface now
                    ask for the root password before allowing non-root users access to the
                    function. When you need to run commands from a terminal window as
                    root, this doesn't help.
         </p>
         <p>
          Your first thought may be to switch to another userid by logging out of
                    the current userid and logging in as the new userid. But what if you
                    only need to run a couple of quick commands as another user? Linux has
                    a solution for you: The
          <code>
           su
          </code>
          (substitute
                    user) and
          <code>
           sudo
          </code>
          commands allow you to
                    temporarily run one or more commands as another user. This is often
                    used for tasks that require root access. Indeed, if you connect in
                    remotely to a system using a terminal program such as
          <code>
           ssh
          </code>
          (or the very insecure
          <code>
           telnet
          </code>
          ), then many Linux distributions will
                    prevent you from signing in as root. This is a good security practice,
                    and we encourage you to not try to circumvent it. Rather, you should
                    sign in as a non-privileged user and then use the
          <code>
           su
          </code>
          or
          <code>
           sudo
          </code>
          command to do the work you need to do with root authority.
         </p>
         <p>
          To summarize, there are two main ways to run an arbitrary command with
                    root authority.
         </p>
         <ol>
          <li>
           Use the
           <code>
            su
           </code>
           command, usually with the
           <code>
            -
           </code>
           option to become root.
          </li>
          <li>
           Use the
           <code>
            sudo
           </code>
           command to execute a
                        single command with root authority.
          </li>
         </ol>
         <p>
          On systems such as Fedora or OpenSUSE, both methods are available,
                    although using
          <code>
           su
          </code>
          is perhaps more common.
                    On Debian-based systems such as Ubuntu, the security model prevents
                    root login, so you can neither log in as root nor use
          <code>
           su
          </code>
          to become root, so you must use
          <code>
           sudo
          </code>
          .
         </p>
         <h3 id="N103C1">
          Using su
         </h3>
         <p>
          Suppose you are logged in and looking at a terminal window, and you are
                    not the root user but need to run a command, such as
          <code>
           fdisk
          </code>
          , which requires root authority. You
                    switch to root using the
          <code>
           su
          </code>
          command alone,
                    or, more commonly, add the
          <code>
           -
          </code>
          option.
         </p>
         <p>
          The
          <code>
           su
          </code>
          command without the
          <code>
           -
          </code>
          option simply switches you to become
                    root, but does not change your environment variables, including your
                    path. The
          <code>
           -
          </code>
          option, which may also be typed
                    as
          <code>
           -l
          </code>
          or
          <code>
           -login
          </code>
          if you really like typing extra letters, allows the login startup
                    files for the substitute user to be read, thus setting things such as
                    the path, environment, and prompt to those of the target user. Listing 2 shows examples of these two
                    forms on our Fedora system. We've used the
          <code>
           pwd
          </code>
          (print working directory) command to
                    show the current working directory in each case. Note how the prompts
                    differ. If you'd like to understand more about how to customize your
                    own prompts or what makes these prompts appear as they do, check out
                    the "
          <a href="http://www.ibm.com/developerworks/linux/library/l-tip-prompt/">
           Prompt magic
          </a>
          " tip on developerWorks.
         </p>
         <h5 id="codesu">
          Listing 2. Switching to the root user
         </h5>
         <div class="codesection">
          <pre class="displaycode">
           [ian@echidna ~]$
           <strong>
            su
           </strong>
           Password:
[root@echidna ian]#
           <strong>
            pwd
           </strong>
           /home/ian
[root@echidna ian]#
           <strong>
            su -
           </strong>
           [root@echidna ~]#
           <strong>
            pwd
           </strong>
           /root
          </pre>
         </div>
         <p>
          You will notice, not surprisingly, that you had to provide a password
                    to switch to root. Once you have root authority, you can use
          <code>
           su
          </code>
          or
          <code>
           su-
          </code>
          to switch to another user or to switch to root with the login option.
                    If you want to switch to a non-root user, just add the id. As before,
                    you can use the
          <code>
           -
          </code>
          option or not, according
                    to your needs. For example:
         </p>
         <blockquote>
          <code>
           su - db2inst1
          </code>
         </blockquote>
         <p>
          To return to the previous id, press
          <code>
           Ctrl-d
          </code>
          or type
          <code>
           exit
          </code>
          and press
          <strong>
           Enter
          </strong>
          if you are using the bash shell, which is
                    the default on most Linux systems.
         </p>
         <p>
          Now that we've learned how to use
          <code>
           su
          </code>
          , let's
                    put it into practice with the
          <code>
           fdisk
          </code>
          command.
         </p>

         <h3 id="N1043F">
          Using sudo
         </h3>
         <p>
          Like the
          <code>
           su
          </code>
          command, the
          <code>
           sudo
          </code>
          command allows you to run commands
                    with the authority of another user. The commands that a given user or
                    class of users may execute are listed in the /etc/sudoers file. In
                    contrast to the
          <code>
           su
          </code>
          command, you do
          <strong>
           not
          </strong>
          need to know the password of the root, or
                    other user, although you will need to provide your own password. The
                    /etc/sudoers file is maintained by root and can be edited using the
          <code>
           visudo
          </code>
          command.
         </p>
         <p>
          Usually, if you are executing multiple
          <code>
           sudo
          </code>
          commands in rapid succession, you will not need to reenter your
                    password for each one. An alternative is to run
          <code>
           sudo
          </code>
          with the
          <code>
           -s
          </code>
          option, which runs a shell for you, from which you can run many
                    commands as the target user until you close the shell. Listing 4 illustrates both of
                    these methods.
         </p>

         <h2 id="h9">
          Using a GUI application as another user
         </h2>
         <p>
          You may have noticed in the discussion of the
          <code>
           su
          </code>
          command in the
          <a href="#h8">
           previous
                        section
          </a>
          that we ran only commands that displayed output in the
                    terminal window. Usually you will be able to run GUI commands too. For
                    example, some installation programs require you to have root authority
                    to install a program and have a GUI installer. If you find you cannot
                    start GUI applications as another user, then read on, as you may have
                    to take additional steps on some distributions in order to run GUI
                    applications as another user.
         </p>
         <p>
          <strong>
           Note:
          </strong>
          Recent distributions often let you have multiple
                    desktops open at once and switch between them using a key sequence
                    such as Ctrl-Alt-F7 or Ctrl-Alt-F8. Depending on what you need to do,
                    this may be another alternative. See the section on
          <a href="#h7">
           Logout
          </a>
          for details on this menu option.
         </p>
         <p>
          GUI applications on Linux use the X Window System, a client-server
                    system designed to allow multiple users to access a computer across a
                    network using windowed applications. An X display is known by a name
                    of the form
          <em>
           hostname:displaynumber.screennumber
          </em>
          . For Linux
                    running on a workstation such as a PC, there is typically only one
                    display with a single screen. In this case, the
          <em>
           displayname
          </em>
          may be, and usually is, omitted so the display is known as :0.0, or
                    sometimes just :0.
         </p>
         <p>
          The X Window System server needs to know the display and also whether
                    you are authorized to connect to the server. Authorization is most
                    commonly done using the MIT-MAGIC-COOKIE-1, which is a long random
                    string that is regenerated whenever the server is reset. So that
                    applications can pass this information to the X server, you will have
                    DISPLAY and XAUTHORITY variables set in your environment. The
                    XAUTHORITY variable will point to a file that is usually only able to
                    be read or written by the owning user as a security precaution. We
                    assume you are using a graphical login if you are reading this, so
                    your startup should have already set these for you. The example in
                        Listing 6 is from our
                    Ubuntu system and shows the values of the DISPLAY and XAUTHORITY
                    variables as well as the ownership for the file pointed to by the
                    XAUTHORITY variable.
         </p>
         <h5 id="u-disp-env1">
          Listing 6. DISPLAY and XAUTHORITY
         </h5>
         <div class="codesection">
          <pre class="displaycode">
           ian@pinguino:~$
           <strong>
            echo $DISPLAY
           </strong>
           :0.0
ian@pinguino:~$
           <strong>
            echo $XAUTHORITY
           </strong>
           /var/run/gdm/auth-for-ian-WoeKHn/database

         <h2 id="h4">
          Removable devices
         </h2>
         <p>
          On Linux and UNIX systems, all files are accessed as part of a
                    single large tree that is rooted at /. To access the files on a
                    CD-ROM, you need to mount the CD-ROM device at some mount point in the
                    file tree. On current distributions, this is usually automated for
                    you; you need only insert the disc and it will be recognized and
                    mounted. Once mounted, it is important to properly unmount the device
                    to avoid data loss.
         </p>

         <h3 id="N10613">
          Safely removing devices
         </h3>
         <p>
          Once a DVD or CD is mounted, you can use the files on the disk as they
                    are now part of your file system. While a CD-ROM is mounted, Linux
                    will lock the CD so that it cannot be ejected with the Eject button.
                    Of course, this doesn't stop you from unplugging an external CD or USB
                    drive, pulling a USB stick out of the USB socket, or ejecting a floppy
                    disk. To avoid potential data loss, you should always remove the
                    device safely by first
          <em>
           unmounting
          </em>
          it.
         </p>
         <p>
          You can unmount devices from the graphical desktop, or from the command
                    line. The
          <code>
           umount
          </code>
          command (note umount
                    without an 'n') unmounts a device, and the
          <code>
           eject
          </code>
          command first unmounts a device and
                    then tries to eject it, for example by opening a CD drawer.
                    Traditionally, mounting and unmounting devices required root
                    authority. Nowadays, it is common to allow
          <em>
           user mounting
          </em>
          , so
                    that a non-root user is able to mount and unmount devices. You may
                    have noticed back in Listing 11
                    the option
          <code>
           uid=1000
          </code>
          . The uid of 1000
                    corresponds to user ian on this system as shown in Listing 12.
         </p>
         <h5 id="s-uid-1000">
          Listing 12. Using the id command for the
current user

          If you use the graphical desktop tools to unmount the device, you may
                    only need to worry about ids if you have logged out and then back in
                    as a different user. So let's look at the graphical tools, then the
                    command line ones.
         </p>
         <p>
          On Ubuntu or Fedora, if you right click on the icon for the removable
                    media you will see a context menu similar to that in Figure 25. You will have an
                    option to either unmount or eject the device. Select the eject option
                    for a CD or DVD and the device will be unmounted, the icon will
                    disappear from your desktop, and your drawer will open. For a USB
                    drive, the option may be unmount rather than eject, and you can safely
                    unplug the device after it is unmounted. If you use an external hard
                    drive with multiple partitions, you need to make sure that all
                    partitions are unmounted before removing or powering off the
                    drive.
         </p>

          <code>

          </code>
          command shows the actual file or
                    directory (/media/openSUSE-DVD-x86_64.0702..001/boot in this case)

         </p>
         <p>
          Normally you would use this information to cleanly close the open
                    process or window using the filesystem. On some Linux systems there is
                    also an
          <code>
           -L
          </code>
          option to
          <code>
           umount
          </code>
          that allows a
          <em>
           lazy
          </em>
          unmount
                    where the filesystem is detached from the filesystem hierarchy
                    immediately, and all references to the filesystem are cleaned up when
                    the filesystem is no longer busy. On OpenSUSE 11.3 this option is not
                    available, so you need to clean up the filesystem references, either
                    by terminating the offending processes or by ensuring they are no
                    longer using affected resources, before unmounting.
         </p>

          We mentioned back in
          <a href="#s-umount1">
           Listing 14
          </a>
          that the OpenSUSE system does not
                    support command line use of
          <code>

          </code>
          for a
                    device that was mounted from the desktop, so we attacked the problem
                    by using root authority. There is another approach using the
          <code>
           halmount
          </code>
          command to unmount the device.
          <em>
           HAL
          </em>
          is a Hardware Abstraction Layer, which provides a
                    consistent application interface to various hardware devices. If you
                    use
          <code>
           halmount
          </code>
          , you will still be unable to
                    unmount a busy device. Listing 17 shows how to use
          <code>
           halmount
          </code>
          and then switch to
          <code>
           root
          </code>
          to forcibly terminate the process that
                    is keeping the disc busy. Note that killing the process in this way
                    may cause data loss, so we do not recommend it unless you are sure you
                    will not lose data. We then switch back to normal user mode and use
          <code>
           halmount
          </code>
          to unmount the filesystem that is
                    no longer busy. Finally we use
          <code>
           eject
          </code>
          to
                    open the drawer and eject the DVD.
         </p>
         <h5 id="s-hal-umount">
          Listing 17. Using halmount, kill, and eject
         </h5>
         <div class="codesection">
          <pre class="displaycode">
           ian@lyrebird:~&gt;
           <strong>
            halmount -u /dev/sr0
           </strong>
           /dev/sr0: org.freedesktop.Hal.Device.Volume.Busy: umount: /media/openSUSE-DVD-x86_64.0702
..001: device is busy.
ian@lyrebird:~&gt;
           <strong>
            lsof /dev/sr0
           </strong>
           COMMAND  PID USER   FD   TYPE DEVICE SIZE/OFF NODE NAME
bash    3824  ian  cwd    DIR   11,0     2048 2048 /media/openSUSE-DVD-x86_64.0702..001/
boot
ian@lyrebird:~&gt;
           <strong>
            su -
           </strong>
           Password:
lyrebird:~ # kill -9 3824
lyrebird:~ #
           <strong>
            ps -ef | grep 3824
           </strong>
           root      5542  5507  0 22:47 pts/2    00:00:00 grep 3824
lyrebird:~ #
           <strong>
            logout
           </strong>
           ian@lyrebird:~&gt;
           <strong>
            halmount -u /dev/sr0
           </strong>
           ian@lyrebird:~&gt;
           <strong>
            eject /dev/sr0
           </strong>
          </pre>
         </div>
         <p>
          We hope this brief introduction to removable media use on Linux will
                    help you enjoy your removable media files and avoid data loss.
         </p>

         <h2 id="h10">
          Adding users and groups
         </h2>
         <p>
          Sometimes you will need to define new users of your system and new
                    groups for those users. For example, you may need to define a user
                    called
          <strong>
           mqm
          </strong>
          and a group also called
          <strong>
           mqm
          </strong>
          (as well as another group called
          <strong>
           mqbrkrs
          </strong>
          when installing the embedded messaging
                    component in WebSphere Application Server), or you may need to create
                    users to administer databases.
         </p>
         <p>
          If you do not specify a user number, graphical tools will usually
                    assign the next available user number. For Fedora and Red Hat systems,
                    user numbers start at 500, so the user you created when you installed
                    your system is probably user 500. For OpenSUSE and Ubuntu, the numbers
                    start at 1000. If you use the same id on several systems as I do, you
                    will probably find it convenient to use the same id and group numbers
                    on each system too.
         </p>
         <p>
          For the purpose of this section, we will add a user called
          <code>
           testuser
          </code>
          with id 2000 and group 2000.
                    Usually you will define the group first and then define the users who
                    will use the group, so that's what we will do here. You can either use
                    the graphical tools for user administration or enter commands in a
                    terminal window. We'll give an overview of the graphical process here
                    using
          <a href="#h10a">
           OpenSUSE's system administration tools
          </a>
          . Then
                    we'll tell you where to find the corresponding
          <a href="#h10b">
           tools
                        on a Fedora or Ubuntu system
          </a>
          . Finally we'll give you the
          <a href="#h10c">
           commands
          </a>
          if you really want to do it from the
                    command line.
         </p>

         <h3 id="h10c">
          Adding users and groups
                    using the command line
         </h3>
         <p>
          You can add or change users and groups from the command line. These
                    tasks require root authority.
         </p>
         <p>
          Information on groups is stored as a flat file in /etc/group. You may
                    use the
          <code>
           groupadd
          </code>
          command to add a new
                    group. This is fairly simple. Adding a new user is a little more
                    complex as there are more parameters, and you will need the numerical
                    number of the user's group. Let's use the
          <code>
           groupadd
          </code>
          command to add our
          <code>
           testuser
          </code>
          group, with group id 2000, and
                    then use the
          <code>
           grep
          </code>
          command to search
                    /etc/group and verify the settings.
          <strong>
           Note
          </strong>
          : If you do
                    not provide a group id, the system will assign the next one that is
                    higher than any existing group id.
         </p>
         <div class="codesection">
          <pre class="displaycode">
           root@pinguino:~#
           <strong>
            groupadd -g 2000 testuser
           </strong>
           root@pinguino:~#
           <strong>
            grep testuser /etc/group
           </strong>
           testuser:x:2000:
          </pre>
         </div>
         <p>
          As you see, the
          <code>
           testuser
          </code>
          group is 2000. Now
                    let's use the
          <code>
           useradd
          </code>
          command to add the
          <code>
           testuser
          </code>
          user. The
          <code>
           -c
          </code>
          option allows us to specify a comment
                    that is usually a user's real name. The
          <code>
           -u
          </code>
          option allows us to specify the numerical id (2000) for the user. The
          <code>
           -d
          </code>
          option allows us to specify the home
                    directory for the user. The
          <code>
           -g
          </code>
          option
                    specifies the user's primary group. Here we use 2000, which is the
          <code>
           testuser
          </code>
          group we just created. The last
                    option we use is the
          <code>
           -G
          </code>
          option to specify
                    additional groups for this user. Here we can use the group name. In
                    this case, we'll add
          <code>
           testuser
          </code>
          to the group
          <code>
           ian
          </code>
          .
         </p>
         <p>
          Once you have added the user, you can use the
          <code>
           grep
          </code>
          command again, and you will see that
                    user
          <code>
           testuser
          </code>
          has been added to the
          <code>
           testuser
          </code>
          and
          <code>
           ian
          </code>
          groups. At this point you have created a new user, but the user does
                    not have a password and cannot log on to the system. Some users do not
                    need to log on, so that would be alright for those users. The root
                    user has the authority to set (or reset) passwords for other users. To
                    do this, you use the
          <code>
           passwd
          </code>
          command and
                    give the username as a parameter. You will be prompted for the new
                    password, and then you will be prompted to retype it for
                    verification.
         </p>
         <div class="codesection">
          <pre class="displaycode">
           root@pinguino:~#
           <strong>
            useradd -c"Test User" -u 2000 -d/home/testuser -g 2000 -G ian \
           </strong>
           &gt;
           <strong>
            testuser
           </strong>
           root@pinguino:~#
           <strong>
            grep testuser /etc/group
           </strong>
           ian:x:1000:testuser
testuser:x:2000:
root@pinguino:~#
           <strong>
            passwd testuser
           </strong>
           Enter new UNIX password:
Retype new UNIX password:
passwd: password updated successfully
          </pre>
         </div>
         <p>
          Finally, you may need to add users to an existing group. You can use
                    the
          <code>
           usermod
          </code>
          command to do this, but you
                    need the list of existing groups for the user as you will replace the
                    list of additional groups. It is often easier to edit /etc/group
                    directly. Make a backup copy first, just in case you make a mistake.
                    To add the editor user to both the ian and testuser groups, edit
                    /etc/group and update the lines for ian and testuser so they look as
                    follows:
         </p>
         <div class="codesection">
          <pre class="displaycode">
           ian:x:1000:testuser,editor
testuser:x:2000:editor
          </pre>
         </div>
         <p>
          You will find much of the user information is stored in /etc/passwd,
                    but you should not edit this file yourself. Use the
          <code>
           useradd
          </code>
          ,
          <code>
           usermod
          </code>
          , and
          <code>
           userdel
          </code>
          commands instead. If you are not a
                    full time system administrator, you will probably find it easier to do
                    occasional manipulation of users and groups through the graphical
                    interfaces.
         </p>
