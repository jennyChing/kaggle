       </a>

    <div class="row">
     <div class="small-12 medium-10 medium-centered large-8 columns leftPageContents" data-break="large">
      <div class="cms" itemprop="articleBody">
       <h4>
        It is becoming ever more important for web applications to have an API allowing others to connect and integrate third-party resources and apps to enhance the overall user experience of your product.
       </h4>
       <p>
        <img alt="Creating a PHP RESTful API in PHP" src="http://static.madebymagnitude.com/blog/323379408533dd5ff061234.93214959_d512eaf2.jpg" style="height:357px; width:726px"/>
       </p>
       <p>
        Last time we took a look at what a REST API was; the various HTTP methods, responses and headers that an API should be able to handle. Read our first part on
        <a href="http://www.madebymagnitude.com/blog/designing-a-rest-api-for-your-mobile-application/" target="_blank">
         Designing a REST API for your Mobile Application
        </a>
        .
       </p>
       <p>
        We briefly touched on how to create your own API, but with the demand for a successful, scalable and powerful API increasing, we wanted to re-visit where we left off. In this example we're going to look into the full power behind the Slim Framework. Though we've previously discussed other
        <a href="http://www.madebymagnitude.com/blog/creating-a-php-rest-routing-class-for-your-application/" target="_blank">
         PHP RESTful routing frameworks that you can use in your application
        </a>
        .
       </p>
       <h3>
        Why the Slim Framework?
       </h3>
       <p>
        For those that don't know much about Slim, we're quite big fans and we're really keen on developing our API's using it. The framework is only classed as a 'micro framework' but will relieve you of a massive overhead. Primarily, it's one hell of a powerful router. It's able to handle all the standard HTTP methods and you can even create your own. You can setup routes, define the parameters, catch any wildcards and then decide whether to redirect, halt or pass onto any middleware you've got running.
       </p>
       <h3>
        What Makes Slim so Powerful?
       </h3>
       <p>
        Slim is built on a middleware and hook architecture that lets you tweak every aspect of its core without actually having to dive into any code. The framework implements a version of the Rack protocol that allows any middleware you've setup to inspect, analyse, or modify the application environment, request, and response before and/or after the Slim application is invoked.
       </p>
       <p>
        Taken from the Slim documentation:
       </p>
       <p>
        "Think of a Slim application as the core of an onion. Each layer of the onion is middleware. When you invoke the Slim applications run() method, the outer-most middleware layer is invoked first. When ready, that middleware layer is responsible for optionally invoking the next middleware layer that it surrounds. This process steps deeper into the onion - through each middleware layer - until the core Slim application is invoked. This stepped process is possible because each middleware layer, and the Slim application itself, all implement a public call() method. When you add new middleware to a Slim application, the added middleware will become a new outer layer and surround the previous outer middleware layer (if available) or the Slim application itself."
       </p>
       <h3>
        Getting Your Own API up and Running
       </h3>
       <p>
        Honestly, Slim is super powerful and can really ease the volume of coding you actually have to do. We've put together code that'll cover setting up your own routing, handling errors, catching nested URLs, pattern matching, setting up conditions, using middleware and sending HTTP error statuses.
       </p>
       <pre class="language-php">
        <code>
         &lt;?php/** 
 * APILogWriter: Custom log writer for our application
 *
 * We must implement write(mixed $message, int $level)
*/
class APILogWriter {	public function write($message, $level = SlimLog::DEBUG) {		# Simple for now
		echo $level.': '.$message.'&lt;br /&gt;';	}}# Load the Slim framework
require 'vendor/autoload.php';# Fire up an app
$app = new SlimSlim(array(
		'mode' =&gt; 'development',
		'log.enabled' =&gt; true,
		'log.level' =&gt; SlimLog::DEBUG,
		'log.writer' =&gt; new APILogWriter()
	)
);/**
 * Setup a GET path for /foo
 */
$app-&gt;get('/users', function() use ($app) {	# We can render a view
    $app-&gt;render('users.php');});$app-&gt;get('/users/:id', function($id) use ($app) {	$profile = ModelUsers::getUserByID($id);	# Render the view, with some data
	$app-&gt;render('profile.php', array(
   			'profile' =&gt; $profile,
			'view' =&gt; true
		)
	);
});/**
 * We can have multiple parameters, as complex as the URL
 * patterns need to be.
 */
$app-&gt;get('/users/:id/:action', function($id, $action) use ($app) {	# Render the view using the action route
	$this-&gt;render('profile/'.$action.'.php');});/** 
 * We can have simple BNF regular expressions that allow us
 * to match repeated patterns.
 *
 * In this instance, this can match /this/is/a/url OR /this/url
 */
$app-&gt;get('/profile/:name+', function ($name) {	/**
	 * In this instance $name will now be an array
	 *
	 * $name = array('this', 'is', 'a', 'url')
	*/})-&gt;conditions(array('name' =&gt; '[a-z]{10,}'));/**
 * We can have optional parameters in the URL where you want to
 * have a partial match on a given pattern
 */
$app-&gt;get('/profile(/:year(/:month(/:day)))', function($year = 0, $month = 0, $day = 0) {	# Stop anything else happening if invalid
	if(!is_int($year)) {
		$app-&gt;stop();
	}	/**
	 * Search for users from the model that match the parameters.
	 * We have to set default values because they're optional
	 * so we can have /profile/2014 for all users created in 2014 or
	 * /profile/2014/03 for all users created in March 2014.
	*/
	$data = ModelUsers::filterByRange($year, $month, $day);})-&gt;name('profileDateFilter')-&gt;conditions(array('year' =&gt; '(19|20)dd'));/**
 *  Above we named the route we just created 'profileDateFilter' which allows
 * us to use the urlFor method to reverse the route. We are also using a constraint
 * on the accepted values for 'year'. It must prefix with 19xx or 20xx
 */
$output = $app-&gt;urlFor('profileDateFilter', array('year' =&gt; 2013, 'month' =&gt; 12, 'day' =&gt; 25));/**
 * We can have a single route respond to multiple
 * HTTP methods by using the map() function
 */
$app-&gt;map('/users/validate', function() {})-&gt;via('GET', 'POST');/**
 * Middleware allow us to nest the route down multiple
 * layers before we make a decision on what to do
 */
function middleOne(SlimRoute $route) {
	echo $route-&gt;getName();	
}function middleTwo(SlimRoute $route) {
	echo $route-&gt;getName();	
}# When we go to the URL, pass down each layer
$app-&gt;get('/middleware', 'middleOne', 'middleTwo', function() {	# Catch any errors
	try {        ModelUser::someGenericMethod();		/**
		 * This is a pre-built redirect method to take
		 * you to another URL
		*/
        $app-&gt;redirect('/middleware/success');    } catch(Exception $e) {		/**
		 * Show the error message view
		*/
        $app-&gt;flash('error', $e-&gt;getMessage());
        $app-&gt;redirect('/middleware/error');    }});/**
 * Slim can also handle error message and
 * send the appropriate HTTP responses
*/
$app-&gt;get('/denied', function() use ($app) {	# Create the data
	$errorData = array('error' =&gt; 'Permission Denied');	# Send a HTTP status of 403
    $app-&gt;render('error.php', $errorData, 403);	# Or we can send
	$app-&gt;halt(403, 'Permission Denied');});# Run the Slim application
$app-&gt;run();?&gt;
        </code>
       </pre>
       <p>
        You can use the Slim framework as a great starting point for any API. The documentation goes in depth on how you can extend the framework to handle multi-level middleware layers that allow you to nest and condition each of your routes.
       </p>
       <p>
        Next time we'll look into using the middleware layers, custom views, HTTP caching, ETag caching and Hooks.
       </p>
