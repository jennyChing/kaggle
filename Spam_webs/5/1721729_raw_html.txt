        <h1 class="p-name">
         Getting Acquainted with Initial
        </h1>
        <p class="time">
         <a href="https://css-tricks.com/author/geoffreygrahamgmail-com/">
          <img alt="" class="avatar avatar-200 photo" height="200" src="https://1.gravatar.com/avatar/ac0d6274a6cea04669e66a20ec0ea204?s=200&amp;d=mm&amp;r=pg" srcset="https://1.gravatar.com/avatar/ac0d6274a6cea04669e66a20ec0ea204?s=400&amp;d=mm&amp;r=pg 2x" width="200"/>
          <span class="p-author">
           Geoff Graham
          </span>
         </a>
         //
         <time class="dt-published" datetime="2015-05-29">
          May 29, 2015
         </time>
        </p>
       </header>
       <div class="module e-content">
        <div class="google-ad">
         <div class="bsarocks bsap_3469a2a501a9e18091036aa0c89f9dcb" id="bsap_1279518">
         </div>
        </div>
        <p>
         If someone walked up to me the other day and asked me what the difference between
         <code>
          inherit
         </code>
         and
         <code>
          initial
         </code>
         is, I may have replied:
        </p>
        <p>
         "What, there's a difference?"
        </p>
        <p>
         <span id="more-202614">
         </span>
        </p>
        <p>
         I've been writing CSS for more than ten years, but I've somehow escaped understanding what exactly
         <code>
          initial
         </code>
         is or does. Call it ignorance, laziness, or luck, but
         <code>
          inherit
         </code>
         has gotten me by and I never thought to look up when
         <code>
          initial
         </code>
         might be used instead. So, this post is gonna share some of the things I learned.
        </p>
        <h3>
         What
         <code>
          initial
         </code>
         means
        </h3>
        <p>
         First off, the spec helps us understand the difference between an initial
         <em>
          keyword
         </em>
         and an initial
         <em>
          value
         </em>
         .
        </p>
        <ul>
         <li>
          <strong>
           <a href="http://dev.w3.org/csswg/css-cascade/#initial">
            Initial keyword
           </a>
           :
          </strong>
          If the cascaded value of a property is the initial keyword, the propertys initial value becomes its specified value.
         </li>
         <li>
          <strong>
           <a href="http://dev.w3.org/csswg/css-cascade/#initial-value">
            Initial value
           </a>
           :
          </strong>
          Each property has an initial value, defined in the propertys definition table. If the property is not an inherited property, and the cascade does not result in a value, then the specified value of the property is its initial value.
         </li>
        </ul>
        <p>
         Umm, okay. I ran those definitions through Google Translate (joking!) and came out with this:
        </p>
        <blockquote>
         <p>
          The initial keyword is what is declared as the property where the initial value is the resulting output, as defined by the browser default.
         </p>
        </blockquote>
        <p>
         So, if the
         <code>
          initial
         </code>
         keyword is used here:
        </p>
        <pre rel="CSS">
         <code>
          .module {
  color: initial;
}
         </code>
        </pre>
        <p>
         ...then the
         <code>
          initial
         </code>
         value might return as
         <code>
          black
         </code>
         , if
         <code>
          black
         </code>
         is the browser default for that element's property.
        </p>
        <h3>
         How it is different from
         <code>
          inherit
         </code>
        </h3>
        <p>
         If you're thinking this
         <a href="http://www.w3.org/TR/CSS2/cascade.html#value-def-inherit">
          sounds a lot like
         </a>
         <code>
          inherit
         </code>
         , then you're absolutely right. It does sounds a heck of lot like that.
        </p>
        <p>
         But
         <code>
          initial
         </code>
         and
         <code>
          inherit
         </code>
         are distinct in the extra step that
         <code>
          inherit
         </code>
         takes to check whether there are other properties it can use in the cascade before it moves to the initial value.
        </p>
        <figure class="align-none media-202641" id="post-202641">
         <img alt="" src="https://cdn.css-tricks.com/wp-content/uploads/2015/05/initial-diagram-inherit.jpg"/>
         <br/>
         <figcaption>
          H1 is looking to inherit a color value, which it finds in the body element.
         </figcaption>
        </figure>
        <figure class="align-none media-202641" id="post-202641">
         <img alt="" src="https://cdn.css-tricks.com/wp-content/uploads/2015/05/initial-diagram.jpg"/>
         <br/>
         <figcaption>
          H1 is told to use its initial value, so it skips over the body element and goes to its roots.
         </figcaption>
        </figure>
        <h3>
         An example of the difference
        </h3>
        <p class="codepen" data-default-tab="result" data-height="320" data-slug-hash="wazyoV" data-theme-id="0" data-user="css-tricks">
         See the Pen
         <a href="http://codepen.io/team/css-tricks/pen/wazyoV/">
          CSS Initial vs. Inherit
         </a>
         by CSS-Tricks (
         <a href="http://codepen.io/css-tricks">
          @css-tricks
         </a>
         ) on
         <a href="http://codepen.io">
          CodePen
         </a>
         .
        </p>
        <p>
         See that? The properties in the left box are all set to inherit the values of the
         <code>
          .module
         </code>
         class since it the parent element. On the flip side, the properties in the right box are set to
         <code>
          initial
         </code>
         , which resets the element's properties to the browser defaults.
        </p>
        <h3>
         When to use
         <code>
          initial
         </code>
        </h3>
        <p>
         I like to think of
         <code>
          initial
         </code>
         as a hard reset. It's easy for styles to get convoluted as CSS grows, and using
         <code>
          initial
         </code>
         is a way to clear things out so an element can go back to its natural state of being. If
         <code>
          initial
         </code>
         were old-school Nintendo, I would use it as the equivalent to pulling out a buggy game cartridge from the console and
         <a href="https://www.youtube.com/watch?v=O3QPRDtWsgw" rel="nofollow">
          blowing into it
         </a>
         (even though doing so
         <a href="http://mentalfloss.com/article/12589/did-blowing-nintendo-cartridges-really-help" rel="nofollow">
          supposedly had no effect
         </a>
         ).
        </p>
        <p>
         But that doesn't mean
         <code>
          initial
         </code>
         is a silver bullet for resets. That's because initial values are still subject to browser defaults, which we know can
         <a href="https://css-tricks.com/numeric-inputs-a-comparison-of-browser-defaults/">
          vary from browser to browser
         </a>
         .
        </p>
        <p>
         Oh wait, you use
         <a href="http://meyerweb.com/eric/tools/css/reset/">
          CSS resets
         </a>
         ? You can expect those to be used as initial values instead.
        </p>
        <p>
         Bottom line: I would use
         <code>
          initial
         </code>
         to completely wipe out any inherited styles from an element and use
         <code>
          inherit
         </code>
         to make sure the element takes its cues from the nearest parent.
        </p>
        <h3>
         A more practical use case
        </h3>
        <p>
         Here's an example of how
         <code>
          initial
         </code>
         can be used to create alternating colored lines in tables.
        </p>
        <p class="codepen" data-default-tab="result" data-height="313" data-slug-hash="rVLmdd" data-theme-id="0" data-user="css-tricks">
         See the Pen
         <a href="http://codepen.io/team/css-tricks/pen/rVLmdd/">
          CSS Initial
         </a>
         by CSS-Tricks (
         <a href="http://codepen.io/css-tricks">
          @css-tricks
         </a>
         ) on
         <a href="http://codepen.io">
          CodePen
         </a>
         .
        </p>

        </p>

        <h3>
         Wrapping Up
        </h3>
        <p>
         I've been racking my brain for some interesting use cases for
         <code>
          initial
         </code>
         . While I see a lot of potential usefulness in being able to use default styles on an element, I just haven't come across them in a practical applicationthough that could say a lot more about me than the property value itself.
        </p>
        <p>
         Where this will come in real handy is when
         <code>
          all
         </code>
         gains more support
         <a href="https://css-tricks.com/almanac/properties/a/all/">
          as a property
         </a>
         . That would make declaring
         <code>
          all: initial
         </code>
         a real powerful tool for creating resets on any element.
        </p>
        <p>
         Please share any situations where
         <code>
          initial
         </code>
         was something you had to use in a project. Bonus points if you've used it for any tricky feats.
        </p>

             <div class="comment-content">
              <p>
               Unfortunately, as Geoff point out, this nice use case wouldnt actually work as you hope.
              </p>
              <p>
               The
               <code>
                initial
               </code>
               value is defined for each
               <em>
                property
               </em>
               , (e.g., for
               <code>
                display
               </code>
               ), not for any particular tag name (e.g.,
               <code>
                li
               </code>
               ).  The association between HTML tag names and display types is applied via a browser style sheet, which is treated just as an ordinary CSS stylesheet that is run before processing any of your websites styles.  Those default styles get reset with
               <code>
                initial
               </code>
               just as much as your own code does.
              </p>
              <p>
               In contrast, there are a very small number of properties, such as
               <code>
                font-family
               </code>
               , where the specs dont specify an initial value.  In that case, the browsers default
               <em>
                is
               </em>
               the initial value.  However, if the browser applies different font families for headings versus paragraphs, that is again the browser style sheet, and an
               <code>
                initial
               </code>
               value will reset them both to the same overall default (whatever that may be).
              </p>

              </p>
              <p>
               The point of most standard CSS reset techniques is to
               <em>
                cancel out
               </em>
               the browsers default HTML stylesheet and instead set values to the CSS initial value.  In these cases, you could use
               <code>
                initial
               </code>
               and get the same effect.
              </p>
              <p>
               However, if you use any other type of CSS reset (to a value other than the initial value defined in the specs), you cant recreate that with
               <code>
                initial
               </code>
               .  For example, if you use a reset to change
               <code>
                box-sizing
               </code>
               to
               <code>
                border-box
               </code>
               on every element and before/after pseudo-element, thats very different from
               <code>
                box-sizing: initial
               </code>
               .
              </p>

             <div class="comment-content">
              <p>
               This article seems to have a fundamental misunderstanding of initial. It is nothing like inherit, and it has nothing to with what properties are set on the root, aside from the root not being commonly styled much.
              </p>
              <p>
               Initial is just the value of a property before any value is set by a style sheet, whether it is a browser (aka UA) style sheet, author (web designer or developer) style sheet, or user style sheet.
              </p>
              <p>
               The UA style sheet can be thought of as the browser defaults, and can vary much more from browser to browser. Those variations are mostly eliminated by reset stylesheets, which also remove a lot of useful values, like margins on paragraphs.
              </p>
              <p>
               But all conformant browsers start with the initial value as defined in the spec. Thats why the UA style sheet has div { display: block }, but they dont need a span { display: inline }, since inline is already the default initial value for the display property anyway, regardless of what the element is, until a stylesheet sets it to something else.
              </p>
              <p>
               Inherit on the other hand, means that the value is taken from the parent element in the HTML. Some properties, like color inherit automatically, without having to set anything in the style sheet. Thats why if you have color: green on a P tag, you dont also have to set it on a SPAN inside the P (unless you are worried that the UA stylesheet has a color explicitly set for SPANs, but that is never the case).
              </p>
              <p>
               If you use the inherit keyword as a value, it forces inheritance of the value from the parent, even if it is a property that doesnt normally inherit automatically, such as float. So if you had float: inherit on element A, and the parent B of that element A had float: left, then element A would also float to the left inside the floating element B.
              </p>

             <div class="comment-content">
              <p>
               @tom I dont think it is really accurate to say they override the usual rules for determining the value of a property. The keywords give a value to a property, same as any value does. Whether or not initial or inherit keywords have an effect still depends on selector specificity and order, whether or not  !important is there, and with author style sheets overriding UA style sheets, same as any other value.
              </p>
              <p>
               What makes keyword values different from most others is that the used value varies from the specified value (not in itself unusual), but with initial getting its used value from the propertys initial value (as defined in the specification of the property), and inherit getting its used value from whatever its HTML parents used value. Those two effects are very different from each other, but they use keywords the same as any property would use a keyword, such as none or solid. All keyword values assign some meaningful value that generally depends on the property, and so do these. They both begin with the letter i, which I suppose can make it easier to confuse them.
              </p>
              <p>
               The fact that they are defined in the same spec doesnt make them similar, any more than border is similar to background because they are in the same spec. Nor are solid and none are similar (as border styles), just because they apply to the same properties in the same spec and are defined in close proximity to each other. The main thing that makes initial and inherit similar is that they can apply to any property.
              </p>
              <p>
               To truly understand what these two keywords do when assigned to a property, you just have to understand what they mean as concepts. The meaning exist regardless of whether or not there was ever a keyword that could be used as a property value. I think some people get confused because the keyword is something you can set. But even when not set, all properties have an initial value and many, many properties inherit anyway. Having an initial value prior to any CSS rules applying is a
               <strong>
                very
               </strong>
               different concept than looking at the HTML hierarchy to see if the parent element has a property value (like color or font size) that also should apply to the child. Once you understand those two concepts, then the meaning of the keywords you can use should be very clear and distinct.
              </p>

           <div class="comment-content">
            <p>
             I hesitate to say it, because I think you know more about CSS than I ever will, but I think youve missed the point of
             <code>
              initial
             </code>
             . The initial value exists because  there needs to be some default for all possible values. The CSS specification cannot assume that there are any user-agent style sheets, so it needs to define a default value for all possible properties. These are mostly what you would expect, but there are some quirks. Because these are default values and do not reference any kind of selector, they are the same for all elements at all times. For example, as noted in a comment above, the initial value for
             <code>
              display
             </code>
             is
             <code>
              inline
             </code>
             , which makes sense for
             <code>
              span
             </code>
             elements, but not so much for
             <code>
              div
             </code>
             or
             <code>
              p
             </code>
             . (You can see a list of initial values in section 4.1 of the
             <a href="http://www.w3.org/TR/CSS/" rel="nofollow">
              Spec
             </a>
             ).
            </p>
            <p>
             So your Google translation of the spec isnt quite right. A better description would be:
            </p>
            <blockquote>
             <p>
              The
              <code>
               initial
              </code>
              keyword is used to set the value of a property to the default value, as defined by the CSS specification.
             </p>
            </blockquote>
            <p>
             (Note there are some additional complications: you will see in the table that some of the initial values are listed as user-agent specific; however, this is separate from any user-agent style sheets, and is best thought of as a hard-coded default of the browser. Additionally, Ive glossed over the difference between the specified value and computed value, which is important because the
             <code>
              initial
             </code>
             keyword actually sets the
             <em>
              specified
             </em>
             value.)
            </p>
            <p>
             So why would you use
             <code>
              initial
             </code>
             ? I honestly cant think of a real reason, outside of curiosity. It certainly cant be used as a hard reset; from the point of view of the CSS parser, theres no difference between
             <code>
              p { display: initial; }
             </code>
             and
             <code>
              p { display: inline; }
             </code>
             , so why would you use the less readable version?
            </p>
