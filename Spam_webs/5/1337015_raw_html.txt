         <h2 id="1.Introduction|outline">
          Introduction
         </h2>
         <p>
          Map-based mashups are omnipresent nowadays, largely because services that
                can be mashed up are widely available.  
                For a map-based mashup, you need services that provide
                cartographic boundaries. The U.S. Census Bureau releases
                data on state, county, and ZIP code boundaries in Environmental Systems
                Research Institute (ESRI) shapefile format.
                In this article, learn how to create a simple service to deliver KML
                files from ESRI shapefiles using open source software.
         </p>
         <div class="ibm-alternate-rule">
          <hr/>
         </div>
         <p class="ibm-ind-link ibm-back-to-top">

         <h2 id="2.Technology and software products used|outline">
          Technology and software products
         </h2>
         <div class="dw-sidebar ibm-inset">
          <p>
           <strong>
            GIS
           </strong>
            Geographical information systems
          </p>
          <p>
           <strong>
            KML
           </strong>
            Keyhole markup language
          </p>
          <p>
           <strong>
            Mashup
           </strong>
            A Web hybrid application that combines
                        data from more than one source into an integrated experience
          </p>
          <p>
           <strong>
            TIGER
           </strong>
            Topologically integrated geographic encoding and
                referencing
          </p>
         </div>
         <p>
          There are many tools and products, both commercial and open source, that
               you can use to work with spatial data and create mashups. The
               example in this article uses open source software: PostGIS as the database,
               GeoTools Java library, the OpenLayers
               mapping library, and Apache Tomcat as the application server.
         </p>
         <dl>
          <dt>
           <strong>
            PostGIS
           </strong>
          </dt>
          <dd>
           A spatial extension to the open source relational database
                    PostgresSQL. PostGIS supports spatial data types and functions as
                    described by the OpenGIS Simple Features Specification for SQL
                    standard.
          </dd>
          <dt>
           <strong>
            GeoTools
           </strong>
          </dt>
          <dd>
           A Java library to perform manipulation of spatial data.
                     GeoTools includes:
           <ul class="ibm-bullet-list">
            <li>
             Support for various spatial
                    data formats (such as ESRI  shapefile, Geography Markup Language
                    (GML), 
                    MapInfo Interchange Format (MIF)), and spatial transformations.
            </li>
            <li>
             Java Topology Suite (JTS) from Vivid Solutions
                    that defines Java classes to represent spatial data.
            </li>
           </ul>
           <p>
            The example solution
                    uses GeoTools for spatial transformations and JTS for
                    converting the binary large object (BLOB) data from the database to a geometry.
           </p>
          </dd>
          <dt>
           <strong>
            OpenLayers
           </strong>
          </dt>
          <dd>
           An open source JavaScript mapping library that lets you
                    add dynamic maps to Web pages. You can overlay markers, lines,
                    and polygons on top of base maps.  The example in this article uses OpenLayers' capability
                    of adding a layer on top of a base map from a KML file.
          </dd>
          <dt>
           <strong>
            Apache Tomcat
           </strong>
          </dt>
          <dd>
           An open source J2EE JSP and servlet container used in the example to deploy the service.
          </dd>
         </dl>

         <h3 id="4.Boundary Data|outline">
          Boundary data
         </h3>
         <p>
          The U.S. Census Bureau uses the Topologically Integrated Geographic
                Encoding and Referencing (TIGER) format and database to describe and
                publish land attributes like roads, buildings, rivers, lakes, and 
                census tracts.  The TIGER database includes a dataset with the cartographic
                boundaries of states, counties, cities, and ZIP codes.  This dataset is in
                ESRI shapefile format (see
          <a href="#resources">
           Resources
          </a>
          ). The example uses the state boundaries in the ESRI shapefile format.
         </p>
         <h3 id="5.Getting the data|outline">
          Getting
            the data
         </h3>
         <p>
          The U.S Census Bureau provides a state boundary dataset in ARC/INFO Export
                (.e00), ARC/INFO Ungenerate, and ESRI shapefile (.shp) formats.  This
                article uses the ESRI shapefile format. State boundaries are available for
                every state individually, and there's a shapefile that includes all state
                boundaries.  The example uses the shapefile for all states
                (st99_d00_shp.zip).
         </p>
         <p>
          Unzip the contents of the st99_d00_shp.zip file to get the st99_d00.shp
           shapefile and associated files (st99_d00.shx and st99_d00.dbf).
         </p>
         <h3 id="6.Importing the data into PostGIS|outline">
          Importing the data into PostGIS
         </h3>
         <p>
          PostGIS does not provide a utility to directly import data in ESRI
                shapefile format into the database (as IBM DB2 does). You have to
                follow a two-step process to import data into PostGIS:
         </p>
         <ol type="1">
          <li>
           PostGIS provides the shp2pgsql tool, which lets you convert ESRI
                    shapefiles into SQL-suitable
           <code>
            INSERT
           </code>
           statements. To use this tool to
                    create a file with SQL
           <code>
            INSERT
           </code>
           statements
                    from the shapefile, use the command
           <code>
            shp2pgsql
                    st99_d00.shp testdb.state_bounds &gt; statebounds.sql
           </code>
           . This command, with no
                    command-line options, will assume the
                        default
           <code>
            -1
           </code>
           value for SRS ID. This is fine for the purposes of this
                article, as you'll use GeoTools for reverse projections and not
                the database features for coordinate transformations.
          </li>
          <li>
           After ensuring that the database is started, you can then run the
           <code>
            psql
           </code>
           command to run
                    the script.
          </li>
         </ol>
         <p>
          By default, the SQL file created by the shp2pgsql tool will contain
          <code>
           table create
          </code>
          and the statement to add the geometry
                column to the table. You can use various command-line options of the tool
                to not create the table. (See
          <a href="#resources">
           Resources
          </a>
          for PostGIS
                documentation.)
         </p>
         <h3 id="N100E7">
          Validating the import
         </h3>
         <p>
          You can validate the import process by using an admin tool like pgAdmin III
                or by executing SQL
          <code>
           SELECT
          </code>
          statements on the
                table. You can try PostGIS functions, such as
          <code>
           AsText()
          </code>
          and
          <code>
           AsBinary()
          </code>
          , in the geometry column to ensure
                that the import was successful.
         </p>

         <h2 id="7.The application|outline">
          The
 application
         </h2>
         <p>
          The application to create KML from the data you just imported will
            be a J2EE servlet. This servlet will:
         </p>
         <ul class="ibm-bullet-list">
          <li>
           Receive the state FIPS code as input.
          </li>
          <li>
           Query the database.
          </li>
          <li>
           Perform coordinate transformation.
          </li>
          <li>
           Create a KML from the data.
          </li>
         </ul>
         <h3 id="N10107">
          Understanding the data
         </h3>
         <p>
          The data imported into the database can have more than one row for a given
                state FIPS code. There could be multiple non-contiguous
                tracts of land that might form a given state. For example, the state of
                Alaska comprises many islands. To keep our application simple, we'll select only the piece of
                land that has the greatest area.
         </p>
         <h3 id="N1010E">
          Querying the database and reading
            the geometry
         </h3>
         <p>
          Geographical data can be represented in two formats: Well
                Known Binary (WKB) and Well Known Text (WKT). PostGIS provides
          <code>
           AsBinary()
          </code>
          and
          <code>
           AsText()
          </code>
          functions to convert the geometry
                column to WKB and WKT representations of the data, respectively. The SQL
                statement to query the data is in Listing 1.
         </p>
         <h5 id="N1011D">
          Listing 1. Query to retrieve the state boundary in WKB format
         </h5>
         <div class="codesection">
          <pre class="displaycode">
           String stateBoundsSql = "SELECT AsBinary(the_geom), name FROM state_bounds WHERE 
   state = '"	+ stateCd + "' ORDER BY area DESC LIMIT 1";
          </pre>
         </div>
         <p>
          The GeoTools library comes with the Java Topology Suite (JTS), which has
                the class
          <code>
           WKBReader
          </code>
          that converts WKB representations of
                geometry into Java objects that represent the geometry.  You will use this
                to convert the state boundary data to a
          <code>
           MultiPolygon
          </code>
          Java object, which is another JTS
                class.  Listing 2 shows an example.
         </p>
         <h5 id="N1012D">
          Listing 2.  Read the geometry into a JTS Java object
         </h5>
         <div class="codesection">
          <pre class="displaycode">
           WKBReader wkbRead =
           <strong>
            new
           </strong>
           WKBReader();
Geometry geom = wkbRead.read(resultSet.getBytes(1));
MultiPolygon multiPolygon = (MultiPolygon) geom;
          </pre>
         </div>
         <h3 id="N10136">
          Coordinate
            transformations
         </h3>
         <p>
          The data you imported from the U.S. Census Bureau is in a spatial reference
                system based on North American Datum 1983 (NAD83). It uses an
                ellipsoid approximation for the Earth. Web-based mapping libraries, such
                as OpenLayers, Google Maps, and Microsoft Bing, are all based
                on the World Geodetic
                System (WGS) datum. To simplify the calculations involved in showing
                data on the Web-based map, these libraries assume a spheroid Earth, instead
                of an ellipsoid Earth. Thus, the data needs to be transformed before we can
                overlay on Web-based maps. You can use the transformation
                functions provided by the GeoTools library, as shown in Listing 3.
         </p>
         <h5 id="N10141">
          Listing 3. Create transformation object
         </h5>
         <div class="codesection">
          <pre class="displaycode">
           String SOURCE_WKT = "GEOGCS[\"GCS_North_American_1983\",DATUM[\"D_North_American_1983\",
   SPHEROID[\"GRS_1980\",6378137.0,298.257222101]],PRIMEM[\"Greenwich\",0.0],UNIT[
   \"Degree\",0.0174532925199433]], AXIS[\"Latitude\",NORTH]]";

String TARGET_WKT = "GEOGCS[\"WGS 84\",DATUM[\"World Geodetic System 1984\",SPHEROID[
   \"WGS 84\", 6378137.0, 298.257223563, AUTHORITY[\"EPSG\",\"7030\"]],AUTHORITY[\"EPSG\",
   \"6326\"]],PRIMEM[\"Greenwich\", 0.0,AUTHORITY[\"EPSG\",\"8901\"]],UNIT[\"degree\",
     0.017453292519943295],AXIS[\"Geodetic latitude\", NORTH],AXIS[\"Geodetic longitude\",
    EAST],AUTHORITY[\"EPSG\",\"4326\"]]";

CRSFactory crsFactory = FactoryFinder.
           <em>
            getCRSFactory
           </em>
           (
           <strong>
            null
           </strong>
           );
CoordinateReferenceSystem sourceCRS = crsFactory
					.createFromWKT(SOURCE_WKT);
CoordinateReferenceSystem targetCRS = crsFactory
					.createFromWKT(TARGET_WKT);
Hints hint =
           <strong>
            new
           </strong>
           Hints(Hints.
           <em>
            LENIENT_DATUM_SHIFT
           </em>
           ,
           <strong>
            true
           </strong>
           );
CoordinateOperationFactory coFactory = FactoryFinder.
           <em>
            getCoordinateOperationFactory
           </em>
           (hint);
CoordinateOperation co = coFactory.createOperation(sourceCRS,targetCRS);
MathTransform transform = co.getMathTransform();
          </pre>
         </div>

         <h2 id="8.Conclusion|outline">
          Conclusion
         </h2>
         <p>
          With mashups, enterprises can unlock the potential of all sorts of data.
                You can now provide users with the capability to create situational applications at a
                fraction of the cost of traditional application development
                paradigms.
         </p>
         <p>
          Location intelligence mashups and map-based mashups are popular categories
                 of mashups.  With the advent of Web-based
                mapping providers, it's now easy to create map-based
                mashups with  little or no capital investment.  In this article,
                you learned how to use open source software to create a KML boundary
                service from an ESRI shapefile that can be used in mashups.
         </p>
