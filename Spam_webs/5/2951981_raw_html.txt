                   <div class="post hentry">
                    <a name="215885192239788164">
                    </a>
                    <h3 class="post-title entry-title">
                     Security Issue: Combining Bcrypt With Other Hash Functions
                    </h3>
                    <div class="post-header">
                     <div class="post-header-line-1">
                     </div>
                    </div>
                    <div class="post-body entry-content">
                     <div style="text-align: justify">
                      The other day, I was directed at an interesting
                      <a href="http://stackoverflow.com/q/28951359/338665">
                       question on StackOverflow
                      </a>
                      asking if
                      <i>
                       password_verify()
                      </i>
                      was safe against DoS attacks using extremely long passwords. Many hashing algorithms depend on the amount of data fed into them, which affects their runtime. This can lead to a DoS attack where an attacker can provide an exceedingly long password and tie up computer resources. It's a really good question to ask of Bcrypt (and
                      <i>
                       password_hash
                      </i>
                      ). As you may know, Bcrypt is limited to 72 character passwords. So on the surface it looks like it shouldn't be vulnerable. But I chose to dig in further to be sure. What I found surprised me.
                      <br/>
                      <br/>
                     </div>
                     <a name="more">
                     </a>
                     <h2>
                      crypt.c
                     </h2>
                     <div style="text-align: justify">
                      To start off, I looked at PHP's implementation of
                      <a href="http://lxr.php.net/xref/PHP_TRUNK/ext/standard/crypt.c#154">
                       <i>
                        crypt()
                       </i>
                      </a>
                      . The function we're interested in is called
                      <i>
                       php_crypt()
                      </i>
                      , and has the following signature:
                      <br/>
                      <br/>
                     </div>
                     <pre class="prettyprint">
                      PHPAPI zend_string *php_crypt(const char *password, const int pass_len, const char *salt, int salt_len)
                     </pre>
                     <div style="text-align: justify">
                      Now, check out the branch that does bcrypt:
                      <br/>
                      <br/>
                     </div>
                     <pre class="prettyprint">
                      } else if (
        salt[0] == '$' &amp;&amp;
        salt[1] == '2' &amp;&amp;
        salt[3] == '$') {
    char output[PHP_MAX_SALT_LEN + 1];

    memset(output, 0, PHP_MAX_SALT_LEN + 1);

    crypt_res = php_crypt_blowfish_rn(password, salt, output, sizeof(output));
    if (!crypt_res) {
        ZEND_SECURE_ZERO(output, PHP_MAX_SALT_LEN + 1);
        return NULL;
    } else {
        result = zend_string_init(output, strlen(output), 0);
        ZEND_SECURE_ZERO(output, PHP_MAX_SALT_LEN + 1);
        return result;
    }
}
                     </pre>
                     <div style="text-align: justify">
                      Notice anything? The
                      <i>
                       password
                      </i>
                      variable is a
                      <i>
                       char*
                      </i>
                      . So
                      <i>
                       php_crypt_blowfish_rn()
                      </i>
                      doesn't know the length of the input password. Interesting. So I wonder how it knows the length? Well, at this point I have an idea, but I'm also scared by it...
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      So digging through
                      <i>
                       php_crypt_blowfish_rn()
                      </i>
                      , I notice that the only place the passsword (now called
                      <i>
                       key
                      </i>
                      ) is sent is the function
                      <a href="http://lxr.php.net/xref/PHP_TRUNK/ext/standard/crypt_blowfish.c#BF_set_key">
                       <i>
                        BF_set_key()
                       </i>
                      </a>
                      . There's a bunch of comments about sign safety and mode switches, but the bulk of it boils down to this pair of nested loops (comments stripped out):
                      <br/>
                      <br/>
                     </div>
                     <pre class="prettyprint">
                      const char *ptr = key;
/* ...snip... */
for (i = 0; i &lt; BF_N + 2; i++) {
    tmp[0] = tmp[1] = 0;
    for (j = 0; j &lt; 4; j++) {
        tmp[0] &lt;&lt;= 8;
        tmp[0] |= (unsigned char)*ptr; /* correct */
        tmp[1] &lt;&lt;= 8;
        tmp[1] |= (BF_word_signed)(signed char)*ptr; /* bug */
        if (j)
            sign |= tmp[1] &amp; 0x80;
        if (!*ptr)
            ptr = key;
        else
            ptr++;
    }
    diff |= tmp[0] ^ tmp[1]; /* Non-zero on any differences */

    expanded[i] = tmp[bug];
    initial[i] = BF_init_state.P[i] ^ tmp[bug];
}
                     </pre>
                     <div style="text-align: justify">
                      If you're not familiar with C,
                      <i>
                       *
                      </i>
                      dereferences a pointer (returns the value it points to). So if we define
                      <i>
                       char *abc = "abc"
                      </i>
                      , then
                      <i>
                       *abc
                      </i>
                      would be
                      <i>
                       'a'
                      </i>
                      (well, technically, the numeric value of the codepoint of
                      <i>
                       'a'
                      </i>
                      ). So then you can increment the pointer
                      <i>
                       abc++
                      </i>
                      and then
                      <i>
                       *abc
                      </i>
                      would equal
                      <i>
                       'b
                      </i>
                      '. This is the standard way that "strings" work in C.
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      The loops basically iterate 72 times (
                      <i>
                       BF_N
                      </i>
                      is 16) and "eats" one byte of the string each iteration.
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      The key point here is the following line of code:
                      <br/>
                      <br/>
                     </div>
                     <pre class="prettyprint">
                      if (!*ptr)
    ptr = key;
else
    ptr++;
                     </pre>
                     <div style="text-align: justify">
                      Basically, if
                      <i>
                       *ptr
                      </i>
                      is ever
                      <i>
                       0
                      </i>
                      , then reset it to the start of the string. This is how strings shorter than 72 characters are represented (since a "c-string" always ends with a null byte).
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      Think about that for a second. That means that
                      <i>
                       "test\0abc"
                      </i>
                      would be treated as
                      <i>
                       "test\0test\0test\0test\0test\0test\0test\0test\0test\0test\0test\0test\0test\0test\0te"
                      </i>
                      . In fact, every string that starts with
                      <i>
                       "test\0"
                      </i>
                      will be treated the same.
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      Basically, it ignores everything after the first null byte.
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      See the problem?
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      Nobody really uses null bytes in passwords anyway. So this isn't a problem. Right?
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      No, in fact, this is by-design. And it's not an issue because nobody uses passwords with null bytes.
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      So if you use
                      <i>
                       password_hash()
                      </i>
                      or
                      <i>
                       crypt()
                      </i>
                      directly, you're 100% safe.
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      But what if you're not using them directly? What if you're "pre-hashing"? Well, then you have a MAJOR problem...
                      <br/>
                      <br/>
                     </div>
                     <h2>
                      The Major Problem
                     </h2>
                     <div style="text-align: justify">
                      Some people think bcrypt isn't enough, and instead choose to "pre-hash" passwords. This allows users to use passwords longer than 72 characters. Something like:
                      <br/>
                      <br/>
                     </div>
                     <pre class="prettyprint">
                      password_hash(hash('sha256', $password, true), PASSWORD_DEFAULT)
                     </pre>
                     <div style="text-align: justify">
                      Additionally, some people want to use a "pepper", so they pre-hash using a HMAC with a private key:
                      <br/>
                      <br/>
                     </div>
                     <pre class="prettyprint">
                      password_hash(hash_hmac('sha256', $password, $key, true), PASSWORD_DEFAULT)
                     </pre>
                     <div style="text-align: justify">
                      The problem here is the last argument to the two hash functions:
                      <i>
                       true
                      </i>
                      . They force raw output. This is normally how cryptographic functions are combined (using raw output rather than encoded output). And given that you can lose entropy from a sha512 by truncating it from 128 characters to 72, using raw output preserves some entropy.
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      But this means that the output can contain null bytes. In fact, it means that on average 1 out of every 256 passwords (or
                      <i>
                       0.39%
                      </i>
                      ) will have a leading null byte. So we only need to try approximately 177 passwords to get a 50% chance of finding a hash with a leading null byte. And we only need to try approximately 177 users to get a 50% chance of finding a user with a leading null byte. So trying 31329 permutations of users and passwords gives us a 25% chance of finding one that will work. That's in the realm of possibility for online attacks (via distributed means).
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      This is
                      <strong>
                       bad
                      </strong>
                      . This is
                      <strong>
                       really bad
                      </strong>
                      .
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      Let's show an example of how we can find these collisions:
                      <br/>
                      <br/>
                     </div>
                     <pre class="prettyprint">
                      $key = "algjhsdiouahwergoiuawhgiouaehnrgzdfgb23523";
$hash_function = "sha256";
$i = 0;
$found = [];

while (count($found) &lt; 2) {
    $pw = base64_encode(str_repeat($i, 5));
    $hash = hash_hmac($hash_function, $pw, $key, true);
    if ($hash[0] === "\0") {
        $found[] = $pw;
    }
    $i++;
}

var_dump($i, $found);
                     </pre>
                     <div style="text-align: justify">
                      I picked a random key. Then I made a "random" looking password (encoding the iteration count). Then I let it run. This is a stupid simple generator to generate these collisions (not even efficient). When I ran it, I got the following output:
                      <br/>
                      <br/>
                     </div>
                     <pre class="prettyprint">
                      int(523)
array(2) {
  [0]=&gt;
  string(16) "MzEzMTMxMzEzMQ=="
  [1]=&gt;
  string(20) "NTIyNTIyNTIyNTIyNTIy"
}
                     </pre>
                     <div style="text-align: justify">
                      Which means for that key, we found 2 colliding passwords in 523 attempts. The attempt number will change with different keys.
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      So let's try that out:
                      <br/>
                      <br/>
                     </div>
                     <pre class="prettyprint">
                      $hash = password_hash(hash_hmac("sha256", $found[0], $key, true), PASSWORD_BCRYPT);

var_dump(password_verify(hash_hmac("sha256", $found[1], $key, true), $hash));
                     </pre>
                     <div style="text-align: justify">
                      And we get:
                      <br/>
                      <br/>
                     </div>
                     <pre class="prettyprint">
                      bool(true)
                     </pre>
                     <div style="text-align: justify">
                      Amazing. Different passwords validate to the same hash.
                      <br/>
                      <br/>
                     </div>
                     <h2>
                      Detecting Problematic Hashes
                     </h2>
                     <div style="text-align: justify">
                      Offline, there's a simple check to see if a given hash was created with a leading null byte:
                      <br/>
                      <br/>
                     </div>
                     <pre class="prettyprint">
                      password_verify("\0", $hash)
                     </pre>
                     <div style="text-align: justify">
                      Testing our hash from above:
                      <br/>
                      <br/>
                     </div>
                     <blockquote>
                      <div style="text-align: justify">
                       $2y$10$2ECy/U3F/NSvAjMcuBeI6uMDmJlI8t8ux0pXOAoajpv2hSH0veOMi
                       <br/>
                       <br/>
                      </div>
                     </blockquote>
                     <div style="text-align: justify">
                      Gives us
                      <i>
                       bool(true)
                      </i>
                      . Which tells us it was created with a null byte as the first character.
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      So in an offline mode, it's trivial to detect these hashes.
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      But even if none of the hashes were created with leading null bytes, it doesn't mean you're safe (assuming you are using raw hashes). The reason is the same thing happens if the 2nd character was null:
                      <br/>
                      <br/>
                     </div>
                     <pre class="prettyprint">
                      a\0bc
a\0cd
a\0ef
                     </pre>
                     <div style="text-align: justify">
                      Will all collide as well. So you have a
                      <i>
                       0.39%
                      </i>
                      chance of colliding at the second character for each given first character. So of all the hashes that start with
                      <i>
                       a
                      </i>
                      , a given hash has a
                      <i>
                       0.39%
                      </i>
                      chance of having a null byte as the second character. Meaning that there's significantly less work to find this collision than to find a full-hash collision.
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      The problem goes all the way down the line.
                      <br/>
                      <br/>
                     </div>
                     <h2>
                      But I use CRYPT_SHA256!
                     </h2>
                     <div style="text-align: justify">
                      Well, looking at
                      <a href="http://lxr.php.net/xref/PHP_TRUNK/ext/standard/crypt.c#154">
                       <i>
                        php_crypt()
                       </i>
                      </a>
                      , we can see that all of the
                      <i>
                       crypt()
                      </i>
                      options exibt this behavior. It's not really centralized around bcrypt (nor is it centralized around PHP, the
                      <i>
                       crypt(3)
                      </i>
                      c library itself has this same issue).
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      I talked about bcrypt in this post mainly because that's what
                      <i>
                       password_hash()
                      </i>
                      uses, and it's the current recommended algorithm in PHP.
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      Note that if you use
                      <i>
                       hash_pbkdf2()
                      </i>
                      , it's not susceptible to this issue. And if you use
                      <a href="https://github.com/DomBlack/php-scrypt">
                       scrypt
                      </a>
                      you're fine as well.
                      <br/>
                      <br/>
                     </div>
                     <h2>
                      The Fix
                     </h2>
                     <div style="text-align: justify">
                      The problem here isn't with bcrypt itself. It's with the combination of bcrypt and other crypto in an unsafe manner. It turns out that not all combinations are unsafe. In fact,
                      <a href="https://blog.mozilla.org/webdev/2012/06/08/lets-talk-about-password-storage/">
                       Mozilla's system
                      </a>
                      , which is basically
                      <i>
                       password_hash(base64_encode(hash_hmac("sha512", $password, $key, true)), PASSWORD_BCRYPT)
                      </i>
                      , is safe because it base64 encodes the raw salt. Additionally, you're safe if you use hex output of the hash/hmac (the final parameter
                      <i>
                       false
                      </i>
                      , which is the default).
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      You are 100% safe if you do one of the following:
                      <br/>
                      <br/>
                     </div>
                     <ul>
                      <li>
                       Use straight
                       <i>
                        bcrypt
                       </i>
                       (don't pre-hash)
                      </li>
                      <li>
                       Use hex output from the pre-hash
                      </li>
                      <li>
                       Base64 encode the raw output of a pre-hash
                      </li>
                     </ul>
                     <div style="text-align: justify">
                      If you are using raw output, encode it first, and you're safe.
                      <br/>
                      <br/>
                     </div>
                     <h2>
                      The Underlying Problem
                     </h2>
                     <div style="text-align: justify">
                      The underlying problem is that combining cryptographic operators that weren't designed to be combined can be disastrous. Is it possible to do so safely? Yes. Is it a good idea to do it? No. This particular case is just one example where combining operations can be exceedingly dangerous.
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      Instead, just use algorithms as they were designed to be used. If you want additional protection beyond bcrypt, encrypt the output:
                      <i>
                       encrypt(password_hash(...), $key)
                      </i>
                      . It's using algorithms as they were designed to be used.
                      <br/>
                      <br/>
                     </div>
                     <div style="text-align: justify">
                      But the bottom line:
                      <strong>
                       never
                      </strong>
                      roll your own crypto. It can have fatal consequences.
                      <br/>
                      <br/>
                     </div>
