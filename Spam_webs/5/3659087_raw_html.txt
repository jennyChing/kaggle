        <h4 class="title media-heading">
         <a href="http://www.browseemall.com/Blog/index.php/2015/05/19/the-optimal-cross-browser-testing-strategy/" rel="bookmark" title="Permanent Link to The Optimal Cross Browser Testing Strategy">
          The Optimal Cross Browser Testing Strategy
         </a>
        </h4>
        <!-- Meta details mobile -->
        <ul class="list-inline meta text-muted">
         <li>
          <i class="fa fa-calendar">
          </i>

        <div class="blog-content">
         <div class="blog-media">
          <img alt="strategy" class="img-responsive wp-post-image" height="487" src="http://www.browseemall.com/Blog/wp-content/uploads/2015/05/strategy.jpg" width="770"/>
         </div>
         <p>
         </p>
         <p>
          Nearly everything works better if you have some kind of plan or strategy. This holds especially true for tasks with a lot of variables like cross browser testing. Below I would like to share our tested and proven cross browser testing strategy we have now used in well over 100 testing projects. Hopefully it will be usefully for some of our fellow developers and testers.
         </p>
         <h2>
          Plan Ahead
         </h2>
         <p>
          A good plan is often half of the solution to any problem. This is especially true for any type of software testing which rarely works that well if it is done as an afterthought long after the development is finished. So we always make sure to consider the following as soon as we start a new project:
         </p>
         <ul>
          <li>
           Know your audience and research in advance what browsers / devices / operating system they are using. Use real world data where ever possible.
          </li>
          <li>
           Start testing as soon as you start developing. Encourage your team members to use different browsers and devices.
          </li>
          <li>
           Develop against a reference list of browsers to avoid being bogged down with bug reports for obscure browsers.
          </li>
          <li>
           Automate functional tests early onifpossible. This will also ease the burden of testing during the development phase
          </li>
         </ul>
         <h2>
          Verify The Technical Side
         </h2>
         <p>
          Many
          <a href="http://www.browseemall.com/Blog/index.php/2014/04/24/5-most-common-browser-compatibility-issues/">
           common cross browser issues
          </a>
          can be fixed with a few simple technical checks. Even better there are tools available to test these sort of things automatically.
         </p>
         <ul>
          <li>
           Always, always use a
           <a href="http://www.browseemall.com/Blog/index.php/2012/03/05/the-cross-browser-compatibility-guide-how-to-choose-the-right-doctype/">
            proper DOCTYPE
           </a>
          </li>
          <li>
           Be aware of the
           <a href="http://www.browseemall.com/Blog/index.php/2012/03/05/the-cross-browser-compatibility-guide-how-to-choose-the-right-doctype/">
            X-UA-Compatibility meta
           </a>
           tag you are using
          </li>
          <li>
           <a href="https://validator.w3.org/">
            Validate
           </a>
           your HTML to find errors
          </li>
          <li>
           <a href="https://jigsaw.w3.org/css-validator/">
            Validate
           </a>
           your CSS to find problems
          </li>
         </ul>
         <h2>
          Test Intelligent
         </h2>
         <p>
          Even after all the tools and automation you will still need to do a fair amount of work to get the cross browser testing done. But dont just jump into it,
          <strong>
           think prior totesting
          </strong>
          !
         </p>
         <ul>
          <li>
           Whenever possible create automated tests for functionality. Try
           <a href="http://www.seleniumhq.org/">
            Selenium
           </a>
           if you want to code or
           <a href="http://www.browserstage.com">
            BrowserStage
           </a>
           if you dont.
          </li>
          <li>
           Dont do any final layout tests until the layout is final.
          </li>
          <li>
           Make a full test run prior to fixing any bugs to minimize the retest effort.
          </li>
          <li>
           Start testing with thebrowser set of highest usage and work your way down in iterations.
          </li>
         </ul>
         <p>
          I think this strategy is a good starting point to base any serious testing effort on. Anything you can add to this? Feel free to tell us in the comments below!
         </p>
