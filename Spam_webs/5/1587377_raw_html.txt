     <p>
      <b>
       C string handling
      </b>
      refers to a group of functions implementing operations on strings in the
      <a href="/wiki/C_standard_library" title="C standard library">
       C standard library
      </a>
      . Various operations, such as copying, concatenation,
      <a href="/wiki/Tokenization_(lexical_analysis)" title="Tokenization (lexical analysis)">
       tokenization
      </a>
      and searching are supported.
     </p>
     <p>
      The only support for strings in the
      <a class="mw-redirect" href="/wiki/C_programming_language" title="C programming language">
       C programming language
      </a>
      itself is that the compiler will translate a quoted string constant into a
      <a href="/wiki/Null-terminated_string" title="Null-terminated string">
       null-terminated string
      </a>
      , which is stored in static memory. The
      <a href="/wiki/C_standard_library" title="C standard library">
       C standard library
      </a>
      , however, provides a large number of commonly used functions designed to manipulate these null-terminated strings.
     </p>

     <h2>
      <span class="mw-headline" id="Definitions">
       Definitions
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=C_string_handling&amp;action=edit&amp;section=1" title="Edit section: Definitions">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      A string is a contiguous sequence of
      <a class="mw-redirect" href="/wiki/Code_unit" title="Code unit">
       code units
      </a>
      terminated by the first zero code (
      <code>
       \0
      </code>
      , corresponding to the
      <a href="/wiki/Null_character" title="Null character">
       null character
      </a>
      ). In C, there are two types of strings:
      <b>
       string
      </b>
      , which is sometimes called
      <b>
       byte string
      </b>
      which uses the type
      <code>
       char
      </code>
      s as code units (one
      <code>
       char
      </code>
      is at least 8 bits), and
      <b>
       wide string
      </b>
      <sup class="reference" id="cite_ref-open-std1_1-0">
       <a href="#cite_note-open-std1-1">
        <span>
         [
        </span>
        1
        <span>
         ]
        </span>
       </a>
      </sup>
      which uses the type
      <code>
       wchar_t
      </code>
      as code units.
     </p>
     <p>
      A common misconception is that all
      <code>
       char
      </code>
      arrays are strings, because
      <a href="/wiki/String_literal" title="String literal">
       string literals
      </a>
      are converted to arrays during the compilation (or translation) phase.
      <sup class="reference" id="cite_ref-2">
       <a href="#cite_note-2">
        <span>
         [
        </span>
        2
        <span>
         ]
        </span>
       </a>
      </sup>
      It is important to remember that a string
      <b>
       ends
      </b>
      at the first zero code unit. An array or string literal that contains a zero before the last byte therefore
      <i>
       contains
      </i>
      a string, or possibly several strings, but is not itself a string.
      <sup class="reference" id="cite_ref-3">
       <a href="#cite_note-3">
        <span>
         [
        </span>
        3
        <span>
         ]
        </span>
       </a>
      </sup>
      Conversely, it is possible to create a
      <code>
       char
      </code>
      array that is not null-terminated and is thus not a string:
      <code>
       char
      </code>
      is often used as a small integer when needing to save memory.
     </p>
     <p>
      The term
      <b>
       pointer to a string
      </b>
      is used in C to describe a pointer to the initial (lowest-addressed) byte of a string.
      <sup class="reference" id="cite_ref-open-std1_1-1">
       <a href="#cite_note-open-std1-1">
        <span>
         [
        </span>
        1
        <span>
         ]
        </span>
       </a>
      </sup>
      In C, pointers are used to pass strings to functions. Documentation (including this page) will often use the term
      <i>
       string
      </i>
      to mean
      <i>
       pointer to a string
      </i>
      .
      <sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">
       [
       <i>
        <a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed">
         <span>
          citation needed
         </span>
        </a>
       </i>
       ]
      </sup>
     </p>
     <p>
      The term
      <b>
       length of a string
      </b>
      is used in C to describe the number of bytes preceding the zero byte.
      <sup class="reference" id="cite_ref-open-std1_1-2">
       <a href="#cite_note-open-std1-1">
        <span>
         [
        </span>
        1
        <span>
         ]
        </span>
       </a>
      </sup>
      <code>
       strlen
      </code>
      is a standardised function commonly used to determine the length of a string. A common mistake is to not realize that a string uses one more unit of memory than this length, in order to store the zero that ends the string.
     </p>
     <h2>
      <span class="mw-headline" id="Character_encodings">
       Character encodings
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=C_string_handling&amp;action=edit&amp;section=2" title="Edit section: Character encodings">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      Each string ends at the first occurrence of the zero code unit of the appropriate kind (
      <code>
       char
      </code>
      or
      <code>
       wchar_t
      </code>
      ). Consequently, a byte string can contain non-
      <a href="/wiki/Null_character" title="Null character">
       NUL
      </a>
      characters in
      <a href="/wiki/ASCII" title="ASCII">
       ASCII
      </a>
      or any
      <a href="/wiki/Extended_ASCII" title="Extended ASCII">
       ASCII extension
      </a>
      , but not characters in encodings such as
      <a href="/wiki/UTF-16" title="UTF-16">
       UTF-16
      </a>
      (even though a 16-bit code unit might be nonzero, its high or low byte might be zero). The encodings that can be stored in wide strings are defined by the width of
      <code>
       wchar_t
      </code>
      . In most implementations,
      <code>
       wchar_t
      </code>
      is at least 16 bits, and so all 16-bit encodings, such as
      <a class="mw-redirect" href="/wiki/UCS-2" title="UCS-2">
       UCS-2
      </a>
      , can be stored. If
      <code>
       wchar_t
      </code>
      is 32-bits, then 32-bit encodings, such as
      <a href="/wiki/UTF-32" title="UTF-32">
       UTF-32
      </a>
      , can be stored.
     </p>
     <p>
      <a href="/wiki/Variable-width_encoding" title="Variable-width encoding">
       Variable-width encodings
      </a>
      can be used in both byte strings and wide strings. String length and offsets are measured in bytes or
      <code>
       wchar_t
      </code>
      , not in "characters", which can be confusing to beginning programmers.
      <a href="/wiki/UTF-8" title="UTF-8">
       UTF-8
      </a>
      and
      <a href="/wiki/Shift_JIS" title="Shift JIS">
       Shift JIS
      </a>
      are often used in C byte strings, while
      <a href="/wiki/UTF-16" title="UTF-16">
       UTF-16
      </a>
      is often used in C wide strings when
      <code>
       wchar_t
      </code>
      is 16 bits. Truncating strings with variable length characters using functions like
      <code>
       strncpy
      </code>
      can produce invalid sequences at the end of the string. This can be unsafe if the truncated parts are interpreted by code that assumes the input is valid.
     </p>
     <p>
      Support for Unicode literals such as
      <code>
       char foo[512] = "";
      </code>
      (UTF-8) or
      <code>
       wchar_t foo[512] = L"";
      </code>
      (UTF-16 or UTF-32) is implementation defined,
      <sup class="reference" id="cite_ref-open-std_4-0">
       <a href="#cite_note-open-std-4">
        <span>
         [
        </span>
        4
        <span>
         ]
        </span>
       </a>
      </sup>
      and may require that the source code be in the same encoding. Some compilers or editors will require entering all non-ASCII characters as
      <code>
       \xNN
      </code>
      sequences for each byte of UTF-8, and/or
      <code>
       \uNNNN
      </code>
      for each word of UTF-16.
     </p>
     <h2>
      <span class="mw-headline" id="Overview_of_functions">
       Overview of functions
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=C_string_handling&amp;action=edit&amp;section=3" title="Edit section: Overview of functions">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      Most of the functions that operate on C strings are declared in the
      <code>
       string.h
      </code>
      header (
      <code>
       cstring
      </code>
      in C++), while functions that operate on C wide strings are declared in the
      <code>
       wchar.h
      </code>
      header (
      <code>
       cwchar
      </code>
      in C++). These headers also contain declarations of functions used for handling memory buffers; the name is thus something of a misnomer.
     </p>
     <p>
      Functions declared in
      <code>
       string.h
      </code>
      are extremely popular since, as a part of the
      <a href="/wiki/C_standard_library" title="C standard library">
       C standard library
      </a>
      , they are guaranteed to work on any platform which supports C. However, some security issues exist with these functions, such as potential
      <a href="/wiki/Buffer_overflow" title="Buffer overflow">
       buffer overflows
      </a>
      when not used carefully and properly, causing the programmers to prefer safer and possibly less portable variants, out of which some popular ones are listed below. Some of these functions also violate
      <a class="mw-redirect" href="/wiki/Const-correctness" title="Const-correctness">
       const-correctness
      </a>
      by accepting a
      <code>
       const
      </code>
      string pointer and returning a non-
      <code>
       const
      </code>
      pointer within the string. To correct this, some have been separated into two
      <a href="/wiki/Function_overloading" title="Function overloading">
       overloaded functions
      </a>
      in the C++ version of the standard library.
     </p>
     <p>
      In historical documentation the term "character" was often used instead of "byte" for C strings, which leads many to believe that these functions somehow do not work for
      <a href="/wiki/UTF-8" title="UTF-8">
       UTF-8
      </a>
      . In fact all lengths are defined as being in bytes and this is true in all implementations, and these functions work as well with UTF-8 as with single-byte encodings. The BSD documentation has been fixed to make this clear, but POSIX, Linux, and Windows documentation still uses "character" in many places where "byte" or "wchar_t" is the correct term.
     </p>
     <p>
      Functions for handling memory buffers can process sequences of bytes that include null-byte as part of the data. Names of these functions typically start with
      <code>
       mem
      </code>
      , as opposite to the
      <code>
       str
      </code>
      prefix.
     </p>

     <p>
      The C standard library contains several functions for numeric conversions. The functions that deal with byte strings are defined in the
      <code>
       stdlib.h
      </code>
      header (
      <code>
       cstdlib
      </code>
      header in C++). The functions that deal with wide strings are defined in the
      <code>
       wchar.h
      </code>
      header (
      <code>
       cwchar
      </code>
      header in C++).
     </p>
     <p>
      The
      <code>
       strtoxxx
      </code>
      functions are not
      <a class="mw-redirect" href="/wiki/Const-correctness" title="Const-correctness">
       const-correct
      </a>
      , since they accept a
      <code>
       const
      </code>
      string pointer and return a non-
      <code>
       const
      </code>
      pointer within the string. Also, since the Normative Amendment 1 (C95),
      <code>
       atoxx
      </code>
      functions are considered subsumed by
      <code>
       strtoxxx
      </code>
      functions, for which reason neither C95 nor any later standard provides wide-character versions of these functions.
      <sup class="reference" id="cite_ref-75">
       <a href="#cite_note-75">
        <span>
         [
        </span>
        73
        <span>
         ]
        </span>
       </a>
      </sup>
     </p>

     <h4>
      <span class="mw-headline" id="Strcat.2Fstrcpy_replacements">
       Strcat/strcpy replacements
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=C_string_handling&amp;action=edit&amp;section=9" title="Edit section: Strcat/strcpy replacements">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h4>
     <p>
      Despite
      <a href="/wiki/C_standard_library#Buffer_overflow_vulnerabilities" title="C standard library">
       the well-established need
      </a>
      to replace
      <code>
       strcat
      </code>
      and
      <code>
       strcpy
      </code>
      with functions that do not allow buffer overflows, no accepted standard has arisen. This is partly due to the mistaken belief by many C programmers that
      <code>
       strncat
      </code>
      and
      <code>
       strncpy
      </code>
      have the desired behavior; however, neither function was designed for this (they were intended to manipulate null-padded fixed-size string buffers, a data format less commonly used in modern software), and the behavior and arguments are non-intuitive and often written incorrectly even by expert programmers.
      <sup class="reference" id="cite_ref-strl_85-2">
       <a href="#cite_note-strl-85">
        <span>
         [
        </span>
        83
        <span>
         ]
        </span>
       </a>
      </sup>
     </p>
     <p>
      As part of its 2004
      <a href="/wiki/Microsoft_Security_Development_Lifecycle" title="Microsoft Security Development Lifecycle">
       Security Development Lifecycle
      </a>
      , Microsoft introduced a family of "secure" functions, such as
      <code>
       strcpy_s
      </code>
      and
      <code>
       strcat_s
      </code>
      (along with many others);
      <sup class="reference" id="cite_ref-89">
       <a href="#cite_note-89">
        <span>
         [
        </span>
        87
        <span>
         ]
        </span>
       </a>
      </sup>
      these functions were later standardized with some minor changes, and are now part of
      <a href="/wiki/C11_(C_standard_revision)" title="C11 (C standard revision)">
       C11 (Annex K)
      </a>
      and ISO/IEC WDTR 24731. These functions perform runtime integrity checks of their arguments; if the checks fail, a user-specified "runtime-constraint handler" function is called.
      <sup class="reference" id="cite_ref-90">
       <a href="#cite_note-90">
        <span>
         [
        </span>
        88
        <span>
         ]
        </span>
       </a>
      </sup>
      If the user has not specified such a function, the default behavior is implementation-defined.
      <sup class="reference" id="cite_ref-91">
       <a href="#cite_note-91">
        <span>
         [
        </span>
        89
        <span>
         ]
        </span>
       </a>
      </sup>
      Microsoft's C runtime will abort the program when the constraints are violated.
      <sup class="reference" id="cite_ref-92">
       <a href="#cite_note-92">
        <span>
         [
        </span>
        90
        <span>
         ]
        </span>
       </a>
      </sup>
      Some functions perform destructive operations before calling the runtime-constraint handler; for example,
      <code>
       strcat_s
      </code>
      sets the destination to the empty string,
      <sup class="reference" id="cite_ref-93">
       <a href="#cite_note-93">
        <span>
         [
        </span>
        91
        <span>
         ]
        </span>
       </a>
      </sup>
      which can make it difficult to recover from error conditions or debug them. These functions attracted considerable criticism because initially they were implemented only on Windows, and at the same time warning messages started to be produced by
      <a class="mw-redirect" href="/wiki/Microsoft_Visual_C%2B%2B" title="Microsoft Visual C++">
       Microsoft Visual C++
      </a>
      , suggesting the programmers to use these functions instead of standard ones. This has been speculated by some to be a Microsoft's attempt to lock developers into its platform.
      <sup class="reference" id="cite_ref-94">
       <a href="#cite_note-94">
        <span>
         [
        </span>
        92
        <span>
         ]
        </span>
       </a>
      </sup>
      Although open-source implementations of these functions are available,
      <sup class="reference" id="cite_ref-95">
       <a href="#cite_note-95">
        <span>
         [
        </span>
        93
        <span>
         ]
        </span>
       </a>
      </sup>
      these functions are not present in common Unix C libraries.
     </p>
     <p>
      More popular
      <sup class="reference" id="cite_ref-96">
       <a href="#cite_note-96">
        <span>
         [
        </span>
        a
        <span>
         ]
        </span>
       </a>
      </sup>
      <code>
       strlcat
      </code>
      and
      <code>
       strlcpy
      </code>
      functions date from 1999 or earlier; they have been criticized on the basis of encouraging the use of C strings and creating more problems than initially trying to solve.
      <sup class="reference" id="cite_ref-libc-alpha-discussion-1_97-0">
       <a href="#cite_note-libc-alpha-discussion-1-97">
        <span>
         [
        </span>
        94
        <span>
         ]
        </span>
       </a>
      </sup>
      <sup class="reference" id="cite_ref-98">
       <a href="#cite_note-98">
        <span>
         [
        </span>
        95
        <span>
         ]
        </span>
       </a>
      </sup>
      Consequently they have not been included in the
      <a class="mw-redirect" href="/wiki/GNU_C_library" title="GNU C library">
       GNU C library
      </a>
      (used by software on Linux), although they are implemented in
      <a href="/wiki/OpenBSD" title="OpenBSD">
       OpenBSD
      </a>
      ,
      <a href="/wiki/FreeBSD" title="FreeBSD">
       FreeBSD
      </a>
      ,
      <a href="/wiki/NetBSD" title="NetBSD">
       NetBSD
      </a>
      ,
      <a class="mw-redirect" href="/wiki/Solaris_Operating_System" title="Solaris Operating System">
       Solaris
      </a>
      ,
      <a class="mw-redirect" href="/wiki/Mac_OS_X" title="Mac OS X">
       Mac OS X
      </a>
      , and
      <a href="/wiki/QNX" title="QNX">
       QNX
      </a>
      . The lack of
      <a class="mw-redirect" href="/wiki/GNU_C_library" title="GNU C library">
       GNU C library
      </a>
      support has not stopped various library authors from using it and bundling a replacement, among other
      <a href="/wiki/Specification_and_Description_Language" title="Specification and Description Language">
       SDL
      </a>
      ,
      <a href="/wiki/GLib" title="GLib">
       GLib
      </a>
      ,
      <a href="/wiki/FFmpeg" title="FFmpeg">
       ffmpeg
      </a>
      ,
      <a href="/wiki/Rsync" title="Rsync">
       rsync
      </a>
      , and even internally in the
      <a href="/wiki/Linux_kernel" title="Linux kernel">
       Linux kernel
      </a>
      . Open source implementations for these functions are available.
      <sup class="reference" id="cite_ref-99">
       <a href="#cite_note-99">
        <span>
         [
        </span>
        96
        <span>
         ]
        </span>
       </a>
      </sup>
      <sup class="reference" id="cite_ref-100">
       <a href="#cite_note-100">
        <span>
         [
        </span>
        97
        <span>
         ]
        </span>
       </a>
      </sup>
     </p>
