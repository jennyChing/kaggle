						&lt;h1 id="firstHeading" class="firstHeading" lang="en"&gt;&lt;span dir="auto"&gt;Introduction to virtualization&lt;/span&gt;&lt;/h1&gt;
						&lt;div id="bodyContent" class="mw-body-content"&gt;
									&lt;div id="siteSub"&gt;From OpenVZ Virtuozzo Containers Wiki&lt;/div&gt;
								&lt;div id="contentSub"&gt;&lt;/div&gt;
												&lt;div id="jump-to-nav" class="mw-jump"&gt;
					Jump to:					&lt;a href="#mw-navigation"&gt;navigation&lt;/a&gt;, 					&lt;a href="#p-search"&gt;search&lt;/a&gt;
				&lt;/div&gt;
				&lt;div id="mw-content-text" lang="en" dir="ltr" class="mw-content-ltr"&gt;&lt;p&gt;Virtualization is a framework or methodology of dividing the resources of a computer into multiple execution environments. Virtualization techniques create multiple isolated partitions &#8212; Virtual Machines (VM) or &lt;a href="/Virtual_Environment" title="Virtual Environment" class="mw-redirect"&gt;Virtual Environments&lt;/a&gt; (&lt;a href="/VE" title="VE" class="mw-redirect"&gt;VEs&lt;/a&gt;) &#8212; on a single physical server. 
&lt;/p&gt;

&lt;h2&gt;&lt;span class="mw-headline" id="Techniques"&gt;Techniques&lt;/span&gt;&lt;/h2&gt;
&lt;p&gt;There are several kinds of virtualization techniques which provide similar features but differ in the degree of abstraction and the methods used for virtualization.
&lt;/p&gt;
&lt;h3&gt;&lt;span class="mw-headline" id="Virtual_machines_.28VMs.29"&gt;Virtual machines (VMs)&lt;/span&gt;&lt;/h3&gt;
&lt;p&gt;&lt;b&gt;Virtual machines&lt;/b&gt; emulate some real or fictional hardware, which in turn requires real resources from the &lt;i&gt;host&lt;/i&gt; (the machine running the VMs). This approach, used by most &lt;i&gt;system emulators&lt;/i&gt;, allows the emulator to run an arbitrary &lt;i&gt;guest operating system&lt;/i&gt; without modifications because guest OS is not aware that it is not running on real hardware. The main issue with this approach is that some CPU instructions require additional privileges and may not be executed in user space thus requiring a &lt;i&gt;virtual machines monitor&lt;/i&gt; (VMM), also called a hypervisor, to analyze executed code and make it safe on-the-fly. Hardware emulation approach is used by &lt;a rel="nofollow" class="external text" href="http://www.vmware.com/"&gt;VMware&lt;/a&gt; products, &lt;a rel="nofollow" class="external text" href="http://www.VirtualBox.org"&gt;VirtualBox&lt;/a&gt;, &lt;a rel="nofollow" class="external text" href="http://fabrice.bellard.free.fr/qemu/"&gt;QEMU&lt;/a&gt;, &lt;a rel="nofollow" class="external text" href="http://www.odin.com/"&gt;Odin&lt;/a&gt; and &lt;a rel="nofollow" class="external text" href="http://www.microsoft.com/windowsserversystem/virtualserver/default.mspx"&gt;Microsoft Virtual Server&lt;/a&gt;.
&lt;/p&gt;
&lt;h3&gt;&lt;span class="mw-headline" id="Paravirtualization"&gt;Paravirtualization&lt;/span&gt;&lt;/h3&gt;
&lt;p&gt;This technique also requires a VMM, but most of its work is performed in the &lt;i&gt;guest OS&lt;/i&gt; code, which in turn is &lt;i&gt;modified&lt;/i&gt; to support this VMM and avoid unnecessary use of privileged instructions. The paravirtualization technique also enables running different OSs on a single server, but requires them to be ported, i.e. they should know they are running under the hypervisor. The paravirtualization approach is used by products such as &lt;a rel="nofollow" class="external text" href="http://www.xen.org"&gt;Xen&lt;/a&gt; and &lt;a rel="nofollow" class="external text" href="http://user-mode-linux.sourceforge.net/"&gt;UML&lt;/a&gt;.
&lt;/p&gt;
&lt;h3&gt;&lt;span class="mw-headline" id="Virtualization_on_the_OS_level.2C_a.k.a._containers_virtualization"&gt;Virtualization on the OS level, a.k.a. containers virtualization&lt;/span&gt;&lt;/h3&gt;
&lt;p&gt;Most applications running on a server can easily share a machine with others, if they could be isolated and secured. Further, in most situations, different operating systems are not required on the same server, merely multiple instances of a single &lt;i&gt;operating system&lt;/i&gt;. OS-level virtualization systems have been designed to provide the required isolation and security to run multiple applications or copies of the same OS (but different distributions of the OS) on the same server. &lt;a rel="nofollow" class="external text" href="http://openvz.org/"&gt;OpenVZ&lt;/a&gt;, &lt;a rel="nofollow" class="external text" href="http://www.swsoft.com/products/virtuozzo"&gt;Virtuozzo&lt;/a&gt;, &lt;a rel="nofollow" class="external text" href="http://linux-vserver.org/"&gt;Linux-VServer&lt;/a&gt;, &lt;a rel="nofollow" class="external text" href="http://www.sun.com/bigadmin/content/zones/"&gt;Solaris Zones&lt;/a&gt; and &lt;a rel="nofollow" class="external text" href="http://www.freebsd.org/doc/en_US.ISO8859-1/books/arch-handbook/jail.html"&gt;FreeBSD Jails&lt;/a&gt; are examples of OS-level virtualization.
&lt;/p&gt;
&lt;h2&gt;&lt;span class="mw-headline" id="Short_comparison"&gt;Short comparison&lt;/span&gt;&lt;/h2&gt;
&lt;p&gt;The three techniques differ in complexity of implementation, breadth of OS support, performance in comparison with standalone server, and level of access to common resources. For example, VMs have wider scope of usage, but poorer performance. Para-VMs have better performance, but can support fewer OSs because one has to modify the original OS.
&lt;/p&gt;&lt;p&gt;Virtualization on the OS level provides the best performance and scalability compared to other approaches. Performance difference of such systems can be as low as 13%, comparing with that of a standalone server. &lt;a href="/Virtual_Environment" title="Virtual Environment" class="mw-redirect"&gt;Virtual Environments&lt;/a&gt; are usually also much simpler to administer as all of them can be accessed and administered from the host system. Generally, such systems are the best choice for server consolidation of same OS workloads.
&lt;/p&gt;
