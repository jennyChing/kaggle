        <h3>
         <span class="octicon octicon-book">
         </span>
         README.md
        </h3>
        <article class="markdown-body entry-content" itemprop="mainContentOfPage">
         <h1>
          <a aria-hidden="true" class="anchor" href="#procamsolver-01" id="user-content-procamsolver-01">
           <span class="octicon octicon-link">
           </span>
          </a>
          ProCamSolver 0.1
         </h1>
         <p>
          This is a C++ library for reconstructing a 3D scene from a number of 2D images of the scene. Using the library, the 3D coordinates of points in the scene can be calculated as well as the position/orientation/intrinsics of each camera. It is assumed that a set of pixel correspondences has already been calculated (that is, we have pairs of pixels - one from each of two different images - that correspond to the same 3D point).
         </p>
         <p>
          If there are enough 2D images of the same 3D scene then a set of pixel correspondences define a 3D reconstruction of the scene (the 'shape') and the position, orientation and intrinsics of each camera (the 'motion') up to a projective transofrmation. This library provides various methods of finding these and a convenient platform for developing new methods. It also provides methods of finding the projective transofrmation that leads to a Euclidean (zero skew) set of cameras.
         </p>
         <p>
          <strong>
           API documentation can be found
           <a href="http://danftang.github.com/ProCamSolver/">
            here
           </a>
           .
          </strong>
         </p>
         <p>
          <em>
           The code is pretty much working at the moment but
           <strong>
            isn't finished
           </strong>
           , I'll be tidying it up, tesing it, adding more documentation and more functionality over the next few weeks
          </em>
         </p>
         <h1>
          <a aria-hidden="true" class="anchor" href="#introduction" id="user-content-introduction">
           <span class="octicon octicon-link">
           </span>
          </a>
          Introduction
         </h1>
         <h2>
          <a aria-hidden="true" class="anchor" href="#quick-mathematical-background" id="user-content-quick-mathematical-background">
           <span class="octicon octicon-link">
           </span>
          </a>
          Quick mathematical background
         </h2>
         <p>
          For the mathematically minded, the problem can be expressed most succinctly in terms of three matrices: The 'shape matrix', the 'motion matrix' and the 'measurement matrix'. The shape matrix represents the 3D coordinates of the points in the scene in a 4xN matrix whose columns are of the form (x_n,y_n,z_n,1)^T, where N is the number of 3D points. The motion matrix consists of the camera projection matrices of each view placed one on top of the other in a 3Mx4 matrix, where M is the number of views. The measurement matrix is a representation of the pixel correspondences in a 3MxN matrix where each column represents to the correspondences of one 3D point, with the 2D pixel coordinates of each view placed one on top of the other. The pixel coordinates are of the form s(x,y,1)^T where s is a scalar called the 'projective depth'.
         </p>
         <p>
          In this form, these matrices have the simple relationship:
         </p>
         <p>
          E = P * S
         </p>
         <p>
          where
         </p>
         <p>
          E = measurement matrix
         </p>
         <p>
          P = motion matrix
         </p>
         <p>
          S = shape matrix
         </p>
         <p>
          The difficulty is that we don't know P, S or the projective depths of E. However, because P and S have only 4 columns/rows respectively, we know that E must be a rank 4 matrix, and this is often enough to give a unique solution.up to a projective transformation.
         </p>
         <h2>
          <a aria-hidden="true" class="anchor" href="#overview-of-methodology" id="user-content-overview-of-methodology">
           <span class="octicon octicon-link">
           </span>
          </a>
          Overview of methodology
         </h2>
         <ol>
          <li>
           Use the
           <strong>
            Measurement Matrix
           </strong>
           to find minimum
           <strong>
            Spanning Tree
           </strong>
           of views and create
           <strong>
            Fundamental Matrices
           </strong>
           for branches of the tree. Use Martinec Pajda's SVD method to approximate the
           <strong>
            Motion Matrix
           </strong>
           .
          </li>
          <li>
           Use the
           <strong>
            Motion Matrix
           </strong>
           from step 1 to fill in occlusions in the
           <strong>
            Measurement Matrix
           </strong>
           . Perform standard SVD on the
           <strong>
            Measurement Matrix
           </strong>
           to get the second level approximation of the
           <strong>
            Motion Matrix
           </strong>
           .
2b.
           <em>
            Euclidean Lift
           </em>
           the
           <strong>
            Motion Matrix
           </strong>
           .
          </li>
          <li>
           Use
           <em>
            Levenberg-Marquardt
           </em>
           to minimise the
           <strong>
            Reprojection Error
           </strong>
           which gives the final
           <strong>
            Motion Matrix
           </strong>
           .
          </li>
          <li>
           Use
           <em>
            KR Decomposition
           </em>
           to recover
           <strong>
            Intrinsics
           </strong>
           and
           <strong>
            Extrinsics
           </strong>
           .
          </li>
         </ol>
         <h2>
          <a aria-hidden="true" class="anchor" href="#usage" id="user-content-usage">
           <span class="octicon octicon-link">
           </span>
          </a>
          Usage
         </h2>
         <ol>
          <li>
           Fill the
           <em>
            Measurement Matrix
           </em>
           (e.g. with the
           <code>
            CorrespenceSet
           </code>
           class, the first set of indices are for cameras, whilst remaining indices are projectors).
          </li>
          <li>
           Use
           <code>
            MotionMatrix::svd_solve(const MeasurementMatrix &amp;)
           </code>
           to gain the first level approximation by minimising the
           <em>
            Frobenius Norm
           </em>
           . This step performs 4 things (as described
           <a href="http://danftang.github.com/ProCamSolver/classMotionMatrix.html#5a07d5b1459cb42e191a3892c9c122ea">
            here
           </a>
           ). This stage gives you a solution with an arbitrary projective transformation.
          </li>
          <li>
           Use
           <code>
            MotionMatrix::diac_euclidean_lift(int N)
           </code>
           , where N is the number of views which are cameras (i.e. not projectors), this transforms our matrices in the
           <em>
            Motion Matrix
           </em>
           into Euclidean transforms. Optionally this can also be performed with
           <code>
            MotionMatrix::diac_euclidean_lift(int N, ImageTransform&lt;F&gt; &amp;)
           </code>
           where we input F known intrinsics (i.e. for F cameras).
          </li>
          <li>
           Perform Bundle Adjustment using
           <code>
            MotionMatrix::bundle_adjust(int f, const MeasurementMatrix &amp;)
           </code>
           , this is a levmar fit to minimise
           <em>
            Reprojection Error
           </em>
           where f is the number of cameras.
           <strong>
            N.B. code also available to perform this stage with known intrinsics
           </strong>
           .
          </li>
          <li>
           (Optional) To create a 3D scan, create a
           <code>
            ShapeMatrix
           </code>
           and use
           <code>
            ShapeMatrix::solve(MeasurementMatrix &amp;, MotionMatrix &amp;)
           </code>
           .
          </li>
          <li>
           Use
           <code>
            MotionMatrix::KR_decompose(ImageTransform&lt;V&gt;&amp; K, ImageTransform&lt;V&gt;&amp; R)
           </code>
           to get
           <strong>
            Intrinsics
           </strong>
           (
           <code>
            K
           </code>
           ) and
           <strong>
            Extrinsics
           </strong>
           (
           <code>
            R
           </code>
           ) where
           <code>
            V
           </code>
           is view count, and use
           <code>
            MotionMatrix::T(i)
           </code>
           to get the translation in projection space, use
           <code>
            K.view(i).inverse() * R.view(i).inverse() * MotionMatrix::T(i)
           </code>
           to recover the world translation of the view.
          </li>
         </ol>
         <h1>
          <a aria-hidden="true" class="anchor" href="#further-work" id="user-content-further-work">
           <span class="octicon octicon-link">
           </span>
          </a>
          Further work
         </h1>
         <h2>
          <a aria-hidden="true" class="anchor" href="#ransac-implementation" id="user-content-ransac-implementation">
           <span class="octicon octicon-link">
           </span>
          </a>
          RANSAC implementation
         </h2>
         <p>
          Implement RANSAC into FundamentalMatrix class which will solve against subsets of the MeasurementMatrix, and then mask the MeasurementMatrix (by deletion of datapoints) for further calculation downstream.
         </p>
         <p>
          RANSAC is best performed on step 1 (Fundamental Matrix) of
          <a href="http://danftang.github.com/ProCamSolver/classMotionMatrix.html#5a07d5b1459cb42e191a3892c9c122ea">
           here
          </a>
          . Svoboda et al (A Convenient Multi-Camera Self-Calibration for Virtual Environment) also perform this on the other steps
         </p>
         <h2>
          <a aria-hidden="true" class="anchor" href="#better-sparse-matrix-methods" id="user-content-better-sparse-matrix-methods">
           <span class="octicon octicon-link">
           </span>
          </a>
          Better sparse matrix methods
         </h2>
         <p>
          With large datasets, we should be selecting a good subset to work with which is an involved process. A strategy would need to be made to find the best way of summarising the data without losing important features.
         </p>
         <h2>
          <a aria-hidden="true" class="anchor" href="#use-the-sparse-matrix-in-the-bundle-adjustment" id="user-content-use-the-sparse-matrix-in-the-bundle-adjustment">
           <span class="octicon octicon-link">
           </span>
          </a>
          Use the sparse matrix in the bundle adjustment
         </h2>
         <p>
          Currently we're filling occlusions before this stage by simulating the 3d points.
         </p>
         <h2>
          <a aria-hidden="true" class="anchor" href="#standalone-app" id="user-content-standalone-app">
           <span class="octicon octicon-link">
           </span>
          </a>
          Standalone app
         </h2>
         <p>
          Since we template against number of views, we could create version of the classes for 3-10 views for example (3 views is the minimum requirement).
         </p>
