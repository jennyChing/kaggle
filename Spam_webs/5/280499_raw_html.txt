             <h3 class="post-title entry-title">
              <a href="http://www.giorgiosironi.com/2010/02/repositories-in-doctrine.html">
               Repositories in Doctrine
              </a>
             </h3>
             <div class="post-header-line-1">
             </div>
             <div class="post-body entry-content">
              <a href="http://www.doctrine-project.org/documentation/manual/2_0/en/working-with-objects" target="_blank">
              </a>
              <i>
               Jebb
              </i>
              wrote to me asking information about the Repository pattern:
              <br/>
              <blockquote>
               I read your blog at
               <a href="http://giorgiosironi.blogspot.com/" target="_blank">
                http://giorgiosironi.blogspot.
                <wbr>
                </wbr>
                com/
               </a>
               and I found it very helpful. I develop applications using DDD and Zend Framework however for persistence ignorance, I use some basic Repository as interface for DAOs. May I ask you how to implement Doctrine ORM in the Repository?
              </blockquote>
              As I explain in depth in
              <a href="http://giorgiosironi.blogspot.com/2009/10/repository-pattern.html">
               my previous post about this pattern
              </a>
              , a Repository is an illusion of an in-memory collection of all persistent ignorant entities of the same class. A Repository may be abstracted beyond an interface defining only the mandatory methods that make sense in the domain model.
              <br/>
              How an Orm fits in this discussion? Very few repositories accomplish their tasks alone: most of them compose an
              <b>
               underlying generic Data Mapper layer
              </b>
              which is usually an Object-Relational Mapper which translates classes and objects in tables and rows. A generic Data Mapper such as Doctrine 2 provides all the possibly imaginable operations on the collection of entities, and the Repository abstraction decouples the other parts of the application from knowing anything about an Orm. This is the primary advantage of a Repository.
              <br/>
              <br/>
              As you may have thought at the
              <b>
               persistence ignorance
              </b>
              reference, with Doctrine 1 there are no chances to implement a real Repository. Doctrine 1 does not provide persistence ignorance because it requires entity classes to extend a base abstract Active Record.
              <br/>
              In Doctrine 2, repositories can be implemented as Plain Old Php Objects: simply injecting the EntityManager and other collaborators they may need in the constructor will suffice. A Factory can then encapsulate the
              <i>
               new
              </i>
              operators.
              <br/>
              This is the standard Repository pattern: a Plain Old Php Object which aggregates whatever is needed to hide the persistence mechanism of objects.
              <br/>
              <br/>
              Doctrine 2 also provides a facility to quickly implement Repositories, but the freedom of implementation of this approach is limited. The process is described in
              <a href="http://www.doctrine-project.org/documentation/manual/2_0/en/working-with-objects">
               the Doctrine manual
              </a>
              and it consists in:
              <br/>
              <ul>
               <li>
                extending an abstract class, which has a protected member $this-&gt;_em you can execute queries with;
               </li>
               <li>
                annotating the entity class with the class name of the concrete repository class;
               </li>
               <li>
                obtaining an instance with $em-&gt;getRepository('EntityClassName'): the EntityManager will create the object and automatically inject itself in the Repository.
               </li>
              </ul>
              Some notes on this architecture:
              <br/>
              <ul>
               <li>
                there is
                <b>
                 no entity-specific interface
                </b>
                ;
               </li>
               <li>
                you
                <b>
                 extend an abstract class
                </b>
                , which may be a cling because you want only some find*() methods to be available. However testing won't be affected since a concrete Repository will always involve a persistence mechanism.
               </li>
               <li>
                it would be useful for
                <b>
                 having different repositories referencing each other
                </b>
                via the EntityManager; but you may manually inject them instead, maintaining the original solution.
               </li>
               <li>
                Doctrine 2
                <b>
                 default Repository class
                </b>
                is instantiated instead if you do not specify a subclass; if you use a lot of repositories and inject some of them in other service classes, the default implementation will be very handy. This would be a pro for taking advantage of the Orm support instead of using POPOs.
               </li>
              </ul>

                underlying generic Data Mapper layer
               </b>
               which is usually an Object-Relational Mapper which translates classes and objects in tables and rows. A generic Data Mapper such as Doctrine 2 provides all the possibly imaginable operations on the collection of entities, and the Repository abstraction decouples the other parts of the application from knowing anything about an Orm. This is the primary advantage of a Repository.
               <br/>
               <br/>
               As you may have thought at the
               <b>
                persistence ignorance
               </b>
               reference, with Doctrine 1 there are no chances to implement a real Repository. Doctrine 1 does not provide persistence ignorance because it requires entity classes to extend a base abstract Active Record.
               <br/>
               In Doctrine 2, repositories can be implemented as Plain Old Php Objects: simply injecting the EntityManager and other collaborators they may need in the constructor will suffice. A Factory can then encapsulate the
               <i>
                new
               </i>
               operators.
               <br/>
               This is the standard Repository pattern: a Plain Old Php Object which aggregates whatever is needed to hide the persistence mechanism of objects.
               <br/>
               <br/>
               Doctrine 2 also provides a facility to quickly implement Repositories, but the freedom of implementation of this approach is limited. The process is described in
               <a href="http://www.doctrine-project.org/documentation/manual/2_0/en/working-with-objects">
                the Doctrine manual
               </a>
               and it consists in:
               <br/>
               <ul>
                <li>
                 extending an abstract class, which has a protected member $this-&gt;_em you can execute queries with;
                </li>
                <li>
                 annotating the entity class with the class name of the concrete repository class;
                </li>
                <li>
                 obtaining an instance with $em-&gt;getRepository('EntityClassName'): the EntityManager will create the object and automatically inject itself in the Repository.
                </li>
               </ul>
               Some notes on this architecture:
               <br/>
               <ul>
                <li>
                 there is
                 <b>
                  no entity-specific interface
                 </b>
                 ;
                </li>
                <li>
                 you
                 <b>
                  extend an abstract class
                 </b>
                 , which may be a cling because you want only some find*() methods to be available. However testing won't be affected since a concrete Repository will always involve a persistence mechanism.
                </li>
                <li>
                 it would be useful for
                 <b>
                  having different repositories referencing each other
                 </b>
                 via the EntityManager; but you may manually inject them instead, maintaining the original solution.
                </li>
                <li>
                 Doctrine 2
                 <b>
                  default Repository class
                 </b>
                 is instantiated instead if you do not specify a subclass; if you use a lot of repositories and inject some of them in other service classes, the default implementation will be very handy. This would be a pro for taking advantage of the Orm support instead of using POPOs.
                </li>
               </ul>
              </div>

               Repositories in Doctrine
