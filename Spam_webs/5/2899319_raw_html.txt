     <p>
      In the field of
      <a href="/wiki/Database" title="Database">
       databases
      </a>
      in
      <a href="/wiki/Computer_science" title="Computer science">
       computer science
      </a>
      , a
      <b>
       transaction log
      </b>
      (also
      <b>
       transaction journal
      </b>
      ,
      <b>
       database log
      </b>
      ,
      <b>
       binary log
      </b>
      or
      <b>
       audit trail
      </b>
      ) is a history of actions executed by a
      <a class="mw-redirect" href="/wiki/Database_management_system" title="Database management system">
       database management system
      </a>
      to guarantee
      <a href="/wiki/ACID" title="ACID">
       ACID
      </a>
      properties over
      <a href="/wiki/Crash_(computing)" title="Crash (computing)">
       crashes
      </a>
      or hardware failures. Physically, a log is a
      <a href="/wiki/Computer_file" title="Computer file">
       file
      </a>
      listing changes to the database, stored in a stable storage format.
     </p>
     <p>
      If, after a start, the database is found in an
      <a href="/wiki/Consistency_(database_systems)" title="Consistency (database systems)">
       inconsistent
      </a>
      state or not been shut down properly, the database management system reviews the database logs for
      <a href="/wiki/Commit_(data_management)" title="Commit (data management)">
       uncommitted
      </a>
      transactions and
      <a href="/wiki/Rollback_(data_management)" title="Rollback (data management)">
       rolls back
      </a>
      the changes made by these
      <a href="/wiki/Database_transaction" title="Database transaction">
       transactions
      </a>
      . Additionally, all transactions that are already committed but whose changes were not yet materialized in the database are re-applied. Both are done to ensure
      <a href="/wiki/Atomicity_(database_systems)" title="Atomicity (database systems)">
       atomicity
      </a>
      and
      <a class="mw-redirect" href="/wiki/Durability_(computer_science)" title="Durability (computer science)">
       durability
      </a>
      of transactions.
     </p>
     <p>
      This term is not to be confused with other, human-readable
      <a class="mw-redirect" href="/wiki/Data_logging#Computer_data_logging" title="Data logging">
       logs
      </a>
      that a database management system usually provides.
     </p>
     <p>
      In
      <a class="mw-redirect" href="/wiki/Computer_storage" title="Computer storage">
       computer storage
      </a>
      , a
      <b>
       journal
      </b>
      is a
      <a href="/wiki/Chronology" title="Chronology">
       chronological
      </a>
      record of
      <a href="/wiki/Data_processing" title="Data processing">
       data processing
      </a>
      operations that may be used to construct or reinstate an historical or alternative version of a
      <a class="mw-redirect" href="/wiki/Computer_system" title="Computer system">
       computer system
      </a>
      or
      <a href="/wiki/Computer_file" title="Computer file">
       computer file
      </a>
      .
     </p>
     <p>
      In
      <a class="mw-redirect" href="/wiki/Database_management_system" title="Database management system">
       database management systems
      </a>
      , a journal is the record of data altered by a given process.
     </p>

     <h2>
      <span class="mw-headline" id="Anatomy_of_a_general_database_log">
       Anatomy of a general database log
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Transaction_log&amp;action=edit&amp;section=1" title="Edit section: Anatomy of a general database log">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      A database log record is made up of:
     </p>
     <ul>
      <li>
       <i>
        Log Sequence Number
       </i>
       : A unique id for a log record. With LSNs, logs can be recovered in constant time. Most logs' LSNs are assigned in monotonically increasing order, which is useful in recovery
       <a href="/wiki/Algorithm" title="Algorithm">
        algorithms
       </a>
       , like
       <a href="/wiki/Algorithms_for_Recovery_and_Isolation_Exploiting_Semantics" title="Algorithms for Recovery and Isolation Exploiting Semantics">
        ARIES
       </a>
       .
      </li>
      <li>
       <i>
        Prev LSN
       </i>
       : A link to their last log record. This implies database logs are constructed in
       <a href="/wiki/Linked_list" title="Linked list">
        linked list
       </a>
       form.
      </li>
      <li>
       <i>
        Transaction ID number
       </i>
       : A reference to the database transaction generating the log record.
      </li>
      <li>
       <i>
        Type
       </i>
       : Describes the type of database log record.
      </li>
      <li>
       Information about the actual changes that triggered the log record to be written.
      </li>
     </ul>
     <h2>
      <span class="mw-headline" id="Types_of_database_log_records">
       Types of database log records
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Transaction_log&amp;action=edit&amp;section=2" title="Edit section: Types of database log records">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      All log records include the general log attributes above, and also other attributes depending on their type (which is recorded in the
      <i>
       Type
      </i>
      attribute, as above).
     </p>
     <ul>
      <li>
       <b>
        Update Log Record
       </b>
       notes an update (change) to the database. It includes this extra information:
       <ul>
        <li>
         <i>
          PageID
         </i>
         : A reference to the Page ID of the modified page.
        </li>
        <li>
         <i>
          Length and Offset
         </i>
         : Length in bytes and offset of the page are usually included.
        </li>
        <li>
         <i>
          Before and After Images
         </i>
         : Includes the value of the bytes of page before and after the page change. Some databases may have logs which include one or both images.
        </li>
       </ul>
      </li>
     </ul>
     <ul>
      <li>
       <b>
        Compensation Log Record
       </b>
       notes the rollback of a particular change to the database. Each correspond with exactly one other Update Log Record (although the corresponding update log record is not typically stored in the Compensation Log Record). It includes this extra information:
       <ul>
        <li>
         <i>
          undoNextLSN
         </i>
         : This field contains the LSN of the next log record that is to be undone for transaction that wrote the last Update Log.
        </li>
       </ul>
      </li>
     </ul>
     <ul>
      <li>
       <b>
        Commit Record
       </b>
       notes a decision to commit a transaction.
      </li>
     </ul>
     <ul>
      <li>
       <b>
        Abort Record
       </b>
       notes a decision to abort and hence roll back a transaction.
      </li>
     </ul>
     <ul>
      <li>
       <b>
        Checkpoint Record
       </b>
       notes that a checkpoint has been made. These are used to speed up recovery. They record information that eliminates the need to read a long way into the log's past. This varies according to checkpoint algorithm. If all dirty pages are flushed while creating the checkpoint (as in
       <a href="/wiki/PostgreSQL" title="PostgreSQL">
        PostgreSQL
       </a>
       ), it might contain:
       <ul>
        <li>
         <i>
          redoLSN
         </i>
         : This is a reference to the first log record that corresponds to a dirty page. i.e. the first update that wasn't flushed at checkpoint time. This is where redo must begin on recovery.
        </li>
        <li>
         <i>
          undoLSN
         </i>
         : This is a reference to the oldest log record of the oldest in-progress transaction. This is the oldest log record needed to undo all in-progress transactions.
        </li>
       </ul>
      </li>
     </ul>
     <ul>
      <li>
       <b>
        Completion Record
       </b>
       notes that all work has been done for this particular transaction. (It has been fully committed or aborted)
      </li>
     </ul>
     <h2>
      <span class="mw-headline" id="Tables">
       Tables
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Transaction_log&amp;action=edit&amp;section=3" title="Edit section: Tables">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      These tables are maintained in memory, and can be efficiently reconstructed (if not exactly, to an equivalent state) from the log and the database:
     </p>
     <ul>
      <li>
       <b>
        Transaction Table
       </b>
       : The table contains one entry for each active transaction. This includes Transaction ID and lastLSN, where lastLSN describes the LSN of the most recent log record for the transaction.
      </li>
     </ul>
     <ul>
      <li>
       <b>
        Dirty Page Table
       </b>
       : The table contains one entry for each dirty page that hasn't been written to disk. The entry contains recLSN, where recLSN is the LSN of the first log record that caused the page to be dirty.
      </li>
     </ul>
     <ul>
      <li>
       <b>
        Transaction Log
       </b>
       : A DBMS uses a transaction log to keep track of all transactions that updates the database. The information stored in this log is used by DBMS for a recovery requirement triggered by 'Roll Back' statement.
      </li>
     </ul>
