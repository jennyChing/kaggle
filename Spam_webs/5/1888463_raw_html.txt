      <p>
       The Windows 8
       <a class="term1 piterm" href="/_iframe_term.php?seek=TCP/IP" rel="sg-term" title="What is TCP/IP ?">
        TCP/IP
       </a>
       implementation shares many traits with earlier Windows versions, however, there are some subtle differences, new syntax for applying tweaks, and some new settings. Windows 8 generally works well by default for many
       <a class="term1 piterm" href="/_iframe_term.php?seek=BROADBAND" rel="sg-term" title="What is broadband ?">
        broadband
       </a>
       connections, yet there is always some room for improvement and we will try to cover all relevant settings and their effect on your connection in detail below.
      </p>
      <p>
       To run some of the outlined commands, you will need to use an administrator account. With earlier Windows versions, we used the netsh command in
       <em>
        elevated command prompt
       </em>
       to tweak most of the
       <a class="term1 piterm" href="/_iframe_term.php?seek=TCP/IP" rel="sg-term" title="What is TCP/IP ?">
        TCP/IP
       </a>
       settings. With the introduction of Windows 8/Server 2012, some of those parameters are only tweakable by using the
       <em>
        Windows PowerShell,
       </em>
       that is why we will be using it for settings when possible. Note that netsh and other DOS commands are still available under
       <em>
        PowerShell
       </em>
       , just as with
       <em>
        Command Prompt
       </em>
       .
      </p>
      <p>
       The settings outlined bellow work with Windows 8, 8.1, Windows 10 Technical Preview, Windows Server 2012, and 2012 R2 (with only minor syntax differences). If you'd rather use our free software to apply all those settings, try the
       <a href="http://www.speedguide.net/downloads.php">
        SG TCP Optimizer
       </a>
       (v.4).
      </p>
      <h2>
       Windows PowerShell
      </h2>
      <p>
       The
       <em>
        Windows PowerShell
       </em>
       is a tool developed by Microsoft designed to replace and extend the functionality of the
       <em>
        command prompt
       </em>
       . It allows you to run all DOS commands similarly to
       <em>
        command prompt
       </em>
       , however, it also provides additional functionality in the form of many customizable commands called
       <em>
        cmdlets
       </em>
       . We will be using
       <em>
        PowerShell
       </em>
       to tweak
       <a class="term1 piterm" href="/_iframe_term.php?seek=TCP/IP" rel="sg-term" title="What is TCP/IP ?">
        TCP/IP
       </a>
       settings, as some of the parameters are no longer tweakable using
       <em>
        command prompt netsh
       </em>
       in Windows 8 and newer OSes.
      </p>
      <p>
       In Windows 8, the easiest way to find
       <em>
        PowerShell
       </em>
       is by searching for PowerShell on your
       <em>
        Start
       </em>
       screen/menu. It can also be accessed from the
       <em>
        Windows System
       </em>
       folder. You should right-click on
       <em>
        PowerShell
       </em>
       and choose "Run as administrator" so that you have sufficient permissions to tweak all applicable settings.
      </p>
      <p>
       The PowerShell commands below are an overview of what we can use to tweak TCP/IP settings under Windows 8:
      </p>
      <p style="margin-left: 20px;">
       <strong>
        Get-NetTCPSetting
       </strong>
       - this can be used to view TCP parameters
       <strong>
        <br/>
        Set-NetTCPSetting
       </strong>
       - this can be used to set TCP parameters
      </p>
      <p style="margin-left: 20px;">
       <em>
        netsh int tcp show global
       </em>
       - you can still see and change some (not all) parameters using netsh
      </p>
      <p>
       <a class="piframe" href="/_iframe_showpic.php?img=/images/pics/reviews/5077_powershell-get-nettcpsetting.png" rel="sg-set">
        <img alt="" class="piframe-left" src="/img.php?x=200&amp;y=116&amp;img=/images/pics/reviews/5077_powershell-get-nettcpsetting.png" title="Click to expand" width="200"/>
       </a>
       <a class="piframe" href="/_iframe_showpic.php?img=/images/pics/reviews/5077_powershell-netsh-win81.png" rel="sg-set">
        <img alt="" class="piframe-left" src="/img.php?x=200&amp;y=116&amp;img=/images/pics/reviews/5077_powershell-netsh-win81.png" title="Click to expand" width="200"/>
       </a>
      </p>
      <p>
       <br/>
      </p>
      <p>
       <br/>
      </p>
      <p>
       <br/>
      </p>

       <br/>

       <br/>
      </p>
      <p>
       There are a few different networking profiles called "templates" in Windows 8 PowerShell under Get-NetTCPSetting.
       <br/>
       In Windows 8 (and 2012 Server): Automatic, Internet, Datacenter, Compat
       <br/>
       In Windows 8.1 (also Windows 10, Windows 2012 Server R2): Automatic, Internet, InternetCustom, Datacenter, DataCenterCustom, Compat

      <h2>
       Receive Window Auto-Tuning Level
      </h2>
      <p>
       The
       <a class="term1 piterm" href="/_iframe_term.php?seek=TCP RECEIVE WINDOW" rel="sg-term" title="What is TCP Receive Window ?">
        TCP Receive Window
       </a>
       has a very important role in TCP connections. It can limit
       <a class="term1 piterm" href="/_iframe_term.php?seek=THROUGHPUT" rel="sg-term" title="What is throughput ?">
        throughput
       </a>
       , especially in high-speed, high-
       <a class="term1 piterm" href="/_iframe_term.php?seek=LATENCY" rel="sg-term" title="What is latency ?">
        latency
       </a>
       environments, such as most internet connections. For more information on how it limits your speed, check our
       <a href="http://www.speedguide.net/articles/the-tcp-window-latency-and-the-bandwidth-delay-2678" target="_blank">
        TCP Window article
       </a>
       .
      </p>
      <p>
       The possible settings are as follows:
      </p>
      <p style="margin-left: 20px;">
       disabled: uses a fixed value for the
       <a class="term1 piterm" href="/_iframe_term.php?seek=TCP RECEIVE WINDOW" rel="sg-term" title="What is tcp receive window ?">
        tcp receive window
       </a>
       . Limits it to 64KB (limited at 65535).
       <br/>
       highlyrestricted: allows the receive window to grow beyond its default value, very conservatively
       <br/>
       restricted: somewhat restricted growth of the
       <a class="term1 piterm" href="/_iframe_term.php?seek=TCP RECEIVE WINDOW" rel="sg-term" title="What is tcp receive window ?">
        tcp receive window
       </a>
       beyond its default value
       <br/>
       normal: default value, allows the receive window to grow to accommodate most conditions
       <br/>
       experimental: allows the receive window to grow to accommodate extreme scenarios (not recommended, it can degrade performance in common scenarios, only intended for research purposes. It enables
       <a class="term1 piterm" href="/_iframe_term.php?seek=RWIN" rel="sg-term" title="What is RWIN ?">
        RWIN
       </a>
       values of over 16 MB)
      </p>
      <p>
       Default setting: normal
       <br/>
       Our recommendation: normal (unless you're experiencing problems)
      </p>
      <p>
       To modify, in
       <em>
        PowerShell
       </em>
       (or elevated command prompt) type:
      </p>
      <p style="margin-left: 20px;">
       <strong>
        netsh int tcp set global autotuninglevel=normal
       </strong>
       <br/>
       (where "normal" is one of the above listed possible states)
      </p>
      <p>
       Same can also be accomplished using
       <em>
        PowerShell
       </em>
       cmdlets, just type:
      </p>
      <p style="margin-left: 20px;">
       <strong>
        Set-NetTCPSetting -SettingName InternetCustom -AutoTuningLevelLocal Normal
       </strong>
      </p>
      <p>
       <br/>
      </p>
      <p>
       Note: You can also try "highlyrestricted" autotuninglevel for up to 10Mbps connections, as it actually uses a higher unscaled
       <a class="term1 piterm" href="/_iframe_term.php?seek=RWIN" rel="sg-term" title="What is RWIN ?">
        RWIN
       </a>
       value (16384 vs. 256 bytes), and still scales up to 262140 bytes.
      </p>

       <br/>
      </p>
      <h2>
       Disable Windows Scaling heuristics
      </h2>
      <p>
       Windows 8 (as with Windows 7) has the ability to automatically change its own TCP Window auto-tuning behavior to a more conservative state regardless of any user settings. When heuristics restricts your autotuning level, you may see this message when viewing netsh settings:
      </p>
      <p>
       "** The above autotuninglevel setting is the result of Windows Scaling heuristics overriding any local/policy configuration on at least one profile."
      </p>
      <p>
       When heuristics restricts autotuning level, the "netsh int tcp show global" command will still (incorrectly) show your user-set autotuning level, you have to use "netsh int tcp show heuristics" to see the actual current heuristics restriction. To enforce the user-set
       <a class="term1 piterm" href="/_iframe_term.php?seek=TCP RECEIVE WINDOW" rel="sg-term" title="What is TCP Receive Window ?">
        TCP Receive Window
       </a>
       autotuning level and prevent heuristics from restricting how the
       <a class="term1 piterm" href="/_iframe_term.php?seek=TCP RECEIVE WINDOW" rel="sg-term" title="What is TCP Receive Window ?">
        TCP Receive Window
       </a>
       grows, disable heuristics. It is best to disable this before applying autotuning level to ensure your user-set autotuning level is retained.
      </p>
      <p>
       default state: OS-dependent (disabled in Windows 8.1, enabled in Windows 7)
       <br/>
       possible settings are: disabled,enabled,default (sets to the Windows default state)
       <br/>
       recommended: disabled (to retain user-set RWIN auto-tuning level)
      </p>
      <p>
       To disable explicitly using netsh:
      </p>
      <p style="margin-left: 20px;">
       <strong>
        netsh int tcp set heuristics disabled
       </strong>
      </p>
      <p>
       To see the current heuristics state:
      </p>
      <p style="margin-left: 20px;">
       netsh int tcp show heuristics
      </p>
      <p>
       Using PowerShell cmdlets:
      </p>
      <p style="margin-left: 20px;">
       Set-NetTCPSetting -SettingName InternetCustom -ScalingHeuristics Disabled
      </p>
      <p>
       Notes:
       <br/>
       If heuristics are not explicitly disabled, Windows can restrict the
       <a class="term1 piterm" href="/_iframe_term.php?seek=RWIN" rel="sg-term" title="What is RWIN ?">
        RWIN
       </a>
       autotuninglevel at any time. When that happens, viewing your settings with
       <em>
        "netsh int tcp show global"
       </em>
       will still incorrectly show user-set autotuninglevel, only
       <em>
        "netsh int tcp show heuristics"
       </em>
       reveals the restriction.
      </p>
      <p>
       The registry value that was used in Windows 7 and earlier is no longer present in the Windows Registry (
       <strike>
        HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\services\Tcpip\Parameters\EnableWsd=0
       </strike>
       (not present for Windows 8))
      </p>
      <p>
       <br/>
      </p>
      <h2>
       Add-On Congestion Control Provider (
       <a class="term1 piterm" href="/_iframe_term.php?seek=CTCP" rel="sg-term" title="What is CTCP ?">
        CTCP
       </a>
       )
      </h2>
      <p>
       The traditional slow-start and congestion avoidance algorithms in TCP help avoid network congestion by gradually increasing the TCP window at the beginning of transfers until the
       <a class="term1 piterm" href="/_iframe_term.php?seek=TCP RECEIVE WINDOW" rel="sg-term" title="What is TCP Receive Window ?">
        TCP Receive Window
       </a>
       boundary is reached, or
       <a class="term1 piterm" href="/_iframe_term.php?seek=PACKET LOSS" rel="sg-term" title="What is packet loss ?">
        packet loss
       </a>
       occurs. For
       <a class="term1 piterm" href="/_iframe_term.php?seek=BROADBAND" rel="sg-term" title="What is broadband ?">
        broadband
       </a>
       internet connections that combine high TCP Window with higher
       <a class="term1 piterm" href="/_iframe_term.php?seek=LATENCY" rel="sg-term" title="What is latency ?">
        latency
       </a>
       (high
       <a class="term1 piterm" href="/_iframe_term.php?seek=BDP" rel="sg-term" title="What is BDP ?">
        BDP
       </a>
       ), these algorithms do not increase the TCP windows fast enough to fully utilize the
       <a class="term1 piterm" href="/_iframe_term.php?seek=BANDWIDTH" rel="sg-term" title="What is bandwidth ?">
        bandwidth
       </a>
       of the connection.
      </p>
      <p>
       Compound TCP (
       <a class="term1 piterm" href="/_iframe_term.php?seek=CTCP" rel="sg-term" title="What is CTCP ?">
        CTCP
       </a>
       ) is a newer congestion control method that increases the TCP Send Window more aggressively for
       <a class="term1 piterm" href="/_iframe_term.php?seek=BROADBAND" rel="sg-term" title="What is broadband ?">
        broadband
       </a>
       connections (with large RWIN and BDP).
       <a class="term1 piterm" href="/_iframe_term.php?seek=CTCP" rel="sg-term" title="What is CTCP ?">
        CTCP
       </a>
       attempts to maximize
       <a class="term1 piterm" href="/_iframe_term.php?seek=THROUGHPUT" rel="sg-term" title="What is throughput ?">
        throughput
       </a>
       by monitoring delay variations and
       <a class="term1 piterm" href="/_iframe_term.php?seek=PACKET LOSS" rel="sg-term" title="What is packet loss ?">
        packet loss
       </a>
       . It also ensures that its behavior does not impact other TCP connections negatively.
       <a class="term1 piterm" href="/_iframe_term.php?seek=CTCP" rel="sg-term" title="What is CTCP ?">
        CTCP
       </a>
       is turned on in Windows 8, and Server editions by default, but off in Windows 7. Keeping
       <a class="term1 piterm" href="/_iframe_term.php?seek=CTCP" rel="sg-term" title="What is CTCP ?">
        CTCP
       </a>
       on can significantly increase
       <a class="term1 piterm" href="/_iframe_term.php?seek=THROUGHPUT" rel="sg-term" title="What is throughput ?">
        throughput
       </a>
       and
       <a class="term1 piterm" href="/_iframe_term.php?seek=PACKET LOSS" rel="sg-term" title="What is packet loss ?">
        packet loss
       </a>
       recovery.
      </p>
      <p>
       Possible settings are: none,
       <a class="term1 piterm" href="/_iframe_term.php?seek=CTCP" rel="sg-term" title="What is ctcp ?">
        ctcp
       </a>
       , dctcp (Datacenter TCP)
       <br/>
       Default :
       <a class="term1 piterm" href="/_iframe_term.php?seek=CTCP" rel="sg-term" title="What is CTCP ?">
        CTCP
       </a>
       ** (netsh incorrectly shows "none" by default, while PowerShell Get-NetTCPSetting shows "CTCP". netsh is deprecated and incorrect)
       <span style="line-height: normal;">
        <br/>
        Recommended:
        <a class="term1 piterm" href="/_iframe_term.php?seek=CTCP" rel="sg-term" title="What is CTCP ?">
         CTCP
        </a>
       </span>
      </p>
      <p>
       <a class="term1 piterm" href="/_iframe_term.php?seek=CTCP" rel="sg-term" title="What is CTCP ?">
        CTCP
       </a>
       - Compound TCP increases the receive window and amount of data sent. It can improve
       <a class="term1 piterm" href="/_iframe_term.php?seek=THROUGHPUT" rel="sg-term" title="What is throughput ?">
        throughput
       </a>
       on higher
       <a class="term1 piterm" href="/_iframe_term.php?seek=LATENCY" rel="sg-term" title="What is latency ?">
        latency
       </a>
       /broadband internet connections.
       <br/>
       DCTCP - Data Center TCP adjusts the TCP Window based on network congestion feedback based on
       <em>
        Explicit Congestion Notification (ECN)
       </em>
       signaling, it is designed to improve
       <a class="term1 piterm" href="/_iframe_term.php?seek=THROUGHPUT" rel="sg-term" title="What is throughput ?">
        throughput
       </a>
       on low
       <a class="term1 piterm" href="/_iframe_term.php?seek=LATENCY" rel="sg-term" title="What is latency ?">
        latency
       </a>
       /local links.
      </p>
      <p>
       Unfortunately, netsh is bugged for this setting, and the PowerShell cmdlet only works for server Windows variants, so there is no longer an easy one line command that can change the congestion provider. Yay for Microsoft! Fortunately, the default setting is already
       <a class="term1 piterm" href="/_iframe_term.php?seek=CTCP" rel="sg-term" title="What is CTCP ?">
        CTCP
       </a>
       , as per the
       <em>
        PowerShell
       </em>
       <strong>
        Get-NetTCPSetting
       </strong>
       cmdlet.
      </p>
      <p>
       To change using
       <em>
        PowerShell
       </em>
       cmdlet:
      </p>
      <p style="margin-left: 20px;">
       <strong>
        Set-NetTCPSetting -SettingName InternetCustom -CongestionProvider CTCP
       </strong>
      </p>
      <p style="margin-left: 20px;">
       ("InternetCustom" template is for Windows 8.1, change to "Custom" for Windows 8)
      </p>
      <p>
       To see the current Congestion provider, use:
       <em>
        Get-NetTcpSetting
       </em>
       <br/>
       Alternatively, you can go wild and use the cmdlet below to see only the congestion provider of the currently used template:
       <br/>
       <em>
        (Get-NetTransportFilter | Where DestinationPrefix -eq '*' | Get-NetTCPSetting).CongestionProvider
       </em>
      </p>

      <h2>
       TCP Chimney Offload
      </h2>
      <p>
       TCP chimney offload enables Windows to offload all TCP processing for a connection to a network adapter (with proper driver support). Offloads are initiated on a per-connection basis and reduce networking-related CPU overhead, theoretically enabling better overall system performance by freeing up CPU time for other tasks. TCP Chimney Offload controls what is referred to as the "TCP Offload Engine" (TOE) in Windows in general. It is a global setting that has to be enabled for many of the other offloads to work. Enabling this setting had some negative effects in the past because of buggy network adapter drivers, however its implementation has gotten much better with time. It is useful for CPU-bound client computers and very fast
       <a class="term1 piterm" href="/_iframe_term.php?seek=BROADBAND" rel="sg-term" title="What is broadband ?">
        broadband
       </a>
       connections, not recommended in some server environments.
      </p>
      <p>
       All the possible states of this setting are as follows:
      </p>
      <p style="margin-left: 20px;">
       automatic - offloads if the connection is 10 GbE, has a
       <a class="term1 piterm" href="/_iframe_term.php?seek=RTT" rel="sg-term" title="What is RTT ?">
        RTT
       </a>
       default - this setting restores chimney offload to the system default.
       <br/>
       disabled - chimney offload disabled.
       <br/>
       enabled - chimney offload enabled.
      </p>
      <p>
       Default: disabled
       <br/>
       Recommended: enabled
      </p>
      <p>
       To set TCP Chimney Offload:
      </p>
      <p style="margin-left: 20px;">
       <strong>
        netsh int tcp set global chimney=enabled
       </strong>
      </p>
      <p>
       Using Powershell, to view the offload state:
      </p>
      <p style="margin-left: 20px;">
       <strong>
        Set-NetOffloadGlobalSetting -Chimney Enabled
       </strong>
       (default: disabled, possible states: enabled/disabled/automatic, recommended: enabled*)
      </p>
      <p>
       To view the current chimney state:
      </p>
      <p style="margin-left: 20px;">
       (Get-NetOffloadGlobalSetting).Chimney
      </p>
      <p>
       or, simply:
      </p>
      <p style="margin-left: 20px;">
       Get-NetOffloadGlobalSetting
      </p>
      <p>
       <br/>
      </p>
      <p>
       <strong>
        Notes:
       </strong>
       <br/>
       * There were issues with buggy Network adapter drivers and the implementation of offloading with older OSes (Windows 2003 Server), however, this has improved over time. One should be more careful using offloading in server environments, as there have been some reports of issues with TCP Chimney Offload and SQL servers under heavy load, affecting both application concurrency and
       <a class="term1 piterm" href="/_iframe_term.php?seek=THROUGHPUT" rel="sg-term" title="What is throughput ?">
        throughput
       </a>
       . See this
       <a href="http://blogs.msdn.com/b/psssql/archive/2008/10/01/windows-scalable-networking-pack-possible-performance-and-concurrency-impacts-to-sql-server-workloads.aspx" target="_blank">
        MSDN reference
       </a>
       and
       <a href="http://support.microsoft.com/en-us/kb/942861" target="_blank">
        MS KB942861
       </a>
       .
      </p>
      <p>
       Chimney Offload does not work together with NetDMA (mute point with Windows 8/8.1 as NetDMA is not supported).
      </p>
      <p>
       Chimney Offload may not be supported with
       <a class="term1 piterm" href="/_iframe_term.php?seek=NAT" rel="sg-term" title="What is NAT ?">
        NAT
       </a>
       /ICS (reference:
       <a href="http://technet.microsoft.com/en-us/library/gg162710" rel="_nofollow" target="_blank">
        Technet TCP Chimney Offload Compatibility
       </a>
       for Windows 2008 R2). However, this only applies if
       <a class="term1 piterm" href="/_iframe_term.php?seek=NAT" rel="sg-term" title="What is NAT ?">
        NAT
       </a>
       /ICS is enabled on the computer running Windows itself, and that is rarely the case.
      </p>
      <p>
       <br/>
      </p>
      <p>
       <br/>
      </p>
      <h2>
       <strike>
        NetDMA State
       </strike>
      </h2>
      <p>
       NetDMA (TCPA) enables support for advanced direct memory access. In essence, it provides the ability to more efficiently move network data by minimizing CPU usage. NetDMA frees the CPU from handling memory data transfers between network card data buffers and application buffers by using a DMA engine.
      </p>
      <p>
       Default: disabled
       <br/>
       Recommended: leave alone, don't bother (setting not supported in Windows 8 and later, according to MS)
      </p>
      <p>
       You can still edit it using:
      </p>
      <p style="margin-left: 20px;">
       netsh int tcp set global netdma=disabled
      </p>
      <p>
       Notes:
       <strong>
        <br/>
        According to Microsoft, NetDMA is not supported in Windows 8/8.1 and changing this setting will have no effect.
       </strong>
       <span style="line-height: normal;">
        See:
       </span>
       <a href="http://msdn.microsoft.com/en-us/library/ff568342(VS.85).aspx" rel="_nofollow" target="_blank">
        MSDN NetDMA
       </a>
      </p>
      <p>
       NetDMA (TCPA)
       <strong>
        does not work together with Chimney Offload
       </strong>
       either, you have to choose one or the other. For NetDMA to work, it must be enabled/supported by your
       <a class="term1 piterm" href="/_iframe_term.php?seek=BIOS" rel="sg-term" title="What is BIOS ?">
        BIOS
       </a>
       and your CPU must support Intel I/O Acceleration Technology (I/OAT).
      </p>
      <p>
       <br/>
      </p>
      <h2>
       Direct Cache Access (DCA)
      </h2>
      <p>
       Direct Cache Access (DCA) allows a capable I/O device, such as a network controller, to deliver data directly into a CPU cache. The objective of DCA is to reduce memory
       <a class="term1 piterm" href="/_iframe_term.php?seek=LATENCY" rel="sg-term" title="What is latency ?">
        latency
       </a>
       and the memory
       <a class="term1 piterm" href="/_iframe_term.php?seek=BANDWIDTH" rel="sg-term" title="What is bandwidth ?">
        bandwidth
       </a>
       requirement in high
       <a class="term1 piterm" href="/_iframe_term.php?seek=BANDWIDTH" rel="sg-term" title="What is bandwidth ?">
        bandwidth
       </a>
       (Gigabit) environments. DCA requires support from the I/O device, system chipset, and CPU(s).
      </p>
      <p>
       Possible states are: enabled, disabled
       <br/>
       Default state: disabled
       <br/>
       Recommended: enabled (provided the CPU/Chipset/NIC support it)
      </p>
      <p>
       To set DCA using netsh:
      </p>
      <p style="margin-left: 20px;">
       <strong>
        netsh int tcp set global dca=enabled
       </strong>
      </p>
      <p>
       <br/>
      </p>
      <p>
       Note: The impact of DCA is more significant with older CPUs.
      </p>
      <p>
       <br/>
      </p>
      <h2>
       <a class="term1 piterm" href="/_iframe_term.php?seek=CHECKSUM" rel="sg-term" title="What is Checksum ?">
        Checksum
       </a>
       Offload
      </h2>
      <p>
       The PowerShell Enable-NEtAdapterChecksumOffload cmdlet enables
       <a class="term1 piterm" href="/_iframe_term.php?seek=CHECKSUM" rel="sg-term" title="What is checksum ?">
        checksum
       </a>
       offloads on the network adapter.
       <a class="term1 piterm" href="/_iframe_term.php?seek=CHECKSUM" rel="sg-term" title="What is Checksum ?">
        Checksum
       </a>
       offloading is also required for some other stateless offloads to work, including Receive Side Scaling (
       <a class="term1 piterm" href="/_iframe_term.php?seek=RSS" rel="sg-term" title="What is RSS ?">
        RSS
       </a>
       ), Receive Segment Coalescing (RSC), and Large Send Offload (LSO).
      </p>
      <p>
       Default state: adapter-dependent
       <br/>
       Recommended: enabled
      </p>
      <p>
       In PowerShell:
      </p>
      <p style="margin-left: 20px;">
       <strong>
        Enable-NetAdapterChecksumOffload -Name *
       </strong>
       (use to enable
       <a class="term1 piterm" href="/_iframe_term.php?seek=CHECKSUM" rel="sg-term" title="What is checksum ?">
        checksum
       </a>
       offload on all adapters that support it)
       <br/>
       Disable-NetAdapterChecksumOffload -Name *  (use to disable
       <a class="term1 piterm" href="/_iframe_term.php?seek=CHECKSUM" rel="sg-term" title="What is checksum ?">
        checksum
       </a>
       offload for all network adapters)
       <br/>
       Get-NetAdapterChecksumOffload    (use to view network adapters that support
       <a class="term1 piterm" href="/_iframe_term.php?seek=CHECKSUM" rel="sg-term" title="What is checksum ?">
        checksum
       </a>
       offload and their state)
      </p>
      <p>
       The above PowerShell cmdlet example enables
       <a class="term1 piterm" href="/_iframe_term.php?seek=CHECKSUM" rel="sg-term" title="What is checksum ?">
        checksum
       </a>
       offloads on all visible network adapters for all
       <a class="term1 piterm" href="/_iframe_term.php?seek=UDP" rel="sg-term" title="What is UDP ?">
        UDP
       </a>
       /TCP/IP version 4 and version 6 traffic (and restarts the network adapters). Alternatively, you can specify only a single network adapter and/or
       <a class="term1 piterm" href="/_iframe_term.php?seek=PROTOCOL" rel="sg-term" title="What is protocol ?">
        protocol
       </a>
       by using command-line switches. For more information on customizing the command, refer to
       <a href="https://technet.microsoft.com/en-us/library/jj130918.aspx" rel="_nofollow" target="_blank">
        this
       </a>
       Technet article. To pick a single adapter and only modify its
       <a class="term1 piterm" href="/_iframe_term.php?seek=CHECKSUM" rel="sg-term" title="What is checksum ?">
        checksum
       </a>
       offload state, find installed adapters using this cmdlet:
       <em>
        Get-NetAdapter
       </em>
      </p>
      <p>
       <br/>
      </p>
      <h2>
       Receive-Side Scaling State (
       <a class="term1 piterm" href="/_iframe_term.php?seek=RSS" rel="sg-term" title="What is RSS ?">
        RSS
       </a>
       )
      </h2>
      <p>
       The receive-side scaling setting enables parallelized processing of received packets on multiple processors, while avoiding
       <a class="term1 piterm" href="/_iframe_term.php?seek=PACKET" rel="sg-term" title="What is packet ?">
        packet
       </a>
       reordering. It avoids
       <a class="term1 piterm" href="/_iframe_term.php?seek=PACKET" rel="sg-term" title="What is packet ?">
        packet
       </a>
       reordering by separating packets into "flows", and using a single processor for processing all the packets for a given flow. Packets are separated into flows by computing a hash value based on specific fields in each
       <a class="term1 piterm" href="/_iframe_term.php?seek=PACKET" rel="sg-term" title="What is packet ?">
        packet
       </a>
       , and the resulting hash values are used to select a processor for processing the flow. This approach ensures that all packets belonging to a given TCP connection will be queued to the same processor, in the same order that they were received by the network adapter.
      </p>
      <p>
       Default: enabled
       <br/>
       Recommended: enabled (provided you have 2 or more processor cores and a NIC that can handle RSS)
      </p>
      <p>
       To change RSS using netsh:
      </p>
      <p style="margin-left: 20px;">
       netsh int tcp set global rss=enabled
      </p>
      <p>
       To change using PowerShell cmdlets:
      </p>
      <p style="margin-left: 20px;">
       <strong>
        Enable-NetAdapterRss -Name *
       </strong>
       (use this to enable
       <a class="term1 piterm" href="/_iframe_term.php?seek=RSS" rel="sg-term" title="What is RSS ?">
        RSS
       </a>
       for all adapters that support it)
       <br/>
       Disable-NetAdapterRss -Name *   (use this to disables
       <a class="term1 piterm" href="/_iframe_term.php?seek=RSS" rel="sg-term" title="What is RSS ?">
        RSS
       </a>
       for all adapters)
       <br/>
       Get-NetAdapterRss -Name *     (use cmdlet to view
       <a class="term1 piterm" href="/_iframe_term.php?seek=RSS" rel="sg-term" title="What is RSS ?">
        RSS
       </a>
       capable Network Adapters)
      </p>
      <p>
       Notes:
       <br/>
       Needs
       <em>
        <a class="term1 piterm" href="/_iframe_term.php?seek=CHECKSUM" rel="sg-term" title="What is Checksum ?">
         Checksum
        </a>
        Offload
       </em>
       to be enabled. Only supported by some network adapters.
      </p>
      <p>
       <br/>
      </p>
      <h2>
       Receive Segment Coalescing State (RSC)
      </h2>
      <p>
       Receive Segment Coalescing (RSC) allows the
       <a class="term1 piterm" href="/_iframe_term.php?seek=NIC" rel="sg-term" title="What is NIC ?">
        NIC
       </a>
       to coalesce multiple
       <a class="term1 piterm" href="/_iframe_term.php?seek=TCP/IP" rel="sg-term" title="What is TCP/IP ?">
        TCP/IP
       </a>
       packets that arrive within a single interrupt into a single larger
       <a class="term1 piterm" href="/_iframe_term.php?seek=PACKET" rel="sg-term" title="What is packet ?">
        packet
       </a>
       (up to 64KB) so that the network stack has to process fewer headers, resulting in a 10% to 30% reduction in I/O overhead depending on the workload, thereby improving performance. Receive Segment Coalescing (RCS) is able to collect packets that are received during the same interrupt cycle and put them together so that they can be more efficiently delivered to the network stack. This can significantly increase the amount of traffic that can be handled without severely impacting the CPU.
      </p>
      <p>
       Possible states: enabled, disabled, default
       <br/>
       Default state: disabled
       <br/>
       Recommended: enabled for better
       <a class="term1 piterm" href="/_iframe_term.php?seek=THROUGHPUT" rel="sg-term" title="What is throughput ?">
        throughput
       </a>
       , disabled for pure gaming to ensure lowest latency.
      </p>
      <p>
       To enable using netsh:
      </p>
      <p style="margin-left: 20px;">
       netsh int tcp set global rsc=enabled
      </p>
      <p>
       To enable using PowerShell cmdlets:
      </p>
      <p style="margin-left: 20px;">
       <strong>
        Enable-NetAdapterRsc -Name *
       </strong>
       (use to enables RSC for all adapters that support it)
       <br/>
       Disable-NetAdapterRsc -Name *    (use to disable RSC for all adapters)
       <br/>
       Get-NetAdapterRsc -Name *      (use to view adapters that support RSC)
      </p>
      <p>
       Notes:
       <br/>
       Needs
       <em>
        <a class="term1 piterm" href="/_iframe_term.php?seek=CHECKSUM" rel="sg-term" title="What is Checksum ?">
         Checksum
        </a>
        Offload
       </em>
       to be enabled. Only supported by some network adapters.
       <br/>
       For gaming, where
       <a class="term1 piterm" href="/_iframe_term.php?seek=LATENCY" rel="sg-term" title="What is latency ?">
        latency
       </a>
       is more important than pure
       <a class="term1 piterm" href="/_iframe_term.php?seek=THROUGHPUT" rel="sg-term" title="What is throughput ?">
        throughput
       </a>
       , any type of
       <a class="term1 piterm" href="/_iframe_term.php?seek=PACKET" rel="sg-term" title="What is packet ?">
        packet
       </a>
       /memory coalescing should be disabled or used very sparingly.
       <a class="term1 piterm" href="/_iframe_term.php?seek=PACKET" rel="sg-term" title="What is Packet ?">
        Packet
       </a>
       /memory coalescing reduces CPU utilization and increases
       <a class="term1 piterm" href="/_iframe_term.php?seek=THROUGHPUT" rel="sg-term" title="What is throughput ?">
        throughput
       </a>
       , however, it causes the network adapter to combine packets before interacting with other hardware, which may slightly increase latency. Also see our
       <a href="http://www.speedguide.net/articles/gaming-tweaks-5812">
        gaming tweaks
       </a>
       article.
      </p>
      <p>
       <br/>
      </p>
      <h2>
       Large Send Offload (LSO)
      </h2>
      <p>
       This setting enables Large Send Offload. When enabled, the network adapter hardware is used to complete data segmentation, theoretically faster than operating system software. Theoretically, this feature may improve transmission performance, and reduce CPU load. LSO reduces the interrupt rate and the amount of bus transactions to the
       <a class="term1 piterm" href="/_iframe_term.php?seek=NIC" rel="sg-term" title="What is NIC ?">
        NIC
       </a>
       . The problem with this setting is buggy implementation on many levels, including Network Adapter Drivers. Intel and Broadcom drivers are known to have this enabled by default, and may have many issues with it. It should only be enabled with the newest Gigabit NICs with recent drivers. Because of the issues mentioned above, we recommend disabling LSO at both the Network Adapter properties, and at the OS level with the setting below.
      </p>
      <p>
       Default: adapter-dependent
       <br/>
       Recommended: disable (both in network adapter properties and in the TCP/IP stack at the OS level)
      </p>
      <p style="margin-left: 20px;">
       <strong>
        Disable-NetAdapterLso -Name *
       </strong>
       (disable LSO for all visible network adapters)
       <br/>
       Enable-NetAdapterLso -Name *  (enable LSO for both
       <a class="term1 piterm" href="/_iframe_term.php?seek=IPV4" rel="sg-term" title="What is IPv4 ?">
        IPv4
       </a>
       and
       <a class="term1 piterm" href="/_iframe_term.php?seek=IPV6" rel="sg-term" title="What is IPv6 ?">
        IPv6
       </a>
       on all network adapters. Be careful with this setting, test before using in production)
       <br/>
       Get-NetAdapterLso -Name * (get a list of network adapters that support LSO)
      </p>

      <h2>
       ECN Capability
      </h2>
      <p>
       ECN (Explicit Congestion Notification,
       <a class="term1 piterm" href="/_iframe_term.php?seek=RFC" rel="sg-term" title="What is RFC ?">
        RFC
       </a>
       3168) is a mechanism that provides routers with an alternate method of communicating network congestion. It is aimed to decrease retransmissions. In essence, ECN assumes that the cause of any
       <a class="term1 piterm" href="/_iframe_term.php?seek=PACKET LOSS" rel="sg-term" title="What is packet loss ?">
        packet loss
       </a>
       is
       <a class="term1 piterm" href="/_iframe_term.php?seek=ROUTER" rel="sg-term" title="What is router ?">
        router
       </a>
       congestion. It allows routers experiencing congestion to mark packets and allow clients to automatically lower their transfer rate to prevent further
       <a class="term1 piterm" href="/_iframe_term.php?seek=PACKET LOSS" rel="sg-term" title="What is packet loss ?">
        packet loss
       </a>
       . Traditionally,
       <a class="term1 piterm" href="/_iframe_term.php?seek=TCP/IP" rel="sg-term" title="What is TCP/IP ?">
        TCP/IP
       </a>
       networks signal congestion by dropping packets. When ECN is successfully negotiated, an ECN-aware
       <a class="term1 piterm" href="/_iframe_term.php?seek=ROUTER" rel="sg-term" title="What is router ?">
        router
       </a>
       may set a bit in the IP
       <a class="term1 piterm" href="/_iframe_term.php?seek=HEADER" rel="sg-term" title="What is header ?">
        header
       </a>
       (in the DiffServ field) instead of dropping a
       <a class="term1 piterm" href="/_iframe_term.php?seek=PACKET" rel="sg-term" title="What is packet ?">
        packet
       </a>
       in order to signal congestion. The receiver echoes the congestion indication to the sender, which must react as though a
       <a class="term1 piterm" href="/_iframe_term.php?seek=PACKET" rel="sg-term" title="What is packet ?">
        packet
       </a>
       drop were detected. ECN is disabled by default in modern Windows
       <a class="term1 piterm" href="/_iframe_term.php?seek=TCP/IP" rel="sg-term" title="What is TCP/IP ?">
        TCP/IP
       </a>
       implementations, as it is possible that it may cause problems with some outdated routers that drop packets with the ECN bit set, rather than ignoring the bit.
      </p>
      <p>
       Possible settings are: enabled, disabled, default (restores the state to the system default).
       <br/>
       Default state: disabled
      </p>
      <p>
       Recommendation: "enabled" only for short-lived, interactive connections and HTTP requests with routers that support it, in the presence of congestion/packet loss, "disabled" otherwise (for pure bulk
       <a class="term1 piterm" href="/_iframe_term.php?seek=THROUGHPUT" rel="sg-term" title="What is throughput ?">
        throughput
       </a>
       with large TCP Window, no regular congestion/packet loss, or outdated routers without ECN support).
      </p>
      <p>
       To change using netsh:
      </p>
      <p style="margin-left: 20px;">
       <strong>
        netsh int tcp set global ecncapability=default
       </strong>
      </p>
      <p>
       To change using PowerShell cmdlets:
      </p>
      <p style="margin-left: 20px;">
       <strong>
        Set-NetTCPSetting -SettingName InternetCustom -EcnCapability Disabled
       </strong>
      </p>
      <p>
       <br/>
      </p>
      <p>
       Notes:
       <br/>
       Alternative netsh syntax is:
       <em>
        netsh int tcp set global ecn=default
       </em>
      </p>
      <p>
       ECN is only effective in combination with AQM (Active Queue Management)
       <a class="term1 piterm" href="/_iframe_term.php?seek=ROUTER" rel="sg-term" title="What is router ?">
        router
       </a>
       policy. It has more noticeable effect on performance with interactive connections, online games, and HTTP requests, in the presence of
       <a class="term1 piterm" href="/_iframe_term.php?seek=ROUTER" rel="sg-term" title="What is router ?">
        router
       </a>
       congestion/packet loss. Its effect on bulk
       <a class="term1 piterm" href="/_iframe_term.php?seek=THROUGHPUT" rel="sg-term" title="What is throughput ?">
        throughput
       </a>
       with large TCP Window are less clear. Currently, we do not recommend enabling this setting, as reportedly it has negative impact on
       <a class="term1 piterm" href="/_iframe_term.php?seek=THROUGHPUT" rel="sg-term" title="What is throughput ?">
        throughput
       </a>
       with some residential US ISPs. EA multiplayer games that require a profile logon do not support ECN as well (you will not be able to logon). However, it can also reduce
       <a class="term1 piterm" href="/_iframe_term.php?seek=LATENCY" rel="sg-term" title="What is latency ?">
        latency
       </a>
       in some games with ECN-capable routers in the presence of
       <a class="term1 piterm" href="/_iframe_term.php?seek=PACKET LOSS" rel="sg-term" title="What is packet loss ?">
        packet loss
       </a>
       (dropped packets).
      </p>

       See also:

        Wikipedia - ECN

       ,

        RFC 3168

      <h2>
       TCP 1323
       <a class="term1 piterm" href="/_iframe_term.php?seek=TIMESTAMPS" rel="sg-term" title="What is Timestamps ?">
        Timestamps
       </a>
      </h2>
      <p>
       <a class="term1 piterm" href="/_iframe_term.php?seek=RFC" rel="sg-term" title="What is RFC ?">
        RFC
       </a>
       1323 ads many useful options, most notably scaling of the TCP Window. "
       <a class="term1 piterm" href="/_iframe_term.php?seek=TIMESTAMPS" rel="sg-term" title="What is Timestamps ?">
        Timestamps
       </a>
       " (TSOpt) is a less commonly used 1323 option that is intended to increase transmission reliability by retransmitting segments that are not acknowledged within some retransmission timeout (RTO) interval. The problem with
       <a class="term1 piterm" href="/_iframe_term.php?seek=TIMESTAMPS" rel="sg-term" title="What is timestamps ?">
        timestamps
       </a>
       is that they add 12 bytes to the 20-byte TCP header of each
       <a class="term1 piterm" href="/_iframe_term.php?seek=PACKET" rel="sg-term" title="What is packet ?">
        packet
       </a>
       , so turning them on causes considerable overhead.
      </p>
      <p>
       Possible states: enabled,disabled,default
       <br/>
       Default state: disabled
       <br/>
       Recommended: disabled
      </p>
      <p>
       To set using netsh:
      </p>
      <p style="margin-left: 20px;">
       <strong>
        netsh int tcp set global timestamps=disabled
       </strong>
      </p>
      <p>
       To set using PowerShell cmdlets:
      </p>
      <p style="margin-left: 20px;">
       <strong>
        Set-NetTCPSetting -SettingName InternetCustom -
        <a class="term1 piterm" href="/_iframe_term.php?seek=TIMESTAMPS" rel="sg-term" title="What is Timestamps ?">
         Timestamps
        </a>
        Disabled
       </strong>
      </p>
      <p>

      <h2>
       Initial RTO and Min RTO
      </h2>
      <p>
       Retransmit timeout (RTO) determines how many milliseconds of unacknowledged data it takes before the connection is aborted. The default timeout for Initial RTO of 3 seconds can usually be lowered for low-
       <a class="term1 piterm" href="/_iframe_term.php?seek=LATENCY" rel="sg-term" title="What is latency ?">
        latency
       </a>
       modern
       <a class="term1 piterm" href="/_iframe_term.php?seek=BROADBAND" rel="sg-term" title="What is broadband ?">
        broadband
       </a>
       connections, unless you're in a remote location, on a satellite internet connection, or experiencing high
       <a class="term1 piterm" href="/_iframe_term.php?seek=LATENCY" rel="sg-term" title="What is latency ?">
        latency
       </a>
       . In high-
       <a class="term1 piterm" href="/_iframe_term.php?seek=LATENCY" rel="sg-term" title="What is latency ?">
        latency
       </a>
       situations, this can increase retransmissions if the RTO value is reached on a regular basis.
      </p>
      <p>
       <strong>
        InitialRTO
       </strong>
       <br/>
       Default value: 3000 (3 seconds)
       <br/>
       Recommended: between 2000 (2 seconds) and 3000 (3 seconds).
      </p>
      <p>
       To set using netsh:
      </p>
      <p style="margin-left: 20px;">
       <strong>
        netsh int tcp set global initialRto=3000
       </strong>
      </p>
      <p>
       To set using PowerShell cmdlets:
      </p>
      <p style="margin-left: 20px;">
       <strong>
        Set-NetTCPSetting -SettingName InternetCustom -InitialRto 3000
       </strong>
      </p>
      <p>
       <br/>
      </p>
      <p>
       <strong>
        MinRTO
       </strong>
       <br/>
       Default value: 300 (ms)
       <br/>
       Recommended: 300 (ms)
      </p>
      <p>
       MinRTO can't be changed under Windows 8 (read-only error), it may be editable under server OSes. The command is:
      </p>
      <p style="margin-left: 20px;">
       set-NetTCPSetting -SettingName InternetCustom -MinRto 300
      </p>
      <p>
       <br/>
      </p>

      <h2>
       InitialCongestionWindow (ICW)
      </h2>
      <p>
       When creating a TCP connection, the sending side performs a "TCP slow start" regardles of the receiver
       <a class="term1 piterm" href="/_iframe_term.php?seek=RWIN" rel="sg-term" title="What is RWIN ?">
        RWIN
       </a>
       value. TCP slow start only sends two frames, waits for ACK response, and increases speed exponentially provided there are no dropped packets. This slow start algorithm can also be activated if there is no traffic for 200ms. This is not optimal for fast internet connections with intermittent bursts of data. This bottleneck can be avoided by increasing the "InitialcongestionWindow" from the default 2 (or 4) frames to 10 (See
       <a class="term1 piterm" href="/_iframe_term.php?seek=RFC" rel="sg-term" title="What is RFC ?">
        RFC
       </a>
       3390 and
       <a class="term1 piterm" href="/_iframe_term.php?seek=RFC" rel="sg-term" title="What is RFC ?">
        RFC
       </a>
       6928).
      </p>
      <p>
       The following should work for 2012 server, however it does not work in Windows 8.1 (read-only setting error):
      </p>
      <p style="margin-left: 20px;">
       <strong>
        Set-NetTCPSetting -SettingName InternetCustom -InitialCongestionWindow 10
       </strong>
      </p>
      <p>
       The following command works for Server 2008 R2 (with hotfix KB2472264 linked below):
      </p>
      <p style="margin-left: 20px;">
       <strong>
        netsh int tcp set supplemental template=custom icw=10
       </strong>
      </p>

        TCP/IP
       </a>
       related Windows Registry parameters that work with Windows 8/8.1 and 2012 Server. Some of the settings we've tested are the same/similar parameters as with previous Windows versions that we have confirmed to work with Windows 8/8.1 as well, although they may have different default values. Below is a list of what we've confirmed to work. To edit the Windows Registry, click the Windows button, and type:
       <em>
        <a class="term1 piterm" href="/_iframe_term.php?seek=REGEDIT" rel="sg-term" title="What is regedit ?">
         regedit

       </em>

         regedit
        </a>
       </em>
       and choose to run as administrator to have the proper write permissions). It is always a good idea to create a restore point and backup the registry before making changes.
      </p>

       <br/>
      </p>
      <h4>
       DefaultTTL
      </h4>
      <p>
       <a class="term1 piterm" href="/_iframe_term.php?seek=TTL" rel="sg-term" title="What is TTL ?">
        TTL
       </a>
       does not directly affect speed, and can be safely left alone in many cases. It is a limit to the time and number of hops/routers a
       <a class="term1 piterm" href="/_iframe_term.php?seek=PACKET" rel="sg-term" title="What is packet ?">
        packet
       </a>
       will travel before being discarded. A number that's too small risks packets being discarded before reaching their destination. A number that's too large (over 128) will cause delay in when lost IP packets are discarded. The default, when the setting is not present in the Windows 8/10/2012 registry the system assumes a value of 128.
      </p>
      <p>
       <span style="color: #FF0000;">
        HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters
       </span>
       <br/>
       <span style="color: #FF0000;">
        <strong>
         DefaultTTL=64
        </strong>
       </span>
       (DWORD, not present by default denoting 128, valid range is 1-255, recommended: 64)
      </p>
      <p>
       <br/>
      </p>
      <h4>
       Host Resolution Priority Tweak
      </h4>
      <p>
       This is intended to increase the priority of
       <a class="term1 piterm" href="/_iframe_term.php?seek=DNS" rel="sg-term" title="What is DNS ?">
        DNS
       </a>
       /hostname resolution, by increasing the priority of four related processes from their defaults. It is important to note that this increases their priority compared to the hundreds of other running processes, while keeping their order. See our
       <a href="http://www.speedguide.net/articles/host-resolution-priority-tweak-1130">
        Host Resolution Priority Tweak article
       </a>
       for more details. The registry location under Windows 8, 8.1, 10, 2012 Server, and 2012 R2 is:
      </p>
      <p>
       <span style="color: rgb(255, 0, 0);">
        HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\Tcpip\ServiceProvider\
       </span>
       <br/>
       <span style="color: rgb(255, 0, 0);">
        <strong>
         LocalPriority = 4
        </strong>
       </span>
       (DWORD, default 499, recommended 4)
       <br/>
       <span style="color: rgb(255, 0, 0);">
        <strong>
         HostPriority = 5
        </strong>
       </span>
       (DWORD, default 500, recommended 5)
       <br/>
       <span style="color: rgb(255, 0, 0);">
        <strong>
         DnsPriority = 6
        </strong>
       </span>
       (DWORD, default 2000, recommended 6)
       <br/>
       <span style="color: rgb(255, 0, 0);">
        <strong>
         NetbtPriority = 7
        </strong>
       </span>
       (DWORD, default 2001, recommended 7)
      </p>
      <p>
       <br/>
      </p>
      <h4>
       MaxUserPort and TCPTimedWaitDelay (port allocation)
      </h4>
      <p>
       Short lived (ephemeral)
       <a class="term1 piterm" href="/_iframe_term.php?seek=TCP/IP" rel="sg-term" title="What is TCP/IP ?">
        TCP/IP
       </a>
       ports above 1024 are allocated as needed by the OS. The Windows 8/2012 defaults are usually sufficient under normal network load. However, under heavy network load it may be necessary to adjust these two registry settings to increase port availability and decrease the time to wait before reclaiming unused ports.
      </p>
      <p>
       If the default limits are exceeded under heavier loads, the following error is observed "
       <em>
        address in use: connect exception
       </em>
       ". To tweak, add/edit the following keys using regedit:
      </p>
      <p>
       <span style="color: rgb(255, 0, 0);">
        HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters
       </span>
       <br/>
       <span style="color: rgb(255, 0, 0);">
        <strong>
         MaxUserPort=65534
        </strong>
       </span>
       (DWORD, not in the registry by default, uses ports 1024-5000 by default. recommended: 16384 to 65534 decimal as necessary) - maximum number of ports to use. 1024 is automatically subtracted from entered value to allow for reserved ports under 1024.
      </p>
      <p>
       <span style="color: rgb(255, 0, 0);">
        <strong>
         TcpTimedWaitDelay=30
        </strong>
       </span>
       (DWORD, not present in registry by default, 120 seconds when not present. Recommended: 30 decimal, denoting 30 seconds) - time to wait before reclaiming ports, in seconds. Default time before reclaiming ports, if value is at 0xffffffff or not present in the registry is 120 (or 240) seconds, depending on your OS. Just reducing the delay is often sufficient without changing MaxUserPort, as it allows for reusing ports more efficiently.
      </p>
      <p>
       See:
       <br/>
       <a href="https://msdn.microsoft.com/en-us/library/windows/desktop/ms737757(v=vs.85).aspx" target="_blank">
        MSDN ms737757
       </a>
       <br/>
       Technet
       <a href="https://technet.microsoft.com/en-us/library/cc938196.aspx" target="_blank">
        MaxUserPort
       </a>
      </p>

        <a class="term1 piterm" href="/_iframe_term.php?seek=PACKET" rel="sg-term" title="What is Packet ?">

      </p>
      <h4>
       <a class="term1 piterm" href="/_iframe_term.php?seek=QOS" rel="sg-term" title="What is QoS ?">
        QoS
       </a>
       Policy
      </h4>
      <p>
       <a class="term1 piterm" href="/_iframe_term.php?seek=QOS" rel="sg-term" title="What is QoS ?">
        QoS
       </a>
       Policy settings under Windows 7/8 can be edited using the Group Policy Editor (gpedit.msc): Computer Configuration -&gt; Windows Settings -&gt; Policy-based
       <a class="term1 piterm" href="/_iframe_term.php?seek=QOS" rel="sg-term" title="What is QoS ?">
        QoS
       </a>
      </p>
      <p>
       In order to define DiffServ (DSCP) values, according to Microsoft the machine needs to have joined a
       <a class="term1 piterm" href="/_iframe_term.php?seek=DOMAIN" rel="sg-term" title="What is domain ?">
        domain
       </a>
       , and interfaces have to see the
       <a class="term1 piterm" href="/_iframe_term.php?seek=DOMAIN" rel="sg-term" title="What is domain ?">
        domain
       </a>
       controller. To overcome this limitation, so that you can tag DSCP values even for adapters that do not have access to a domain, use the following hidden registry key:
      </p>
      <p>
       <span style="color: #FF0000;">
        HKEY_LOCAL_MACHINE\System\CurrentControlSet\Services\Tcpip\QoS
       </span>
       (you may have to create the QoS key)
       <br/>
       <span style="color: #FF0000;">
        "Do not use NLA"="1"
       </span>
       (REG_SZ string value, not DWORD, not present by default, recommended: 1 if you plan to edit DSCP values via gpedit.msc)
      </p>
      <p>
       <em>
        Notes:
        <br/>
        gpedit.msc is only provided with the Professional/Ultimate and Server Windows variants.
        <br/>
        This registry key may also be needed in systems with multiple network adapters present.
       </em>
      </p>
      <p>
       <br/>
      </p>
      <h4>
       Network Memory Allocation (Event ID 2017 error)
      </h4>
      <p>
       When using Windows to serve many/large files over the local network, it is possible to sometimes run into memory allocation errors related to the Windows share, especially with clients that use different operating systems. When this happens, you can usually see the following error in the Event Viewer System log:
       <br/>
       Event ID: 2017 "The server was unable to allocate from the system nonpaged pool because the server reached the configured limit for nonpaged pool allocations." It is also possible to get an error indicating that "Not enough server storage is available to process this command". To avoid those errors, you need to change the way Windows allocates memory for network services and file sharing. The below settings optimize the machine as a file server so it would allocate resources accordingly. There are two related registry settings:
      </p>
      <p>
       <span style="color: #FF0000;">
        HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Session Manager\Memory Management
       </span>
       <br/>
       <span style="color: #FF0000;">
        <strong>
         LargeSystemCache=1
        </strong>
       </span>
       (DWORD, default value: 0, recommended value: 1)
       <br/>
       A value of zero establishes a cache of ~8 MB, a value of 1 allows the cache to expand up to the amount of physical memory minus 4 MB, as necessary.
      </p>
      <p>
       <span style="color: #FF0000;">
        HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\LanmanServer\Parameters
       </span>
       <br/>
       <strong>
        <span style="color: #FF0000;">
         Size=3
        </span>
       </strong>
       (DWORD, default value: 1, recommended value: 3)
       <br/>
       minimizes used memory
       <br/>
       balance used memory
       <br/>
       optimal setting for file sharing and network applications
      </p>
      <p>
       <br/>
      </p>
      <h4>
       Internet Explorer Web Browser Tweaks
      </h4>
      <p>
       MaxConnectionsPerServer / MaxConnectionsPer1_OServer Internet Explorer browser tweaks adjust the maximum number of concurrent connections your web browser can establish to any single website.
      </p>
      <p>
       <span style="color: rgb(255, 0, 0);">
        HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Internet Explorer\MAIN\FeatureControl\
       </span>
       <br/>
       <span style="color: rgb(255, 0, 0);">
        FEATURE_MAXCONNECTIONSPER1_0SERVER\explorer.exe=10
       </span>
       (DWORD, default: 4, recommended: 8, valid range: 2-128)
       <br/>
       <span style="color: rgb(255, 0, 0);">
        FEATURE_MAXCONNECTIONSPERSERVER\explorer.exe=10
       </span>
       (DWORD, default: 2, recommended: 8, valid range: 2-128)
      </p>
      <p>
       In addition to explorer.exe, you can also add a 32bit DWORD "iexplore.exe" (according to
       <a href="https://msdn.microsoft.com/en-us/library/ie/ee330731.aspx" target="_blank">
        MSDN
       </a>
       ), and set it to the same corresponding value as "explorer.exe"
       <br/>
       <span style="color: rgb(255, 0, 0);">
        FEATURE_MAXCONNECTIONSPER1_0SERVER\iexplore.exe=10
       </span>
       (DWORD, not present by default, recommended: 8, valid range: 2-128)
       <br/>
       <span style="color: rgb(255, 0, 0);">
        FEATURE_MAXCONNECTIONSPERSERVER\iexplore.exe=10
       </span>
       (DWORD, not present by default, recommended: 8, valid range: 2-128)
      </p>
      <p>
       For more information, see our
       <a href="http://www.speedguide.net/articles/internet-explorer-chrome-firefox-web-browser-tweaks-2448">
        Web Browser Tweaks
       </a>
       article
      </p>
      <p>
       Note: Increasing this setting much over 10 can cause incomplete page loading and other issues with some pages.
      </p>
      <p>
       <br/>
      </p>
      <h4>
       Network Throttling Index (Gaming)
      </h4>
      <p>
       By default, Windows 8 continues to implement a network throttling mechanism to restrict the processing of non-multimedia network traffic to 10 packets per millisecond (a bit over 100 Mbits/second). The idea behind such throttling is that processing of network packets can be a resource-intensive task, and it may need to be throttled to give prioritized CPU access to multimedia programs. In some cases, such as Gigabit networks and some online games, for example, it is beneficial to turn off such throttling all together for achieving maximum throughput.
      </p>
      <p>
       <span style="color: rgb(255, 0, 0);">
        HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Multimedia\SystemProfile
       </span>
       <br/>
       <span style="color: rgb(255, 0, 0);">
        <strong>
         NetworkThrottlingIndex=ffffffff
        </strong>
       </span>
       (DWORD, default: 10, recommended: 10 for media sharing, ffffffff for gaming and max
       <a class="term1 piterm" href="/_iframe_term.php?seek=THROUGHPUT" rel="sg-term" title="What is throughput ?">
        throughput
       </a>
       , valid range: 1 through 70 decimal or ffffffff to completely disable throttling)
      </p>
      <p>
       It is only recommended to change this setting in saturated Gigabit
       <a class="term1 piterm" href="/_iframe_term.php?seek=LAN" rel="sg-term" title="What is LAN ?">
        LAN
       </a>
       environments, where you do not want to give priority to multimedia playback. Reportedly, disabling throttling by using ffffffff can also help reduce
       <a class="term1 piterm" href="/_iframe_term.php?seek=PING" rel="sg-term" title="What is ping ?">
        ping
       </a>
       spikes in some online games. Games that may be affected by this throttling: Source Engine games (TF2, Left 4 Dead, CS:S), HoN, CoD, Overlord series.
      </p>
      <p>
       <br/>
      </p>
      <h4>
       SystemResponsiveness (Gaming and Multimedia)
      </h4>
      <p>
       In Windows 8/8.1, just like with Windows 7, multimedia applications use the "Multimedia Class Scheduler" service (MMCSS) to ensure priritized access to CPU resources, without denying CPU resources to lower-priority background applications. However, this also reserves 20% of CPU by default for background processes, your multimedia streaming and some games can only utilize up to 80% of the CPU. This setting, in combination with the above "NetworkThrottlingIndex" can help some games and video streaming. We recommend reducing the reserved CPU for background processes from the default of 20%.
      </p>
      <p>
       Navigate to:
      </p>
      <p>
       <span style="color: #FF0000;">
        HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Multimedia\SystemProfile
       </span>
       <br/>
       <span style="color: #FF0000;">
        <strong>
         SystemResponsiveness=10
        </strong>
       </span>
       (DWORD, default is 20 denoting 20% of CPU reserved, recommended: decimal 10, or 0 for pure gaming/multimedia performance)
      </p>
      <p>
       Notes: The number in this key is rounded by MMCSS to the nearest 10.In some server operating systems (Windows 2008 Server), theSystemResponsivenessmay be set to 100, instead of 20 by default. This is by design, giving higher priority to background services over multimedia.
      </p>
      <p>
       <br/>
       See also:
       <a href="https://msdn.microsoft.com/en-us/library/windows/desktop/ms684247(v=vs.85).aspx" target="_blank">
        MSDN ms684247
       </a>
      </p>
      <p>
       <br/>
      </p>
      <h4>
       Disable Nagle's algorithm (Gaming)
      </h4>
      <p>
       Nagle's algorithm is designed to allow several small packets to be combined together into a single, larger
       <a class="term1 piterm" href="/_iframe_term.php?seek=PACKET" rel="sg-term" title="What is packet ?">
        packet
       </a>
       for more efficient transmissions. While this improves
       <a class="term1 piterm" href="/_iframe_term.php?seek=THROUGHPUT" rel="sg-term" title="What is throughput ?">
        throughput
       </a>
       efficiency and reduces TCP/IP
       <a class="term1 piterm" href="/_iframe_term.php?seek=HEADER" rel="sg-term" title="What is header ?">
        header
       </a>
       overhead, it also briefly delays transmission of small packets. Disabling "nagling" can help reduce
       <a class="term1 piterm" href="/_iframe_term.php?seek=LATENCY" rel="sg-term" title="What is latency ?">
        latency
       </a>
       /ping in some games. Keep in mind that disabling Nagle's algorithm may also have some negative effect on file transfers. Nagle's algorithm is enabled in Windows by default. To implement this tweak and disable Nagle's algorithm, modify the following registry keys.
      </p>
      <p>
       This setting configures the maximum number of outstanding ACKs:
       <br/>
       <span style="color: rgb(255, 0, 0);">
        HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters\Interfaces\{NIC-id}
       </span>
       <br/>
       There will be multiple
       <a class="term1 piterm" href="/_iframe_term.php?seek=NIC" rel="sg-term" title="What is NIC ?">
        NIC
       </a>
       interfaces listed there, for example: {1660430C-B14A-4AC2-8F83-B653E83E8297}. Find the correct one with your IP address listed. Under this {NIC-id} key, create a new DWORD value:
       <br/>
       <span style="color: rgb(255, 0, 0);">
        <strong>
         TcpAckFrequency=1
        </strong>
       </span>
       (DWORD value, not present by default interpreted as 2, 1=disable, 2=default, specifies number of outstanding ACKs before ignoring delayed ACK timer). For gaming performance, recommended is 1 (disable). For pure
       <a class="term1 piterm" href="/_iframe_term.php?seek=THROUGHPUT" rel="sg-term" title="What is throughput ?">
        throughput
       </a>
       and data streaming, you can experiment with small values over 2. Wifi performance may see a slight improvement with disabled TcpAckFrequency as well.
      </p>
      <p>
       In the same location, add a new DWORD value:
       <br/>
       <strong>
        <span style="color: rgb(255, 0, 0);">
         TCPNoDelay=1
        </span>
       </strong>
       (DWORD value, not present by default, 0 to enable Nagle's algorithm, 1 to disable)
      </p>
      <p>
       To configure the ACK interval timeout (only has effect if nagling is enabled), find the following key:
       <br/>
       <span style="background-color: rgb(255, 255, 255); color: rgb(255, 0, 0); font-family: Verdana, Arial, Helvetica, sans-serif; font-size: 12px; font-style: normal; font-variant: normal; font-weight: normal; line-height: normal; text-align: left;">
        TcpDelAckTicks=0
       </span>
       (DWORD value, not present by default interpreted as 2, 0=disable nagling, 1-6=100-600 ms). Note you can also set this to 1 to reduce the nagle effect from the default of 200ms without disabling it.
      </p>
      <p>
       For Server Operating Systems that have Microsoft Message Queuing (MSMQ) installed, or if you have the MSMQ registry hive present, also add TCPNoDelay to:
       <br/>
       <span style="color: #FF0000;">
        HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\MSMQ\Parameters
       </span>
       <br/>
       <span style="color: #FF0000;">
        TCPNoDelay=1
       </span>
       (DWORD, not present by default, 0 to enable Nagle's algorithm, 1 to disable)
      </p>
      <p>
       See also:
       <a href="http://www.speedguide.net/articles/gaming-tweaks-5812">
        Gaming Tweaks
       </a>
       article
