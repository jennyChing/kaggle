     <h1>
      Debunking Myths: Penetration Testing is a Waste of Time
     </h1>
     <div class="viewdate">
      Thursday, April 04, 2013
     </div>
     <div class="viewavatar">
      <label>
       Contributed By:
      </label>
      <br/>
      <strong>
       <a href="/userprofile/8807.html">
        Rohit Sethi
       </a>
      </strong>
      <br/>
      <br/>
      <a href="/userprofile/8807.html">
       <img alt="219bfe49c4e7e1a3760f307bfecb9954" src="/images/../uploads/avatar/219bfe49c4e7e1a3760f307bfecb9954.jpg"/>
      </a>
     </div>
     <div class="post_text">
      <p style="margin: 0px 0px 10px; color: rgb(51, 51, 51); font-family: 'Helvetica Neue', Helvetica, Arial, sans-serif; font-size: 14px; line-height: 20px;">
       <span style="font-family:trebuchet ms,helvetica,sans-serif;">
        <span style="font-size: 14px;">
         Suppose you hire a consultancy to perform a black-box assessment of your software. After executing the test, the firm produces a report outlining several vulnerabilities with your application. You remediate the vulnerabilities, submit the application for re-testing, and the next report comes back clean  i.e. without any vulnerabilities. At best, this simply tells you that your application cant be broken into by the same testers in the same time frame. On the other hand, it doesnt tell you:
        </span>
       </span>
      </p>
      <ul style="padding: 0px; margin: 0px 0px 10px 25px; color: rgb(51, 51, 51); font-family: 'Helvetica Neue', Helvetica, Arial, sans-serif; font-size: 14px; line-height: 20px;">
       <li>
        <span style="font-family:trebuchet ms,helvetica,sans-serif;">
         <span style="font-size: 14px;">
          What are the potential threats to your application?
         </span>
        </span>
       </li>
       <li>
        <span style="font-family:trebuchet ms,helvetica,sans-serif;">
         <span style="font-size: 14px;">
          Which threats is your application not vulnerable to?
         </span>
        </span>
       </li>
       <li>
        <span style="font-family:trebuchet ms,helvetica,sans-serif;">
         <span style="font-size: 14px;">
          Which threats did the testers
          <i>
           not
          </i>
          assess your application for? Which threats were not possible to test from a runtime perspective?
         </span>
        </span>
       </li>
       <li>
        <span style="font-family:trebuchet ms,helvetica,sans-serif;">
         <span style="font-size: 14px;">
          How did time and other constraints on the test affect the reliability of results? For example, if the testers had 5 more days, what other security tests would they have executed?
         </span>
        </span>
       </li>
       <li>
        <span style="font-family:trebuchet ms,helvetica,sans-serif;">
         <span style="font-size: 14px;">
          What was the skill level of the testers and would you get the same set of results from a different tester or another consultancy?
         </span>
        </span>
       </li>
      </ul>
      <p style="margin: 0px 0px 10px; color: rgb(51, 51, 51); font-family: 'Helvetica Neue', Helvetica, Arial, sans-serif; font-size: 14px; line-height: 20px;">
       <span style="font-family:trebuchet ms,helvetica,sans-serif;">
        <span style="font-size: 14px;">
         In our experience, organizations arent able to answer most of these questions. The tester doesnt understand application internals and the organization requesting the test doesnt know much about the security posture of their software. Were not the only ones who acknowledge this issue: Haroon Meer discussed the
         <a href="http://blog.thinkst.com/2012/03/penetration-testing-considered-harmful.html" style="color: rgb(0, 136, 204); text-decoration: none;">
          challenges of penetration testing
         </a>
         at 44con. Most of these issues apply to every form of verification: automated dynamic testing, automated static testing, manual penetration testing, and manual code review. In fact a
         <a href="http://www.cs.berkeley.edu/~daw/papers/coderev-essos13.pdf" style="color: rgb(0, 136, 204); text-decoration: none;">
          recent paper
         </a>
         describes similar challenges in source code review.
        </span>
       </span>
      </p>
      <p style="margin: 0px 0px 10px; color: rgb(51, 51, 51); font-family: 'Helvetica Neue', Helvetica, Arial, sans-serif; font-size: 14px; line-height: 20px;">
       <span style="font-family:trebuchet ms,helvetica,sans-serif;">
        <span style="font-size: 14px;">
         The opaque nature of verification means effectivemanagement of
         <a href="http://blog.sdelements.com/making-security-requirements-a-reality/" style="color: rgb(0, 136, 204); text-decoration: none;" title="Making Security Requirements a Reality">
          software security requirements
         </a>
         is essential. With requirements listed, testers can specify both whether they have assessed a particular requirement and the techniques they used to do so. Critics argue that penetration testers shouldnt follow a checklist approach to auditing because no checklist can cover the breadth of obscure and
         <a href="http://labs.securitycompass.com/appsec-2/domain-driven-security/" style="color: rgb(0, 136, 204); text-decoration: none;">
          domain-specific vulnerabilities
         </a>
         . Yet the flexibility to find unique issues does not obviate the need to verify well understood requirements. The situation is very similar for standard software Quality Assurance (QA): good QA testers both verify functional requirements AND think outside the box about creative ways to break functionality. Simply testing blindly and reporting defects without verifying functional requirements would dramatically reduce the utility of quality assurance. Why accept a lower standard from security testing?
        </span>
       </span>
      </p>
      <p>
       <span style="font-family:trebuchet ms,helvetica,sans-serif;">
        <span style="font-size: 14px;">
         <span style="color: rgb(51, 51, 51); line-height: 20px;">
          Before you perform your next security verification activity, make sure you havesoftware security requirementsto measure against and that you define which requirements are in-scope for the verification. If you engage manual penetration testers or source code reviewers, it should be relatively simple for them to specify which requirements they tested for. If you use an automated tool or service, work with your vendor to find out what requirements their tool or service cannot reliably test for. Your tester/product/service is unlike to guarantee an absence of false negatives (i.e. certify that your application is not vulnerable to SQL injection), but knowing what they did and did not test for can dramatically help increase the confidence that your system does not contain known, preventable security flaws.
         </span>
        </span>
       </span>
      </p>
      <p>
       <span style="font-family:trebuchet ms,helvetica,sans-serif;">
        <span style="font-size: 14px;">
         <em>
          Cross-posted from the
          <a href="http://blog.sdelements.com/is-penetration-testing-a-waste-of-time/">
           SD Elements blog
          </a>
          .
         </em>
        </span>
       </span>
      </p>
