           <div class="post-text" itemprop="text">
            <p>
             We have an employee whose last name is Null. He kills our employee lookup application when his last name is used as the search term (which happens to be quite often now). The error received (thanks Fiddler!) is
            </p>
            <pre>
             <code>
              &lt;soapenv:Fault&gt;
   &lt;faultcode&gt;soapenv:Server.userException&lt;/faultcode&gt;
   &lt;faultstring&gt;coldfusion.xml.rpc.CFCInvocationException: [coldfusion.runtime.MissingArgumentException : The SEARCHSTRING parameter to the getFacultyNames function is required but was not passed in.]&lt;/faultstring&gt;
             </code>
            </pre>
            <p>
             Cute, huh?
            </p>
            <p>
             The parameter's type is string.
            </p>
            <p>
             I am using:
            </p>
            <ul>
             <li>
              WSDL (SOAP).
             </li>
             <li>
              Flex 3.5
             </li>
             <li>
              ActionScript 3
             </li>
             <li>
              ColdFusion 8
             </li>
            </ul>
            <p>
             Note that the error DOES NOT occur when calling the webservice as an object from a ColdFusion page.
            </p>

           <div class="post-text" itemprop="text">
            <h1>
             Tracking it down
            </h1>
            <p>
             At first I thought this was a coercion bug where
             <code>
              null
             </code>
             was getting coerced to
             <code>
              "null"
             </code>
             and a test of
             <code>
              "null" == null
             </code>
             was passing. It's not.
             <strong>
              I was close, but so very, very wrong. Sorry about that!
             </strong>
            </p>
            <p>
             I've since done lots of
             <a href="http://wonderfl.net/c/dd23/read">
              fiddling on wonderfl.net
             </a>
             and tracing through the code in
             <code>
              mx.rpc.xml.*
             </code>
             . At line 1795 of
             <code>
              XMLEncoder
             </code>
             (in the 3.5 source), in
             <code>
              setValue
             </code>
             , all of the XMLEncoding boils down to
            </p>
            <pre>
             <code>
              currentChild.appendChild(xmlSpecialCharsFilter(Object(value)));
             </code>
            </pre>
            <p>
             which is essentially the same as:
            </p>
            <pre>
             <code>
              currentChild.appendChild("null");
             </code>
            </pre>
            <p>
             This code, according to my original fiddle, returns an empty XML element. But why?
            </p>
            <p>
            </p>
            <h1>
             Cause
            </h1>
            <p>
             According to commenter Justin Mclean on bug report
             <a href="https://issues.apache.org/jira/browse/FLEX-33644">
              FLEX-33664
             </a>
             , the following is the culprit (see last two tests in my
             <a href="http://wonderfl.net/c/dd23/read">
              fiddle
             </a>
             which verify this):
            </p>
            <pre>
             <code>
              var thisIsNotNull:XML = &lt;root&gt;null&lt;/root&gt;;
if(thisIsNotNull == null){
    // always branches here, as (thisIsNotNull == null) strangely returns true
    // despite the fact that thisIsNotNull is a valid instance of type XML
}
             </code>
            </pre>
            <p>
             When
             <code>
              currentChild.appendChild
             </code>
             is passed the string
             <code>
              "null"
             </code>
             , it first converts it to a root XML element with text
             <code>
              null
             </code>
             , and then tests that element against the null literal. This is a weak equality test, so either the XML containing null is coerced to the null type, or the null type is coerced to a root xml element containing the string "null", and the test passes where it arguably should fail. One fix might be to always use
             <a href="http://help.adobe.com/en_US/FlashPlatform/reference/actionscript/3/operators.html#strict_equality">
              strict equality
             </a>
             tests when checking XML (or anything, really) for "nullness."
            </p>
            <p>
            </p>
            <h1>
             Solution
            </h1>
            The only reasonable workaround I can think of, short of fixing this bug in every damn version of ActionScript, is to test fields for "null" and
            <strong>
             escape them as
             <a href="http://stackoverflow.com/q/1239466/203705">
              CDATA values
             </a>
             .
            </strong>
            <p>
             <strong>
              CDATA values are the most appropriate way to mutate an entire text value that would otherwise cause encoding/decoding problems.
             </strong>
             Hex encoding, for instance, is meant for individual characters. CDATA values are preferred when you're escaping the entire text of an element. The biggest reason for this is that it maintains human readability.
            </p>

              JBoss
             </a>
             5 server...). I remember extending the SOAP encoder and performing an explicit check on how NaN is handled.

          <td class="answercell">
           <div class="post-text" itemprop="text">
            <p>
             @doc_180 had the right concept, except he is focused on numbers, whereas the original poster had issues with strings.
            </p>
            <p>
             The solution is to change the
             <code>
              mx.rpc.xml.XMLEncoder
             </code>
             file. This is line 121
            </p>
            <pre>
             <code>
              if (content != null)
        result += content;
             </code>
            </pre>
            <p>
             [I looked at Flex 4.5.1 SDK; line numbers may differ in other versions]
            </p>
            <p>
             Basically, the validation fails because 'content is null' and therefore your argument is not added to the outgoing SOAP Packet; thus causing the missing parameter error.
            </p>
            <p>
             You have to extend this class to remove the validation.  Then there is a big snowball up the chain, modifying SOAPEncoder to use your modified XMLEncoder, and then modifying Operation to use your modified SOAPEncoder, and then moidfying WebService to use your alternate Operation class.
            </p>
            <p>
             I spent a few hours on it, but need to move on.  It'll probably take a day or two.
            </p>
            <p>
             You may be able to just fix the XMLEncoder line and do some monkey patching to use your own class.
            </p>
            <p>
             I'll also add that if you switch to using RemoteObject/AMF with ColdFusion, the null is passed without problems.
            </p>
            <hr/>
            <p>
             <strong>
              11/16/2013 update
             </strong>
             :
            </p>
            <p>
             I have one more recent addition to my last comment about RemoteObject/AMF.  If you are using CF10; then properties with a null value on an object are removed from the server side object.  So, you have to check for the properties existence before accessing it or you will get a runtime error.  Check like this:
            </p>
            <pre>
             <code>
              &lt;cfif (structKeyExists(arguments.myObject,'propertyName')&gt;
 &lt;!--- no property code ---&gt;
&lt;cfelse&gt;
 &lt;!--- handle property  normally ---&gt;
&lt;/cfif&gt;
             </code>
            </pre>
            <p>
             This is a change in behavior from CF9; where the null properties would turn into empty strings.
            </p>
            <hr/>
            <p>
             Edit 12/6/2013
            </p>
            <p>
             Since there was a question about how nulls are treated here is a quick sample application to demonstrate how a string "null" will relate to the reserved word null.
            </p>
            <pre>
             <code>
              &lt;?xml version="1.0" encoding="utf-8"?&gt;
&lt;s:Application xmlns:fx="http://ns.adobe.com/mxml/2009" 
               xmlns:s="library://ns.adobe.com/flex/spark" 
               xmlns:mx="library://ns.adobe.com/flex/mx" minWidth="955" minHeight="600" initialize="application1_initializeHandler(event)"&gt;
    &lt;fx:Script&gt;
        &lt;![CDATA[
            import mx.events.FlexEvent;

            protected function application1_initializeHandler(event:FlexEvent):void
            {
                var s :String = "null";
                if(s != null){
                    trace('null string is not equal to null reserved word using the != condition');
                } else {
                    trace('null string is equal to null reserved word using the != condition');
                }

                if(s == null){
                    trace('null string is equal to null reserved word using the == condition');
                } else {
                    trace('null string is not equal to null reserved word using the == condition');
                }

                if(s === null){
                    trace('null string is equal to null reserved word using the === condition');
                } else {
                    trace('null string is not equal to null reserved word using the === condition');
                }

            }

        ]]&gt;
    &lt;/fx:Script&gt;
    &lt;fx:Declarations&gt;
        &lt;!-- Place non-visual elements (e.g., services, value objects) here --&gt;
    &lt;/fx:Declarations&gt;
&lt;/s:Application&gt;
             </code>
            </pre>
            <p>
             The trace output is:
            </p>
            <blockquote>
             <p>
              null string is not equal to null reserved word using the != condition
             </p>
             <p>
              null string is not equal to null reserved word using the == condition
             </p>
             <p>
              null string is not equal to null reserved word using the === condition
             </p>
            </blockquote>

            <p>
             Stringifying a
             <code>
              null
             </code>
             value in
             <a href="http://en.wikipedia.org/wiki/ActionScript">
              ActionScript
             </a>
             will give the string
             <code>
              "NULL"
             </code>
             . My suspicion is that someone has decided that it is, therefore, a good idea to decode the string
             <code>
              "NULL"
             </code>
             as
             <code>
              null
             </code>
             , causing the breakage you see here -- probably because they were passing in
             <code>
              null
             </code>
             objects and getting strings in the database, when they didn't want that (so be sure to check for that kind of bug, too).
            </p>

          <td class="answercell">
           <div class="post-text" itemprop="text">
            <p>
             Well, I guess that Flex' implementation of the SOAP Encoder seems to serialize null values incorrectly. Serializing them as a String Null doesn't seem to be a good solution. The formally correct version seems to be to pass a null value as:
            </p>
            <pre>
             <code>
              &lt;childtag2 xsi:nil="true" /&gt;
             </code>
            </pre>
            <p>
             So the value of "Null" would be nothing else than a valid string, which is exactly what you are looking for.
            </p>
            <p>
             I guess getting this fixed in Apache Flex shouldn't be that hard to get done. I would recommend opening a Jira issue or to contact the guys of the apache-flex mailinglist. However this would only fix the client side. I can't say if ColdFusion will be able to work with null values encoded this way.
            </p>
            <p>
             See also Radu Cotescu's blog post
             <em>
              <a href="http://radu.cotescu.com/how-to-send-null-values-in-soapui-requests/" rel="nofollow">
               How to send null values in soapUI requests
              </a>
             </em>
             .
            </p>
