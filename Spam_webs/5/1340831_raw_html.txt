     <p>
      <a href="/wiki/Transmission_Control_Protocol" title="Transmission Control Protocol">
       Transmission Control Protocol
      </a>
      (TCP) uses a
      <a class="mw-redirect" href="/wiki/Network_congestion-avoidance" title="Network congestion-avoidance">
       network congestion-avoidance
      </a>
      algorithm that includes various aspects of an
      <a href="/wiki/Additive_increase/multiplicative_decrease" title="Additive increase/multiplicative decrease">
       additive increase/multiplicative decrease
      </a>
      (AIMD) scheme, with other schemes such as
      <a href="/wiki/Slow-start" title="Slow-start">
       slow-start
      </a>
      to achieve congestion avoidance.
     </p>
     <p>
      The
      <b>
       TCP congestion-avoidance algorithm
      </b>
      is the primary basis for
      <a class="mw-redirect" href="/wiki/Congestion_control" title="Congestion control">
       congestion control
      </a>
      in the Internet.

     <h2>
      <span class="mw-headline" id="Naming_history">
       Naming history
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=TCP_congestion-avoidance_algorithm&amp;action=edit&amp;section=1" title="Edit section: Naming history">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      Two such variations are those offered by TCP Tahoe and Reno. The two algorithms were retrospectively named after the
      <a class="mw-redirect" href="/wiki/4.3BSD" title="4.3BSD">
       4.3BSD
      </a>
      operating system in which each first appeared (which were themselves named after
      <a href="/wiki/Lake_Tahoe" title="Lake Tahoe">
       Lake Tahoe
      </a>
      and the nearby city of
      <a href="/wiki/Reno,_Nevada" title="Reno, Nevada">
       Reno, Nevada
      </a>
      ). The "Tahoe" algorithm first appeared in 4.3BSD-Tahoe (which was made to support the
      <a href="/wiki/Computer_Consoles_Inc.#Power_5_and_Power_6_computers" title="Computer Consoles Inc.">
       CCI Power 6/32 Tahoe
      </a>
      minicomputer), and was made available to non-AT&amp;T licensees as part of the 4.3BSD Networking Release 1; this ensured its wide distribution and implementation. Improvements, described below, were made in 4.3BSD-Reno and subsequently released to the public as "Networking Release 2" and later 4.4BSD-Lite. The "TCP Foo" names for the algorithms appear to have originated in a 1996 paper by Kevin Fall and Sally Floyd.
      <sup class="reference" id="cite_ref-6">
       <a href="#cite_note-6">
        <span>
         [
        </span>
        6
        <span>
         ]
        </span>
       </a>
      </sup>
     </p>
     <h2>
      <span class="mw-headline" id="TCP_Tahoe_and_Reno">
       TCP Tahoe and Reno
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=TCP_congestion-avoidance_algorithm&amp;action=edit&amp;section=2" title="Edit section: TCP Tahoe and Reno">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      To avoid
      <a class="mw-redirect" href="/wiki/Congestive_collapse" title="Congestive collapse">
       congestion collapse
      </a>
      , TCP uses a multi-faceted congestion-control strategy. For each connection, TCP maintains a
      <i>
       <a href="/wiki/Congestion_window" title="Congestion window">
        congestion window
       </a>
      </i>
      , limiting the total number of unacknowledged packets that may be in transit end-to-end. This is somewhat analogous to TCP's
      <a class="mw-redirect" href="/wiki/Sliding_Window_Protocol" title="Sliding Window Protocol">
       sliding window
      </a>
      used for
      <a href="/wiki/Transmission_Control_Protocol#Flow_control" title="Transmission Control Protocol">
       flow control
      </a>
      . TCP uses a mechanism called
      <i>
       <a href="/wiki/Slow-start" title="Slow-start">
        slow start
       </a>
      </i>
      <sup class="reference" id="cite_ref-7">
       <a href="#cite_note-7">
        <span>
         [
        </span>
        7
        <span>
         ]
        </span>
       </a>
      </sup>
      to increase the congestion window after a connection is initialized and after a timeout. It starts with a window of two times the
      <a href="/wiki/Maximum_segment_size" title="Maximum segment size">
       maximum segment size
      </a>
      (MSS). Although the initial rate is low, the rate of increase is very rapid:
      <b>
       for every packet
      </b>
      acknowledged, the congestion window increases by 1 MSS so that the congestion window effectively doubles for every
      <a href="/wiki/Round-trip_delay_time" title="Round-trip delay time">
       round-trip time
      </a>
      (RTT). When the congestion window exceeds the
      <i>
       ssthresh
      </i>
      threshold, the algorithm enters a new state, called
      <a class="mw-redirect" href="/wiki/Congestion_avoidance" title="Congestion avoidance">
       congestion avoidance
      </a>
      . In some implementations (
      <i>
       e.g.
      </i>
      , Linux), the initial ssthresh is large, and so the first slow start usually ends after a loss. However, ssthresh is updated at the end of each slow start, and will often affect subsequent slow starts triggered by
      <a href="/wiki/Timeout_(computing)" title="Timeout (computing)">
       timeouts
      </a>
      .
     </p>
     <p>
      <a class="mw-redirect" href="/wiki/Congestion_avoidance" title="Congestion avoidance">
       Congestion avoidance
      </a>
      : As long as non-duplicate ACKs are received, the congestion window is additively increased by one MSS every round trip time. When a packet is lost, the likelihood of duplicate ACKs being received is very high (it's possible though unlikely that the stream just underwent extreme packet reordering, which would also prompt duplicate ACKs). The behavior of Tahoe and Reno differ in how they detect and react to packet loss:
     </p>
     <ul>
      <li>
       Tahoe: Common Tahoe implementations detect congestion only by setting a timer for receiving a related ACK. Tahoe sets the slow start threshold to half of the current congestion window, reduces the congestion window to 1 MSS, and resets to slow-start state.
       <sup class="reference" id="cite_ref-kurose_8-0">
        <a href="#cite_note-kurose-8">
         <span>
          [
         </span>
         8
         <span>
          ]
         </span>
        </a>
       </sup>
      </li>
      <li>
       Reno: If three duplicate ACKs are received (i.e., four ACKs acknowledging the same packet, which are not piggybacked on data, and do not change the receiver's advertised window), Reno will halve the congestion window (instead of setting it to 1 MSS like Tahoe), set the slow start threshold equal to the new congestion window, perform a fast retransmit, and enter a phase called
       <a href="/wiki/Slow-start#Fast_recovery" title="Slow-start">
        Fast Recovery
       </a>
       . If an ACK times out, slow start is used as it is with Tahoe.
       <sup class="reference" id="cite_ref-kurose_8-1">
        <a href="#cite_note-kurose-8">
         <span>
          [
         </span>
         8
         <span>
          ]
         </span>
        </a>
       </sup>
      </li>
     </ul>
     <p>
      The two main differences between Tahoe and Reno are:
     </p>
     <ol>
      <li>
       Tahoe only uses a timeout for detecting congestion, while Reno uses timeout and Fast-Retransmit
      </li>
      <li>
       Tahoe sets the congestion window to 1 after packet loss, while Reno sets it to half of the latest congestion window.
      </li>
     </ol>
     <p>
      <a href="/wiki/Slow-start#Fast_recovery" title="Slow-start">
       Fast Recovery
      </a>
      (Reno only): In this state, TCP retransmits the missing packet that was signaled by three duplicate ACKs, and waits for an acknowledgment of the entire transmit window before returning to congestion avoidance. If there is no acknowledgment, TCP Reno experiences a timeout and enters the slow-start state.
     </p>
     <p>
      Both algorithms reduce congestion window to 1 MSS on a timeout event.
     </p>
     <p>
      A frozen state is sometimes entered by both algorithms where the congestion window is set to 0 (zero).
      <sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">
       [
       <i>
        <a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed">
         <span title="This claim needs references to reliable sources. (January 2014)">
          citation needed
         </span>
        </a>
       </i>
       ]
      </sup>
     </p>
     <h2>
      <span class="mw-headline" id="TCP_Vegas">
       TCP Vegas
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=TCP_congestion-avoidance_algorithm&amp;action=edit&amp;section=3" title="Edit section: TCP Vegas">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <div class="hatnote relarticle mainarticle">
      Main article:
      <a href="/wiki/TCP_Vegas" title="TCP Vegas">
       TCP Vegas
      </a>
     </div>
     <p>
      Until the mid-1990s, all of TCP's set timeouts and measured round-trip delays were based upon only the last transmitted packet in the transmit buffer.
      <a href="/wiki/University_of_Arizona" title="University of Arizona">
       University of Arizona
      </a>
      researchers Larry Peterson and
      <a href="/wiki/Lawrence_Brakmo" title="Lawrence Brakmo">
       Lawrence Brakmo
      </a>
      introduced TCP Vegas, named after the largest Nevada city, in which timeouts were set and round-trip delays were measured for every packet in the transmit buffer. In addition, TCP Vegas uses additive increases in the congestion window. This variant was not widely deployed outside Peterson's laboratory. In a comparison study of various TCP congestion control algorithms, TCP Vegas appeared to be the smoothest followed by TCP CUBIC.
      <sup class="reference" id="cite_ref-9">
       <a href="#cite_note-9">
        <span>
         [
        </span>
        9
        <span>
         ]
        </span>
       </a>
      </sup>
     </p>
     <p>
      TCP Vegas was deployed as default congestion control method for
      <a href="/wiki/DD-WRT" title="DD-WRT">
       DD-WRT
      </a>
      firmwares v24 SP2.
      <sup class="reference" id="cite_ref-10">
       <a href="#cite_note-10">
        <span>
         [
        </span>
        10
        <span>
         ]
        </span>
       </a>
      </sup>
     </p>
     <h2>
      <span class="mw-headline" id="TCP_New_Reno">
       TCP New Reno
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=TCP_congestion-avoidance_algorithm&amp;action=edit&amp;section=4" title="Edit section: TCP New Reno">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      TCP New Reno, defined by
      <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc6582" rel="nofollow">
       RFC 6582
      </a>
      (which obsoletes previous definitions in
      <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc3782" rel="nofollow">
       RFC 3782
      </a>
      and
      <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc2582" rel="nofollow">
       RFC 2582
      </a>
      ), improves retransmission during the fast-recovery phase of TCP Reno. During fast recovery, for every duplicate ACK that is returned to TCP New Reno, a new unsent packet from the end of the congestion window is sent, to keep the transmit window full. For every ACK that makes partial progress in the sequence space, the sender assumes that the ACK points to a new hole, and the next packet beyond the ACKed sequence number is sent.
     </p>
     <p>
      Because the timeout timer is reset whenever there is progress in the transmit buffer, this allows New Reno to fill large holes, or multiple holes, in the sequence space  much like TCP SACK. Because New Reno can send new packets at the end of the congestion window during fast recovery, high throughput is maintained during the hole-filling process, even when there are multiple holes, of multiple packets each. When TCP enters fast recovery it records the highest outstanding unacknowledged packet sequence number. When this sequence number is acknowledged, TCP returns to the congestion avoidance state.
     </p>
     <p>
      A problem occurs with New Reno when there are no packet losses but instead, packets are reordered by more than 3 packet sequence numbers. When this happens, New Reno mistakenly enters fast recovery, but when the reordered packet is delivered, ACK sequence-number progress occurs and from there until the end of fast recovery, every bit of sequence-number progress produces a duplicate and needless retransmission that is immediately ACKed.
     </p>
     <p>
      New Reno performs as well as SACK at low packet error rates, and substantially outperforms Reno at high error rates.
     </p>
     <h2>
      <span class="mw-headline" id="TCP_Hybla">
       TCP Hybla
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=TCP_congestion-avoidance_algorithm&amp;action=edit&amp;section=5" title="Edit section: TCP Hybla">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      TCP Hybla
      <sup class="reference" id="cite_ref-11">
       <a href="#cite_note-11">
        <span>
         [
        </span>
        11
        <span>
         ]
        </span>
       </a>
      </sup>
      aims to eliminate penalization of TCP connections that incorporate a high-latency terrestrial or satellite radio link, due to their longer round trip times. It stems from an analytical evaluation of the congestion window dynamics, which suggests the necessary modifications to remove the performance dependence on RTT.
     </p>
     <h2>
      <span class="mw-headline" id="TCP_BIC">
       TCP BIC
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=TCP_congestion-avoidance_algorithm&amp;action=edit&amp;section=6" title="Edit section: TCP BIC">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <div class="hatnote relarticle mainarticle">
      Main article:
      <a href="/wiki/BIC_TCP" title="BIC TCP">
       BIC TCP
      </a>
     </div>
     <p>
      <a href="/wiki/BIC_TCP" title="BIC TCP">
       Binary Increase Congestion control
      </a>
      is an implementation of
      <a href="/wiki/Transmission_Control_Protocol" title="Transmission Control Protocol">
       TCP
      </a>
      with an optimized congestion control algorithm for high speed networks with high latency (called
      <b>
       LFN
      </b>
      ,
      <a class="mw-redirect" href="/wiki/Long_fat_network" title="Long fat network">
       long fat networks
      </a>
      , in
      <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc1072" rel="nofollow">
       RFC 1072
      </a>
      <sup class="reference" id="cite_ref-12">
       <a href="#cite_note-12">
        <span>
         [
        </span>
        12
        <span>
         ]
        </span>
       </a>
      </sup>
      ). BIC is used by default in
      <a href="/wiki/Linux_kernel" title="Linux kernel">
       Linux kernels
      </a>
      2.6.8 through 2.6.18.
     </p>
     <h2>
      <span class="mw-headline" id="TCP_CUBIC">
       TCP CUBIC
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=TCP_congestion-avoidance_algorithm&amp;action=edit&amp;section=7" title="Edit section: TCP CUBIC">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <div class="hatnote relarticle mainarticle">
      Main article:
      <a href="/wiki/CUBIC_TCP" title="CUBIC TCP">
       CUBIC TCP
      </a>
     </div>
     <p>
      <a href="/wiki/CUBIC_TCP" title="CUBIC TCP">
       CUBIC
      </a>
      is a less aggressive and more systematic derivative of BIC, in which the window is a cubic function of time since the last congestion event, with the inflection point set to the window prior to the event. CUBIC is used by default in
      <a href="/wiki/Linux_kernel" title="Linux kernel">
       Linux kernels
      </a>
      from version 2.6.19 to 3.1.
     </p>
     <h2>
      <span class="mw-headline" id="Compound_TCP">
       Compound TCP
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=TCP_congestion-avoidance_algorithm&amp;action=edit&amp;section=8" title="Edit section: Compound TCP">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <div class="hatnote relarticle mainarticle">
      Main article:
      <a href="/wiki/Compound_TCP" title="Compound TCP">
       Compound TCP
      </a>
     </div>
     <p>
      Compound TCP is a
      <a href="/wiki/Microsoft" title="Microsoft">
       Microsoft
      </a>
      implementation of TCP which maintains two different congestion windows simultaneously, with the goal of achieving good performance on LFNs while not impairing
      <a href="/wiki/Fairness_measure" title="Fairness measure">
       fairness
      </a>
      . It has been widely deployed with Microsoft
      <a href="/wiki/Windows_Vista" title="Windows Vista">
       Windows Vista
      </a>
      and
      <a href="/wiki/Windows_Server_2008" title="Windows Server 2008">
       Windows Server 2008
      </a>
      and has been ported to older Microsoft Windows versions as well as
      <a href="/wiki/Linux" title="Linux">
       Linux
      </a>
      .
     </p>
     <h2>
      <span class="mw-headline" id="TCP_Proportional_Rate_Reduction">
       TCP Proportional Rate Reduction
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=TCP_congestion-avoidance_algorithm&amp;action=edit&amp;section=9" title="Edit section: TCP Proportional Rate Reduction">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      TCP Proportional Rate Reduction (PRR)
      <sup class="reference" id="cite_ref-13">
       <a href="#cite_note-13">
        <span>
         [
        </span>
        13
        <span>
         ]
        </span>
       </a>
      </sup>
      is an algorithm designed to improve the accuracy of data sent during recovery. The algorithm ensures that the window size after recovery is as close as possible to the
      <a href="/wiki/Slow-start" title="Slow-start">
       Slow-start
      </a>
      threshold. In tests performed by
      <a href="/wiki/Google" title="Google">
       Google
      </a>
      , PRR resulted in a 3-10% reduction in average latency and recovery timeouts reduced by 5%.
      <sup class="reference" id="cite_ref-14">
       <a href="#cite_note-14">
        <span>
         [
        </span>
        14
        <span>
         ]
        </span>
       </a>
      </sup>
      PRR is used by default in
      <a href="/wiki/Linux_kernel" title="Linux kernel">
       Linux kernels
      </a>
      since version 3.2.
      <sup class="reference" id="cite_ref-15">
       <a href="#cite_note-15">
        <span>
         [
        </span>
        15
        <span>
         ]
        </span>
       </a>
      </sup>
     </p>

      <a href="#TCP_New_Reno">
       TCP New Reno
      </a>
      was the most commonly implemented algorithm, SACK support is very common and is an extension to Reno/New Reno. Most others are competing proposals which still need evaluation. Starting with 2.6.8 the Linux kernel switched the default implementation from NewReno to
      <a href="/wiki/BIC_TCP" title="BIC TCP">
       BIC
      </a>
      . The default implementation was again changed to CUBIC in the 2.6.19 version.
      <a href="/wiki/FreeBSD" title="FreeBSD">
       FreeBSD
      </a>
      uses NewReno as the default algorithm. However, it supports a number of other choices.
      <sup class="reference" id="cite_ref-23">
       <a href="#cite_note-23">
        <span>
         [
        </span>

        <span>

        </span>

     <p>
      When the per-flow product of bandwidth and latency increases, regardless of the queuing scheme, TCP becomes inefficient and prone to instability. This becomes increasingly important as the Internet evolves to incorporate very high-bandwidth optical links.
     </p>
     <p>
      TCP Interactive (iTCP)
      <sup class="reference" id="cite_ref-24">
       <a href="#cite_note-24">
        <span>
         [
        </span>
        24
        <span>
         ]
        </span>
       </a>
      </sup>
      allows applications to subscribe to TCP events and respond accordingly enabling various functional extensions to TCP from outside TCP layer. Most TCP congestion schemes work internally. iTCP additionally enables advanced applications to directly participate in congestion control such as to control the source generation rate.
     </p>
     <p>
      <a href="/wiki/Zeta-TCP" title="Zeta-TCP">
       Zeta-TCP
      </a>
      detects the congestions from both the latency and loss rate measures, and applies different
      <a href="/wiki/Congestion_window" title="Congestion window">
       CWND
      </a>
      backoff strategies based on the likelihood of the congestions to maximize the
      <a href="/wiki/Goodput" title="Goodput">
       goodput
      </a>
      . It also has a couple of other improvements to accurately detect the packet losses, avoiding RTO retransmission; and accelerate/control the inbound (download) traffic.
     </p>
