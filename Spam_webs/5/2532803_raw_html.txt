             <h3 class="post-title entry-title">
              <a href="http://www.dbacorner.com/2007/08/status-25-cannot-connect-on-socket.html">
               ORA-19511: VERITAS NetBackup: Status 25: Cannot connect on socket
              </a>
             </h3>
             <div class="post-header">
              <div class="post-header-line-1">
              </div>
             </div>
             <div class="post-body entry-content">
              <a href="http://seer.support.veritas.com/docs/264886.htm">
               Status 25: "Cannot connect on socket"
              </a>
              <br/>
              <br/>
              <div class="bodytext" id="EXACT_ERROR">
               <span class="BodyToolboxHeader">
                Exact Error Message
               </span>
               <br/>
               <span class="exacterror">
                25 cannot connect on socket
               </span>
               <br/>
               <br/>
              </div>
              <span class="BodyToolboxHeader">
               Details:
              </span>
              <br/>
              <b>
               Overview:
              </b>
              <br/>
              When performing backups or restores,  socket errors are being produced.
              <br/>
              <br/>
              <b>
               Troubleshooting:
              </b>
              <br/>
              <b>
               Please  follow all steps within the VERITAS NetBackup (tm) Troubleshooting Guide or the  NetBackup Troubleshooter within the Activity Monitor for this status code before  continuing.
              </b>
              <br/>
              <br/>
              <b>
               Please confirm hardware and software compatibility  before continuing. A list of compatible hardware and software may be obtained  within the VERITAS NetBackup Release Notes or on the VERITAS Support Web  site.
              </b>
              <br/>
              <br/>
              If the above does not resolve the issue, please  continue.
              <br/>
              <br/>
              <br/>
              <b>
               1.  Status code 25 caused by no open sockets on the  Master/Media Server
              </b>
              <br/>
              <br/>
              Please look for the following log  messages:
              <br/>
              <br/>
              <b>
               Master Log Files:
              </b>
              <br/>
              <br/>
              <b>
               Media Server Log  Files:
              </b>
              <br/>
              <b>
               bpbrm:
              </b>
              <br/>
              <tt>
               <span class="FIXEDFONT">
                02:44:21.877  [1092.2944]  nb_bind_on_port_addr: port 4998  unavailable
               </span>
              </tt>
              <br/>
              <tt>
               <span class="FIXEDFONT">
                02:44:21.877 [1092.2944]   nb_getsockconnected: cannot connect() to MasterServer, Only one usage  of each socket address (protocol/network address/port) is normally permitted.
               </span>
              </tt>
              <br/>
              <tt>
               <span class="FIXEDFONT">
                02:44:21.877 [1092.2944]   db_begin: nb_getsockconnected() failed: Only one usage of each socket address  (protocol/network address/port) is normally  permitted.  (10048)
               </span>
              </tt>
              <br/>
              <tt>
               <span class="FIXEDFONT">
                02:44:21.877  [1092.2944]  db_FLISTsend: db_begin() failed: cannot connect on  socket
               </span>
              </tt>
              <br/>
              <tt>
               <span class="FIXEDFONT">
                02:44:21.877 [1092.2944]   bpbrm handle_backup: db_FLISTsend failed: cannot connect on socket  (25)
               </span>
              </tt>
              <br/>
              <tt>
               <span class="FIXEDFONT">
                02:44:21.877 [1092.2944]   nb_bind_on_port_addr: port 4999  unavailable
               </span>
              </tt>
              <br/>
              <br/>
              <b>
               Client Log  Files:
              </b>
              <br/>
              <br/>
              <b>
               Resolution:
              </b>
              <br/>
              A socket in a
              <b>
               TIME_WAIT
              </b>
              state may  be made available sooner by reducing the
              <b>
               tcp_time_wait_interval
              </b>
              parameter. The default for this parameter on Solaris is 240000 milliseconds  or 4 minutes. The default for Hewlett-Packard (HP) is a value of 60000  milliseconds or 1 minute. The default on AIX is set in increments of 15 seconds.  The default value is 1 or 15 seconds. On AIX, this is the lowest it may be set  to. VERITAS has found that the
              <b>
               tcp_time_wait_interval
              </b>
              can often be set to  1 second with no adverse effects. However, customers should adjust the
              <b>
               TCP_TIME_WAIT_INTERVAL
              </b>
              in increments; lowering the value until backups no  longer fail with the status 25.
              <br/>
              <br/>
              The value of this parameter may be  obtained and changed by running the following command:
              <br/>
              <br/>
              For Solaris 2.6 or  earlier:
              <br/>
              <br/>
              <tt>
               <span class="FIXEDFONT">
                #ndd -get /dev/tcp  tcp_close_wait_interval
               </span>
              </tt>
              <br/>
              <tt>
               <span class="FIXEDFONT">
                #240000
               </span>
              </tt>
              <br/>
              <tt>
               <span class="FIXEDFONT">
                #ndd  -set /dev/tcp tcp_close_wait_interval 10000
               </span>
              </tt>
              (10  seconds)
              <br/>
              <br/>
              <br/>
              For Solaris 7 or  later:
              <br/>
              <br/>
              <tt>
               <span class="FIXEDFONT">
                #ndd -get /dev/tcp  tcp_time_wait_interval
               </span>
              </tt>
              <br/>
              <tt>
               <span class="FIXEDFONT">
                #240000
               </span>
              </tt>
              <br/>
              <tt>
               <span class="FIXEDFONT">
                #ndd  -set /dev/tcp tcp_time_wait_interval 10000
               </span>
              </tt>
              (10  seconds)
              <br/>
              <br/>
              <br/>
              The
              <tt>
               <span class="FIXEDFONT">
                ndd
               </span>
              </tt>
              command  makes the change immediately, without a need for a reboot. This setting will go  back to default after a reboot. To make the change permanent, the command can be  added to the appropriate TCP/IP startup script. On Solaris, this is
              <b>
               /etc/rc2.d/S69inet
              </b>
              .
              <br/>
              <br/>
              For HP-UX 11, use the following  command:
              <br/>
              <br/>
              <tt>
               <span class="FIXEDFONT">
                #ndd -get /dev/tcp  tcp_time_wait_interval
               </span>
              </tt>
              <br/>
              <tt>
               <span class="FIXEDFONT">
                #60000
               </span>
              </tt>
              <br/>
              <tt>
               <span class="FIXEDFONT">
                #ndd  -set /dev/tcp tcp_time_wait_interval 10000 (10  seconds)
               </span>
              </tt>
              <br/>
              <br/>
              <b>
               Note:
              </b>
              The equivalent command on HP-UX 10.x is
              <tt>
               <span class="FIXEDFONT">
                nettune
               </span>
              </tt>
              instead of
              <tt>
               <span class="FIXEDFONT">
                ndd
               </span>
              </tt>
              .
              <br/>
              <br/>
              The
              <tt>
               <span class="FIXEDFONT">
                ndd
               </span>
              </tt>
              command makes the change  immediately, without a need for a reboot. This setting will go back to default  after a reboot. To make the change permanent, the command can be added to the  appropriate TCP/IP startup script. On HP-UX 11, see
              <b>
               /etc/rc.config.d/nddconf
              </b>
              for examples on how to set it.
              <br/>
              <br/>
              It is not  necessary to change the
              <b>
               tcp_timewait
              </b>
              default parameter on AIX, but for  reference:
              <br/>
              <br/>
              To display the value, enter:
              <br/>
              <tt>
               <span class="FIXEDFONT">
                #no -a
               </span>
              </tt>
              <br/>
              or
              <br/>
              <tt>
               <span class="FIXEDFONT">
                #no -o  tcp_timewait
               </span>
              </tt>
              <br/>
              <br/>
              To change the value,  enter:
              <br/>
              <tt>
               <span class="FIXEDFONT">
                #no -o  tcp_timewait=1
               </span>
              </tt>
              <br/>
              <br/>
              Where
              <value>
               is a positive integer,  with each increment representing 15 seconds. The default
               <value>
                is 1 for  15 seconds. The change takes effect immediately. The change is effective until  the next boot. A permanent change is made by adding the command to
                <b>
                 /etc/rc.net
                </b>
                .
                <br/>
                <br/>
                <b>
                 Windows System:
                </b>
                <br/>
                Ports in
                <b>
                 TIME_WAIT
                </b>
                will be seen by running
                <tt>
                 <span class="FIXEDFONT">
                  netstat -a
                 </span>
                </tt>
                from  a command prompt on the master or media server. This can be an indication that  ports are not being released quickly enough to allow another connection on that  port. The registry setting
                <b>
                 TcpTimedWaitDelay
                </b>
                can be added to reduce the  time a port will stay in this state.
                <br/>
                <br/>
                1. Go to
                <b>
                 Start | Run
                </b>
                , type
                <tt>
                 <span class="FIXEDFONT">
                  regedit
                 </span>
                </tt>
                , and click
                <b>
                 OK
                </b>
                <br/>
                2.  Navigate to
                <b>
                 HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters
                </b>
                <br/>
                3.  Highlight the
                <b>
                 Parameters
                </b>
                key, right-click and select
                <b>
                 New | DWORD  Value
                </b>
                <br/>
                4. Type the name
                <b>
                 TcpTimedWaitDelay
                </b>
                <br/>
                5. Double-click
                <b>
                 TcpTimedWaitDelay
                </b>
                and enter a decimal value of 30
                <br/>
                By default, this  value is 240 seconds (4 minutes). It is recommended that this be changed to a  value between 30-60 (seconds) in decimal, to decrease the wait time before a  port becomes available.
                <br/>
                <br/>
                <b>
                 Description
                </b>
                :
                <br/>
                This parameter  determines the length of time that a connection will stay in the
                <b>
                 TIME_WAIT
                </b>
                state when being closed. While a connection is in the
                <b>
                 TIME_WAIT
                </b>
                state, the socket pair cannot be re-used. This is also known as  the "2MSL" state, as by RFC the value should be twice the maximum segment  lifetime on the network. See RFC793 for further details.
               </value>
              </value>

                       <br/>
                       Session rejected: TCP/IP connection failure [Same as ANS1017E]
                       <br/>
                       This is what the client sees and reports, but has no idea why.
                       <br/>
                       The cause is best sought in the ADSM server Activity Log for that time.
                       <br/>
                       Could be a real datacomm problem; or...
                       <br/>
                       Grossest problem: the TSM server is down.
                       <br/>
                       <br/>
                       If you get this condition after supposedly changing the client and server to use a different port number (e.g., 1502), and the Activity Log has no significant information about the attempted session, use 'netstat' or 'lsof' or similar utility in the server operating system to
                       <br/>
                       verify that the *SM server is actually serving the port number that you believe it should be. (You *did* code the port numbers into both the client and server options files, right?)
                       <br/>
                       An administrator may have done a 'CANcel SEssion'.
                       <br/>
                       If during a Backup, likely the server cancelling it due to higher priority task like DB Backup starting and needing a tape drive...particularly when there is a drive shortage.  Look in the server Activity Log around that time and you will likely see  "ANR0492I All drives in use. Session 22668 for node ________ (AIX) being preempted by higher priority operation.".
                       <br/>
                       <br/>
                       Or look in the Activity Log for a "ANR0481W Session NNN for node
                       <nodename>
                        (
                        <nodetype>
                         ) terminated - client did not respond within NN seconds." message, which reflects a server COMMTimeout value that is too low.
                         <br/>
                         <br/>
                         Message "ANR0482W Session
                         <sessionnumber>
                          for
                          <nodenode>
                           name (
                           <clientplatform>
                            ) terminated - idle for more than N minutes." is telling you that the sever IDLETimeout value is too low.  Remember that longstanding clients may take considerable time to rummage around in their file systems looking for new files to back up.
                            <br/>
                            <br/>
                            Another problem is in starting your client scheduler process from
                            <br/>
                            /etc/inittab, but failing to specify redirection - you need:
                            <br/>
                            dsmc::once:/usr/bin/dsmc sched &gt; /dev/null 2&gt;&amp;1    # TSM scheduler
                            <br/>
                            <br/>
                            An unusual cause is in having the client and server defined to use the same port number!
                            <br/>
                            <br/>
                            Might also be a firewall rejecting the TSM client as it tries to reach the server through that firewall.
                            <br/>
                            <br/>
                            <br/>
                           </clientplatform>
                          </nodenode>
                         </sessionnumber>
