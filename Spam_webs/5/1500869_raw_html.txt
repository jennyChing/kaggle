     <h2>
      <span class="mw-headline" id="Introduction">
       Introduction
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=1" title="Edit section: Introduction">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      This book addresses 32-bit Windows commands applicable to modern versions of Windows based on the Windows NT environment. It does not address commands that are specific to DOS environments and to DOS-based operating systems, such as Windows 95, Windows 98, and Windows Me, whose Microsoft-supplied command interpreters are in fact DOS programs, not Win32 programs.
     </p>
     <p>
      You can find out which version of cmd.exe you are running using the
      <a href="#VER">
       VER
      </a>
      command.
     </p>
     <p>
      This book first describes using the Windows NT command interpreter, how it receives, parses, and processes commands from users. Then it describes various commands available.
     </p>
     <p>
      To obtain an extensive list of Windows commands and their short summaries, open the command prompt on any Windows computer, and type
      <tt>
       help
      </tt>
      . To find out about a particular command, type the name of the command followed by "/?".
     </p>
     <p>
      The subject of this book is also known as "batch programming", even though "batch" refers not only to batch files for MS DOS and Windows command interpreter. Other subject terms include "batch file programming", "batch file scripting", "Windows batch command", "Windows batch file", "Windows command line", "Windows command prompt", and "Windows shell scripting".
     </p>
     <h2>
      <span class="mw-headline" id="Using_the_Windows_command_interpreter">
       Using the Windows command interpreter
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=2" title="Edit section: Using the Windows command interpreter">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <h3>
      <span class="mw-headline" id="How_a_command_line_is_interpreted">
       How a command line is interpreted
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=3" title="Edit section: How a command line is interpreted">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      The parsing of a command line into a sequence of commands is complex, and varies subtly from command interpreter to command interpreter. There are, however, four main components:
     </p>
     <dl>
      <dt>
       Variable substitution
      </dt>
      <dd>
       A command line is scanned for variable specifications, and any found are replaced with the contents of those variables.
      </dd>
      <dt>
       Quoting
      </dt>
      <dd>
       Special characters can be quoted, to remove their special meanings.
      </dd>
      <dt>
       Syntax
      </dt>
      <dd>
       Command lines are developed into a sequence of commands according to a syntax.
      </dd>
      <dt>
       Redirection
      </dt>
      <dd>
       Redirection specifications are applied, and removed from the command line, before an individual command in a sequence is executed.
      </dd>
     </dl>
     <h4>
      <span class="mw-headline" id="Variable_substitution">
       Variable substitution
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=4" title="Edit section: Variable substitution">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h4>
     <p>
      Command lines can contain variable specifications. These comprise a
      <tt>
       %
      </tt>
      character followed by a name. The name is ended by a second
      <tt>
       %
      </tt>
      character, except in special cases such as the batch file parameters
      <tt>
       %1
      </tt>
      ,
      <tt>
       %2
      </tt>
      , and so forth.
     </p>
     <p>
      Variable specifications are replaced with values. The value used to replace a variable specification is as follows:
     </p>
     <ul>
      <li>
       For variable specification names that match the names of environment variables, the replacement is the value of the named environment variable. For example:
       <tt>
        %PATH%
       </tt>
       is replaced by the value of the PATH environment variable.
      </li>
      <li>
       For variable specifications that name batch file parameters (i.e. that are non-negative decimal numbers), the replacement is the value of the parameter taken from the arguments with which the batch file was invoked (subject to any subsequent modifications by the
       <b>
        SHIFT
       </b>
       command). For example:
       <tt>
        %2
       </tt>
       is replaced by the value of the second batch file parameter.
      </li>
     </ul>
     <h5>
      <span class="mw-headline" id="Special_names">
       Special names
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=5" title="Edit section: Special names">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h5>
     <p>
      Some variable names are not visible using
      <a href="#SET">
       SET
      </a>
      command. Rather, they are made available for reading using the% notation. To find out about them, type "help set".
     </p>
     <p>
      Special variable names and what they expand to:
     </p>
     <table border="1" style="border-collapse: collapse;">
      <tr>
       <th>
        Name
       </th>
       <th>
        Replacement Value Used
       </th>
      </tr>
      <tr>
       <td>
        %CD%
       </td>
       <td>
        The current directory, not ending in a slash character if it is not in the root directory of the current drive
       </td>
      </tr>
      <tr>
       <td>
        %TIME%
       </td>
       <td>
        The system time in HH:MM:SS.mm format.
       </td>
      </tr>
      <tr>
       <td>
        %DATE%
       </td>
       <td>
        The system date in a format specific to localization.
       </td>
      </tr>
      <tr>
       <td>
        %RANDOM%
       </td>
       <td>
        A generated pseudo-random number between 0 and 32767.
       </td>
      </tr>
      <tr>
       <td>
        %ERRORLEVEL%
       </td>
       <td>
        The error level returned by the last executed command, or by the last called batch script.
       </td>
      </tr>
      <tr>
       <td>
        %CMDEXTVERSION%
       </td>
       <td>
        The version number of the Command Processor Extensions currently used by cmd.exe.
       </td>
      </tr>
      <tr>
       <td>
        %CMDCMDLINE%
       </td>
       <td>
        The content of the command line used when the current cmd.exe was started.
       </td>
      </tr>
     </table>
     <p>
      Links:
     </p>
     <ul>
      <li>
       <a class="external text" href="http://ss64.com/nt/syntax-variables.html" rel="nofollow">
        Windows Environment Variables at ss64.com
       </a>
      </li>
      <li>
       <a class="external text" href="http://technet.microsoft.com/en-us/library/bb490954.aspx" rel="nofollow">
        Command shell overview at Microsoft
       </a>
      </li>
     </ul>
     <h4>
      <span class="mw-headline" id="Quoting">
       Quoting
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=6" title="Edit section: Quoting">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h4>
     <p>
      To prevent the metacharacters that control command syntax and redirection from having their special meanings, quoting is used. This takes two forms:
     </p>
     <ul>
      <li>
       Although they don't process quotation marks, in command arguments, specially themselves, but simply pass them along as-is to the commands executed, command interpreters
       <i>
        do
       </i>
       recognise when quotation marks (") surround metacharacters. The &amp; metacharacter does not have its usual effect on command syntax if it is within a pair of quotation marks.
       <dl>
        <dd>
         e.g. The command line
         <tt>
          echo "&amp;"
         </tt>
         will invoke the
         <b>
          ECHO
         </b>
         command passing it the three characters
         <tt>
          "&amp;"
         </tt>
         as its command tail, rather than split the command line in twain at the &amp; character as would occur if the character were not within quotation marks.
        </dd>
       </dl>
      </li>
      <li>
       The "escape" character (always a
       <tt>
        ^
       </tt>
       in Microsoft's CMD) used in front of a metacharacter also prevents that metacharacter from having its usual effect on command syntax. The "escape" character itself is stripped from the command line before it is passed to the command actually being invoked.
       <dl>
        <dd>
         e.g. The command line
         <tt>
          echo ^&amp;
         </tt>
         will invoke the
         <b>
          ECHO
         </b>
         command passing it the character
         <tt>
          &amp;
         </tt>
         as its command tail, rather than split the command line in twain at the &amp; character as would otherwise occur.
        </dd>
       </dl>
      </li>
     </ul>
     <h4>
      <span class="mw-headline" id="Syntax">
       Syntax
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=7" title="Edit section: Syntax">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h4>
     <p>
      Command lines are developed into a sequence of commands according to a syntax. In that syntax,
      <i>
       simple commands
      </i>
      may be combined to form
      <i>
       pipelines
      </i>
      , which may in turn be combined to form
      <i>
       compound commands
      </i>
      , which finally may be turned into
      <i>
       parenthesized commands
      </i>
      .
     </p>
     <p>
      A simple command is just a command name, a command tail, and some redirection specifications. An example of a simple command is
      <tt>
       dir *.txt &gt; somefile
      </tt>
      .
     </p>
     <p>
      A pipeline is several simple commands joined together with the "pipe" metacharacter"|", also known as the "vertical bar". The standard output of the simple command preceding each vertical bar is connected to the standard input of the simple command following it, via a pipe. The command interpreter runs all of the simple commands in the pipeline in parallel. An example of a pipeline (comprising two simple commands) is
      <tt>
       dir *.txt | more
      </tt>
      .
     </p>
     <p>
      A compound command is a set of pipelines separated by conjunctions. The pipelines are executed sequentially, one after the other, and the conjunction controls whether the command interpreter executes the next pipeline or not. An example of a compound command (comprising two pipelines, which themselves are just simple commands) is
      <tt>
       move file.txt file.bak &amp;&amp; dir &gt; file.txt
      </tt>
      .
     </p>
     <p>
      The conjunctions are:
     </p>
     <dl>
      <dt>
       &amp;
      </dt>
      <dd>
       The simplest conjunction. The next pipeline is always executed after the current one has completed executing.
      </dd>
      <dt>
       &amp;&amp;
      </dt>
      <dd>
       A positive conditional conjunction. The next pipeline is executed if the current one completes executing with a zero exit status.
      </dd>
      <dt>
       ||
      </dt>
      <dd>
       A negative conditional conjunction. The next pipeline is executed if the current one completes executing with a non-zero exit status.
      </dd>
     </dl>
     <p>
      A parenthesized command is a compound command enclosed in parentheses (i.e.
      <tt>
       (
      </tt>
      and
      <tt>
       )
      </tt>
      ). From the point of view of syntax, this turns a compound command into a simple command, whose overall output can be redirected.
     </p>
     <p>
      For example: The command line
      <tt>
       ( pushd temp &amp; dir &amp; popd ) &gt; somefile
      </tt>
      causes the standard output of the entire compound command
      <tt>
       ( pushd temp &amp; dir &amp; popd )
      </tt>
      to be redirected to
      <i>
       somefile
      </i>
      .
     </p>
     <h4>
      <span class="mw-headline" id="Redirection">
       Redirection
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=8" title="Edit section: Redirection">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h4>
     <p>
      Redirection specifications are applied, and removed from the command line, before an individual command in a sequence is executed. Redirection specifications control where the standard input, standard output, and standard error file handles for a simple command point. They override any effects to those file handles that may have resulted from pipelining. (See the preceding section on command syntax.) Redirection signs &gt; and &gt;&gt; can be prefixed with 1 for the standard output (same as no prefix) or 2 for the standard error.
     </p>
     <p>
      The redirection specifications are:
     </p>
     <dl>
      <dt>
       <tt>
        &lt; filename
       </tt>
      </dt>
      <dd>
       Redirect standard input to read from the named file.
      </dd>
      <dt>
       <tt>
        &gt; filename
       </tt>
      </dt>
      <dd>
       Redirect standard output to write to the named file, overwriting its previous contents.
      </dd>
      <dt>
       <tt>
        &gt;&gt; filename
       </tt>
      </dt>
      <dd>
       Redirect standard output to write to the named file, appending to the end of its previous contents.
      </dd>
      <dt>
       <tt>
        &gt;&amp;
        <i>
         h
        </i>
       </tt>
      </dt>
      <dd>
       Redirect to handle
       <i>
        h
       </i>
       , where handle is any of 0standard input, 1standard output, 2standard error, and more.
      </dd>
      <dt>
       <tt>
        &lt;&amp;
        <i>
         h
        </i>
       </tt>
      </dt>
      <dd>
       Redirect from handle
       <i>
        h
       </i>
       .
      </dd>
     </dl>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       dir *.txt &gt;listing.log
       <ul>
        <li>
         Redirects the output of the dir command to listing.log file.
        </li>
       </ul>
      </li>
      <li>
       dir *.txt 2&gt;NUL
       <ul>
        <li>
         Redirects errors of the dir command to nowhere.
        </li>
       </ul>
      </li>
      <li>
       dir *.txt &gt;&gt;listing.log
       <ul>
        <li>
         Redirects the output of the dir command to listing.log file, appending to the file. Thereby, the content of the file before the redirected command was executed does not get lost.
        </li>
       </ul>
      </li>
      <li>
       dir *.txt &gt;listing.log 2&gt;&amp;1
       <ul>
        <li>
         Redirects the output of the dir command to listing.log file, along with the error messages.
        </li>
       </ul>
      </li>
      <li>
       dir *.txt &gt;listing.log 2&gt;listing-errors.log
       <ul>
        <li>
         Redirects the output of the dir command to listing.log file, and the error messages to listing-errors.log file.
        </li>
       </ul>
      </li>
      <li>
       &gt;myfile.txt echo Hello
       <ul>
        <li>
         The redirection can precede the command.
        </li>
       </ul>
      </li>
      <li>
       echo Hello &amp; echo World &gt;myfile.txt
       <ul>
        <li>
         Only the 2nd echo gets redirected.
        </li>
       </ul>
      </li>
      <li>
       (echo Hello &amp; echo World) &gt;myfile.txt
       <ul>
        <li>
         Output of both echos gets redirected.
        </li>
       </ul>
      </li>
      <li>
       type con &gt;myfile.txt
       <ul>
        <li>
         Redirects console input (con) to the file. Thus, allows multi-line user input terminated by user pressing Control + Z. See also
         <a href="#User_input">
          #User input
         </a>
         .
        </li>
       </ul>
      </li>
      <li>
       (for%i in (1,2,3) do @echo%i) &gt; myfile.txt
       <ul>
        <li>
         Redirects the entire output of the loop to the file.
        </li>
       </ul>
      </li>
      <li>
       for%i in (1,2,3) do @echo%i &gt; myfile.txt
       <ul>
        <li>
         Starts redirection anew each time the body of the loop is entered, losing the output of all but the latest loop iteration.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Links:
     </p>
     <ul>
      <li>
       <a class="external text" href="http://ss64.com/nt/syntax-redirection.html" rel="nofollow">
        Redirection at ss64.com
       </a>
      </li>
      <li>
       <a class="external text" href="http://www.microsoft.com/resources/documentation/windows/xp/all/proddocs/en-us/redirection.mspx?mfr=true" rel="nofollow">
        Using command redirection operators at Microsoft
       </a>
      </li>
     </ul>

       How a command is executed
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=9" title="Edit section: How a command is executed">

      (...)
     </p>
     <h3>
      <span class="mw-headline" id="Batch_reloading">
       Batch reloading
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=10" title="Edit section: Batch reloading">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      The command interpreter reloads the content of a batch after each execution of a line or a bracketed group.
     </p>
     <p>
      If you start the following batch and change "echo A" to "echo B" in the batch shortly after starting it, the output will be B.
     </p>
     <pre>
      @echo off
ping -n 6 127.0.0.1 &gt;nul &amp; REM wait
echo A
     </pre>
     <p>
      What is on a single line does matter; changing "echo A" in the following batch after running it has no impact:
     </p>
     <pre>
      @echo off
ping -n 6 127.0.0.1 &gt;nul &amp; echo A
     </pre>
     <p>
      Nor have after-start changes have any impact on commands bracketed with ( and ). Thus, changing "echo A" after starting the following batch has no impact:
     </p>
     <pre>
      @echo off
for /L %%i in (1,1,10) do (
  echo A
  ping -n 2 127.0.0.1 &gt;nul &amp; REM wait
)
     </pre>
     <p>
      Ditto for any other enclosing, including this one:
     </p>
     <pre>
      @echo off
(
ping -n 6 127.0.0.1 &gt;nul &amp; REM wait
echo A
)
     </pre>
     <h3>
      <span class="mw-headline" id="Environment_variables">
       Environment variables
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=11" title="Edit section: Environment variables">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      The environment variables of the command interpreter process are inherited by the processes of any (external) commands that it executes. A few environment variables are used by the command interpreter itself. Changing them changes its operation.
     </p>
     <p>
      Environment variables are affected by the
      <a href="#SET">
       SET
      </a>
      ,
      <a href="#PATH">
       PATH
      </a>
      , and
      <a href="#PROMPT">
       PROMPT
      </a>
      commands.
     </p>
     <p>
      To unset a variable, set it to empty string, such as "set myvar=".
     </p>
     <p>
      The command interpreter inherits its initial set of environment variables from the process that created it. In the case of command interpreters invoked from desktop shortcuts this will be Windows Explorer, for example.
     </p>
     <p>
      Command interpreters generally have textual user interfaces, not graphical ones, and so do not recognize the Windows message that informs applications that the environment variable template in the Registry has been changed. Changing the environment variables in Control Panel will cause Windows Explorer to update its own environment variables from the template in the Registry, and thus change the environment variables that any
      <i>
       subsequently invoked
      </i>
      command interpreters will inherit. However, it will not cause command interpreters that
      <i>
       are already running
      </i>
      to update their environment variables from the template in the Registry.
     </p>
     <h4>
      <span class="mw-headline" id="COMSPEC">
       COMSPEC
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=12" title="Edit section: COMSPEC">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h4>
     <p>
      The COMSPEC environment variable contains the full pathname of the command interpreter program file. This is just inherited from the parent process, and is thus indirectly derived from the setting of COMSPEC in the environment variable template in the Registry.
     </p>
     <h4>
      <span class="mw-headline" id="PATH">
       PATH
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=13" title="Edit section: PATH">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h4>
     <p>
      The value of the PATH environment variable comprises a list of directory names, separated by semi-colon characters. This is the list of directories that are searched, in order, when locating the program file of an external command to execute.
     </p>
     <h4>
      <span class="mw-headline" id="PATHEXT">
       PATHEXT
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=14" title="Edit section: PATHEXT">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h4>
     <p>
      The value of the PATHEXT environment variable comprises a list of filename extensions, separated by semi-colon characters. This is the list of filename extensions that are applied, in order, when locating the program file of an external command to execute.
     </p>
     <p>
      An example content of PATHEXT printed by "echo%PATHEXT%":
     </p>
     <ul>
      <li>
       .COM;.EXE;.BAT;.CMD;.VBS;.VBE;.JS;.JSE;.WSF;.WSH;.MSC
      </li>
     </ul>
     <p>
      By adding ".PL" to the variable, you can ensure Perl programs get run from the command line even when typed without the ".pl" extension. Thus, instead of typing "mydiff.pl a.txt b.txt", you can type "mydiff a.txt b.txt".
     </p>
     <p>
      Adding ".PL" to the variable in Windows Vista and later:
     </p>
     <ul>
      <li>
       setx PATHEXT%PATHEXT%;.PL
       <ul>
        <li>
         If you use "set" available in Windows XP, the effect will be temporary and impacting only the current console or process.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Links:
     </p>
     <ul>
      <li>
       <a class="external text" href="http://ss64.com/nt/syntax-variables.html" rel="nofollow">
        Windows Environment Variables at ss64
       </a>
      </li>
      <li>
       <a class="external text" href="http://stackoverflow.com/questions/9037346/making-python-scripts-run-on-windows-without-specifying-py-extension" rel="nofollow">
        Making Python scripts run on Windows without specifying .py extension at stackoverflow
       </a>
      </li>
     </ul>

     <h3>
      <span class="mw-headline" id="Switches">
       Switches
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=16" title="Edit section: Switches">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Most Windows commands provide switches AKA options to direct their behavior.
     </p>
     <p>
      Observations:
     </p>
     <ul>
      <li>
       Switches most often consist of a single-letter; some switches consist of a sequence of multiple letters.
      </li>
      <li>
       Switches are preceded with a slash (/) rather than, as in some other operating systems, with a minus sign (-).
      </li>
      <li>
       Switches are case-insensitive rather than, as in some other operating systems, case-sensitive.
      </li>
      <li>
       If a command from another operating system is ported to Windows (such as grep), it usually retains the option conventions from the original operating system, including the use of minus sign and case-sensitivity.
      </li>
     </ul>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       dir /?
       <ul>
        <li>
         Displays the help. This option is provided by many commands.
        </li>
       </ul>
      </li>
      <li>
       dir /b /s
       <ul>
        <li>
         Lists all files and folders in the current folder recursively. Two switches are used: b and s.
        </li>
       </ul>
      </li>
      <li>
       dir /bs
       <ul>
        <li>
         Does not work; switches cannot be accumulated behind a single slash.
        </li>
       </ul>
      </li>
      <li>
       findstr /ric:"id: *[0-9]*" File.txt
       <ul>
        <li>
         Unlike many other commands, findstr allows the accumulation of switches behind a single slash. Indeed, r, i and c are single-letter switches.
        </li>
       </ul>
      </li>
      <li>
       dir/b/s
       <ul>
        <li>
         Works. In dir, removing whitespace between the command and the first switch or between the switches does not make a difference; thus, does the same as dir /b /s.
        </li>
       </ul>
      </li>
      <li>
       tree/f/a
       <ul>
        <li>
         Does not work, unlike tree /f /a. In tree, separation by whitespace is mandatory. Nor does find/i/v work.
        </li>
       </ul>
      </li>
      <li>
       dir /od
       <ul>
        <li>
         The switch letter o is further modified by a single letter specifying that ordering should be by date. The letter d is not a switch by itself. Similar cases include dir /ad and more /t4.
        </li>
       </ul>
      </li>
      <li>
       dir /B /S
       <ul>
        <li>
         The switches are case-insensitive, unlike in some other operating systems.
        </li>
       </ul>
      </li>
      <li>
       sort /r file.txt
       <ul>
        <li>
         Sorts the file in a reverse order.
        </li>
       </ul>
      </li>
      <li>
       sort /reverse file.txt
       <ul>
        <li>
         Sort allows the switch string to be longer than a single-letter.
        </li>
       </ul>
      </li>
      <li>
       sort /reve file.txt
       <ul>
        <li>
         Sort allows the specified switch string to be a substring of the complete long name of the switch. Thus, does the same as the above.
        </li>
       </ul>
      </li>
      <li>
       sort /reva file.txt
       <ul>
        <li>
         Does not work, since "reva" is not a substring of "reverse".
        </li>
       </ul>
      </li>
      <li>
       taskkill /im AcroRd32.exe
       <ul>
        <li>
         Taskkill requires a multiletter switch name for /im; shortening to /i does not work.
        </li>
       </ul>
      </li>
      <li>
       java -version
       <ul>
        <li>
         Java, which originated in the environment of another operating system family, uses the minus convention for its switches AKA options.
        </li>
       </ul>
      </li>
      <li>
       grep --help
       <ul>
        <li>
         If GNU grep is installed, it requires multi-letter switches to be preceded by two dashes.
        </li>
       </ul>
      </li>
     </ul>
     <h3>
      <span class="mw-headline" id="Error_level">
       Error level
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=17" title="Edit section: Error level">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Commands usually set error level at the end of their execution. In Windows NT and later, it is a 32-bit signed integer; in MS DOS, it used to be an integer from 0 to 255. Keywords: return code, exit code, exit status.
     </p>
     <p>
      The conventional meaning of the error level:
     </p>
     <ul>
      <li>
       0 - success
      </li>
      <li>
       not 0 - failure
      </li>
      <li>
       The error levels being set are usually positive.
      </li>
      <li>
       If the command does not distinguish various kinds of failure, the error level on failure is usually 1.
      </li>
     </ul>
     <p>
      Uses of the error level:
     </p>
     <ul>
      <li>
       It can be tested using &amp;&amp; and ||; see also
       <a href="#Syntax">
        #Syntax
       </a>
       .
      </li>
      <li>
       It can be tested using
       <a href="#IF">
        IF
       </a>
       .
      </li>
      <li>
       The value can be accessed from ERRORLEVEL variable.
      </li>
     </ul>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       dir &gt;NUL &amp;&amp; echo Success
       <ul>
        <li>
         The part after &amp;&amp; is executed only if the error level is zero.
        </li>
       </ul>
      </li>
      <li>
       color 00 || echo Failure
       <ul>
        <li>
         The part after || is executed only if the error level is non-zero, whether positive or negative.
        </li>
       </ul>
      </li>
      <li>
       color 00 || (
       <br/>
       echo Failure
       <br/>
       )
       <ul>
        <li>
         Multiline bracketing works as well.
        </li>
       </ul>
      </li>
      <li>
       echo%ERRORLEVEL%
       <ul>
        <li>
         Displays the error level without changing it.
        </li>
       </ul>
      </li>
      <li>
       if%errorlevel% equ 0 echo The error level is zero, meaning success.
      </li>
      <li>
       if%errorlevel% neq 0 echo The error level is non-zero, meaning failure.
      </li>
      <li>
       if errorlevel 1 echo The error level is &gt;= 1, meaning failure via positive error level.
       <ul>
        <li>
         Does not cover failure via negative error level. Note the "&gt;=" part: this is not the same as if%errorlevel% equ 1.
        </li>
       </ul>
      </li>
      <li>
       exit /b 1
       <ul>
        <li>
         Returns a batch file, setting the error level to 1.
        </li>
       </ul>
      </li>
      <li>
       cmd /c "exit /b 10"
       <ul>
        <li>
         In the middle of a batch file or on the command line, sets the error level to 10.
        </li>
       </ul>
      </li>
      <li>
       (cmd /c "exit /b 0" &amp;&amp; Echo Success) &amp; (cmd /c "exit /b -1" || Echo Failure)
       <ul>
        <li>
         As above, showing the error level is indeed affected.
        </li>
       </ul>
      </li>
      <li>
       (cmd /c "exit /b 0" &amp; cmd /c "exit /b 1") || Echo Failure
       <ul>
        <li>
         The error level of a chain created by &amp; is the error level of the last command of the chain.
        </li>
       </ul>
      </li>
      <li>
       cmd /c "exit /b -1" &amp; if not errorlevel 1 echo Would-be success
       <ul>
        <li>
         The "if not errorlevel 1" test, which might appear to test for success, passes on negative numbers: it tests on "not error level &gt;= 1", which is "error level &lt;= 0".
        </li>
       </ul>
      </li>
      <li>
       set myerrorlevel=%errorlevel%
       <ul>
        <li>
         Remembers the error level for later.
        </li>
       </ul>
      </li>
      <li>
       set errorlevel=0
       <ul>
        <li>
         To be avoided: overshadows the built-in errorlevel variable. Ensures that subsequent accesses via%ERRORLEVEL% return 0 rather than the actual error level.
        </li>
       </ul>
      </li>
      <li>
       cmd /c "exit /b 0"
       <br/>
       if 1 equ 1 ( cmd /c "exit /b 1" &amp; echo%errorlevel% )
       <ul>
        <li>
         Displays 0, since%errorlevel% gets expanded before cmd /c "exit /b 1" gets executed.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Links:
     </p>
     <ul>
      <li>
       <a class="external text" href="http://ss64.com/nt/errorlevel.html" rel="nofollow">
        Error level at ss64
       </a>
      </li>
     </ul>

     </h3>
     <p>
      Getting a substring of a variable by position and length:
     </p>
     <p>
      Before running the following examples, ensure that%a% equals "abcd" by running this:
     </p>
     <ul>
      <li>
       set a=abcd
      </li>
     </ul>
     <p>
      The examples:
     </p>
     <ul>
      <li>

        </li>
       </ul>
      </li>

        <li>
         Result: b
        </li>
       </ul>
      </li>
      <li>
       echo%a:~0,2%
       <ul>
        <li>
         Result: ab
        </li>
       </ul>
      </li>
      <li>
       echo%a:~1,2%
       <ul>
        <li>
         Result: bc
        </li>
       </ul>
      </li>
      <li>
       echo%a:~1%
       <ul>
        <li>
         Result: bcd
        </li>
       </ul>
      </li>
      <li>
       echo%a:~-1%
       <ul>
        <li>
         Result: d
        </li>
       </ul>
      </li>
      <li>
       echo%a:~-2%
       <ul>
        <li>
         Result: cd
        </li>
       </ul>
      </li>
      <li>
       echo%a:~0,-2%
       <ul>
        <li>
         Result: ab
        </li>
       </ul>
      </li>
      <li>
       echo%a:~0,-1%
       <ul>
        <li>
         Result: abc
        </li>
       </ul>
      </li>
      <li>
       echo%a:~1,-1%
       <ul>
        <li>
         Result: bc
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Testing substring containment:
     </p>
     <ul>
      <li>
       if not "%a:bc=%"=="%a%" echo yes
       <ul>
        <li>
         If variable a contains "bc" as a substring, echo "yes".
        </li>
        <li>
         This test is a trick that uses string replacement, discussed below.
        </li>
        <li>
         This test does not work if the variable contains a quotation mark.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Testing for "starts with":
     </p>
     <ul>
      <li>
       if%a:~0,1%==a echo yes
       <ul>
        <li>
         If variable a starts with "a", echo "yes".
        </li>
       </ul>
      </li>
      <li>
       if%a:~0,2%==ab echo yes
       <ul>
        <li>
         If variable a starts with "ab", echo "yes".
        </li>
       </ul>
      </li>
     </ul>
     <p>
      String replacement:
     </p>
     <ul>
      <li>
       set a=abcd &amp; echo%a:c=%
       <ul>
        <li>
         Result: abd
        </li>
       </ul>
      </li>
      <li>
       set a=abcd &amp; echo%a:c=e%
       <ul>
        <li>
         Result: abed
        </li>
       </ul>
      </li>
      <li>
       set a=abcd &amp; echo%a:*c=%
       <ul>
        <li>
         Result: d
        </li>
        <li>
         The asterisk only works at the beginning of the sought pattern; it does not work at the end or in the middle.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      See also the help for
      <a href="#SET">
       SET
      </a>
      command: set /?.
     </p>
     <p>
      Splitting a string by any of " ", ",", and ";":
     </p>
     <dl>
      <dd>
       set myvar=a b,c;d
      </dd>
      <dd>
       for%%a in (%myvar%) do echo%%a
      </dd>
     </dl>
     <p>
      Splitting a string by semicolon, assuming the string contains no quotation marks:
     </p>
     <pre>
      @echo off
set myvar=a b;c;d
set strippedvar=%myvar%
:repeat
for /f "delims=;" %%a in ("%strippedvar%") do echo %%a
set prestrippedvar=%strippedvar%
set strippedvar=%strippedvar:*;=%
if not "%prestrippedvar:;=%"=="%prestrippedvar%" goto :repeat
     </pre>

     </p>
     <p>
      In passing arguments to a batch script, characters used for argument separation are the following ones:
     </p>
     <ul>
      <li>
       space
      </li>
      <li>
       comma
      </li>
      <li>
       semicolon
      </li>
      <li>
       equal sign
      </li>
      <li>
       tab character
      </li>

     </p>

       test.bat a b c d
      </li>
      <li>
       test.bat a,b,c,d
      </li>
      <li>
       test.bat a, b, c, d
      </li>
      <li>
       test.bat a;b;c;d
      </li>
      <li>
       test.bat a=b=c=d
      </li>
      <li>
       test.bat a b,c;,;=d
      </li>
     </ul>
     <p>
      Yes, even the line with "a b,c;,;=d" passes four arguments, since a sequence of separating characters is considered a single separator.
     </p>
     <p>
      To have a space, comma or semicolon in the argument value, you can pass the value enclosed in quotation marks. However, the quotation marks become part of the argument value. To get rid of the enclosing quotation marks when referring to the argument in the script, you can use%~&lt;number&gt; described in
      <a href="#Percent_tilde">
       #Percent tilde
      </a>
      .

     <h3>
      <span class="mw-headline" id="Wildcards">
       Wildcards
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=20" title="Edit section: Wildcards">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Many commands accept file name wildcards--characters that do not stand for themselves and enable matching of a group of filenames.
     </p>
     <p>
      Wildcards:
     </p>
     <ul>
      <li>
       * (asterisk): any sequence of characters
      </li>
      <li>
       ? (question mark): a single character other than a period (".") or, if part of a sequence of question marks at the end of a maximum period-free part of a file name, possibly zero number of characters; see examples for clarification
      </li>
     </ul>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       dir *.txt
       <ul>
        <li>
         Matches Myfile.txt, Plan.txt and any other file with the .txt extension.
        </li>
       </ul>
      </li>
      <li>
       dir *txt
       <ul>
        <li>
         The period does not need to be included. However, this will also match files named without the period convention, such as myfiletxt.
        </li>
       </ul>
      </li>
      <li>
       ren *.cxx *.cpp
       <ul>
        <li>
         Renames all files with .cxx extension to have .cpp extension.
        </li>
       </ul>
      </li>
      <li>
       dir a?b.txt
       <ul>
        <li>
         Matches files aab.txt, abb.txt, a0b.txt, etc.
        </li>
        <li>
         Does not match ab.txt, since a question mark followed by a character other than a question mark or period cannot match zero characters.
        </li>
        <li>
         Does not match a.b.txt, since a question mark cannot match a period.
        </li>
       </ul>
      </li>
      <li>
       dir???.txt
       <ul>
        <li>
         Matches .txt, a.txt, aa.txt, and aaa.txt, among others, since each question mark in the sequence followed by a period can match zero number of characters.
        </li>
       </ul>
      </li>
      <li>
       dir a???.b???.txt???
       <ul>
        <li>
         Matches a.b.txt, among others. While the last question mark sequence is not followed by a period, it is still a sequence at the end of a maximum period-free part of a file name.
        </li>
       </ul>
      </li>
      <li>
       dir????????.txt &amp; @REM eight question marks
       <ul>
        <li>
         Matches the same files as *.txt, since each file also has a short file name that has no more than 8 characters before .txt.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Quirk with short file names: the wildcard matching is performed both on long file names and the usually hidden short 8 chars + period + 3 chars file names. This can lead to bad surprises.
     </p>
     <p>
      Unlike shells of some other operating systems, the cmd.exe shell does not perform wildcard expansion (replacement of the pattern containing wildcards with the list of file names matching the pattern) on its own. It is the responsibility of each program to treat wildcards as such. This enables such things as "ren *.txt *.bat", since the ren command actually sees the * wildcard rather than a list of files matching the wildcard. Thus, "echo *.txt" does not display files in the current folder matching the pattern but rather literally displays "*.txt". Another consequence is that you can write "findstr a.*txt" without fearing that the "a.*txt" part gets replaced with the names of some files in the current folder. Furthermore, recursive "findstr /s pattern *.txt" is possible, while in some other operating systems, the "*.txt" part would get replaced with the file names found in the current folder, disregarding nested folders.
     </p>
     <p>
      Commands accepting wildcards include
      <a href="#ATTRIB">
       ATTRIB
      </a>
      ,
      <a href="#COPY">
       COPY
      </a>
      ,
      <a href="#DIR">
       DIR
      </a>
      ,
      <a href="#FINDSTR">
       FINDSTR
      </a>
      ,
      <a href="#FOR">
       FOR
      </a>
      ,
      <a href="#REN">
       REN
      </a>
      , etc.
     </p>
     <p>
      Links:
     </p>

      </span>
      <span class="mw-editsection">

      Functions AKA subprograms can be emulated using
      <a href="#CALL">

      , labels,
      <a href="#SETLOCAL">

     <h3>
      <span class="mw-headline" id="Calculation">
       Calculation
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=24" title="Edit section: Calculation">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Batch scripts can do simple 32-bit integer arithmetic and bitwise manipulation using
      <a href="#SET">
       SET
      </a>
      /a command. The largest supported integer is 2147483647 = 2 ^ 31 - 1. The smallest supported integer is -2147483648 = - (2 ^ 31), assignable with the trick of set /a num=-2147483647-1. The syntax is reminiscent of the C language.
     </p>
     <p>
      Arithmetic operators include *, /,% (modulo), +, -. In a batch, modulo has to be entered as "%%".
     </p>
     <p>
      Bitwise operators interpret the number as a sequence of 32 binary digits. These are ~ (complement), &amp; (and), | (or), ^ (xor), &lt;&lt; (left shift), &gt;&gt; (right shift).
     </p>
     <p>
      A logical operator of negation is!: it turns zero into one and non-zero into zero.
     </p>
     <p>
      A combination operator is ,: it allows more calculations in one set command.
     </p>
     <p>
      Combined assignment operators are modeled on "+=", which, in "a+=b", means "a=a+b". Thus, "a-=b" means "a=a-b". Similarly for *=, /=,%=, &amp;=, ^=, |=, &lt;&lt;=, and &gt;&gt;=.
     </p>
     <p>
      The precedence order of supported operators, each predecence level separated by , is as follows: ( )  * /% + -  &lt;&lt; &gt;&gt;  &amp;  ^  |  = *= /=%= += -= &amp;= ^= |= &lt;&lt;= &gt;&gt;=  ,
     </p>
     <p>
      Literals can be entered as decimal (1234), hexadecimal (0xffff, leading 0x), and octal (0777, leading 0).
     </p>
     <p>
      The internal bit representation of negative numbers is two's complement. This provides a connection between arithmetic operations and bit operations. For instance, -2147483648 is represented as 0x80000000, and therefore set /a num=~(-2147483647-1) yields 2147483647, which equals 0x7FFFFFFF (type set /a num=0x7FFFFFFF to check).
     </p>
     <p>
      As some of the operators have special meaning for the command interpreter, an expression using them needs to be enclosed in quotation marks, such as this:
     </p>
     <ul>
      <li>
       set /a num="255^127"
      </li>
      <li>
       set /a "num=255^127"
       <ul>
        <li>
         Alternative placement of quotation marks.
        </li>
       </ul>
      </li>
      <li>
       set /a num=255^^127
       <ul>
        <li>
         Escape ^ using ^ instead of quotation marks.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       set n1=40 &amp; set n2=25
       <p>
        set /a n3=%n1%+%n2%
       </p>
       <ul>
        <li>
         Uses the standard percent notation for variable expansion.
        </li>
       </ul>
      </li>
      <li>
       set n1=40 &amp; set n2=25
       <p>
        set /a n3=n1+n2
       </p>
       <ul>
        <li>
         Avoids the percent notation around variable names as unneeded for /a.
        </li>
       </ul>
      </li>
      <li>
       set /a num="255^127"
       <ul>
        <li>
         Encloses "^" in quotation marks to prevent its special meaning for the command interpreter.
        </li>
       </ul>
      </li>
      <li>
       set /a n1 = (10 + 5)/5
       <ul>
        <li>
         The spaces around = do not matter with /a. However, getting used to it lends itself to writing "set var = value" without /a, which sets the value of "var " rather than "var".
        </li>
       </ul>
      </li>
      <li>
       set /a n1=2+3,n2=4*7
       <ul>
        <li>
         Performs two calculations.
        </li>
       </ul>
      </li>
      <li>
       set /a n1=n2=2
       <ul>
        <li>
         Has the same effect as n1=2,n2=2.
        </li>
       </ul>
      </li>
      <li>
       set n1=40 &amp; set n2=25 &amp; set /a n3=n1+n2
       <ul>
        <li>
         Works as expected.
        </li>
       </ul>
      </li>
      <li>
       set /a n1=2,n2=3,n3=n1+n2
       <ul>
        <li>
         Works as expected.
        </li>
       </ul>
      </li>
      <li>
       set n1=40 &amp; set n2=25 &amp; set /a n3=%n1%+%n2%
       <ul>
        <li>
         Does not work unless n1 and n2 were set previously. The variable specifications "%n1%" and "%n2"% get expanded
         <i>
          before
         </i>
         the first set command is executed. Dropping percent notation makes it work.
        </li>
       </ul>
      </li>
      <li>
       set /a n1=2,n2=3,n3=%n1%+%n2%
       <ul>
        <li>
         Does not work unless n1 and n2 were set previously, for the reason stated in the previous example.
        </li>
       </ul>
      </li>
      <li>
       set /a n1=0xffff
       <ul>
        <li>
         Sets n1 using hexadecimal notation.
        </li>
       </ul>
      </li>
      <li>
       set /a n1=0777
       <ul>
        <li>
         Sets n1 using octal notation.
        </li>
       </ul>
      </li>
     </ul>

       </span>

       </span>

       </span>

     <h3>
      <span class="mw-headline" id="Limitations">
       Limitations
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=25" title="Edit section: Limitations">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      There is no touch command familiar from other operating systems. The touch command would modify the last-modification timestamp of a file without changing its content.
     </p>
     <p>
      One workaround, with unclear reliability and applicability across various Windows versions, is this:
     </p>
     <ul>
      <li>
       copy /b file.txt+,,
      </li>
     </ul>
     <p>
      Links:
     </p>

     <h3>
      <span class="mw-headline" id="ASSOC">
       ASSOC
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=28" title="Edit section: ASSOC">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Associates an extension with a file type (FTYPE), displays existing associations, or deletes an association. See also
      <a href="#FTYPE">
       FTYPE
      </a>
      .
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       assoc
       <ul>
        <li>
         Lists all associations, in the format "&lt;file extension&gt;=&lt;file type&gt;", as, for example, ".pl=Perl" or ".xls=Excel.Sheet.8".
        </li>
       </ul>
      </li>
      <li>
       assoc | find ".doc"
       <ul>
        <li>
         Lists all associations containing ".doc" substring.
        </li>
       </ul>
      </li>
     </ul>

     <h3>
      <span class="mw-headline" id="CD">
       CD
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=31" title="Edit section: CD">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Changes to a different directory, or displays the current directory. However, if a different drive letter is used, it does not switch to that different drive or volume.
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       cd
      </li>
      <li>
       cd C:\Program Files
      </li>
      <li>
       cd \Program Files
      </li>
      <li>
       cd Documents
      </li>
      <li>
       cd%USERPROFILE%
      </li>
      <li>
       cd /d C:\Program Files
       <ul>
        <li>
         Changes to the directory of the C: drive even if C: is not the current drive.
        </li>
       </ul>
      </li>
      <li>
       C: &amp; cd C:\Program Files.
       <ul>
        <li>
         Changes to the directory of the C: drive even if C: is not the current drive.
        </li>
       </ul>
      </li>
      <li>
       cd ..
       <ul>
        <li>
         Changes to the parent directory. Does nothing if already in the root directory.
        </li>
       </ul>
      </li>
      <li>
       cd ..\..
       <ul>
        <li>
         Changes to the parent directory two levels up.
        </li>
       </ul>
      </li>
      <li>
       C: &amp; cd C:\Windows\System32 &amp; cd ..\..\Program Files
       <ul>
        <li>
         Uses ".." to navigate through the directory three up and down
        </li>
       </ul>
      </li>
     </ul>
     <p>
      No surrounding quotes are needed around paths with spaces.
     </p>
     <p>
      Links:
     </p>
     <ul>
      <li>
       <a class="external text" href="http://ss64.com/nt/cd.html" rel="nofollow">

     <h3>
      <span class="mw-headline" id="COPY">
       COPY
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=35" title="Edit section: COPY">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Copies files. See also
      <a href="#MOVE">
       MOVE
      </a>
      .
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       copy F:\File.txt
       <ul>
        <li>
         Copies the file into the current directory, assuming the current directory is not F:\.
        </li>
       </ul>
      </li>
      <li>
       copy "F:\My File.txt"
       <ul>
        <li>
         As above; quotation marks are needed to surround a file with spaces.
        </li>
       </ul>
      </li>
      <li>
       copy F:\*.txt
       <ul>
        <li>
         Copies the files located at F:\ and ending in dot txt into the current directory, assuming the current directory is not F:\.
        </li>
       </ul>
      </li>
      <li>
       copy F:\*.txt .
       <ul>
        <li>
         Does the same as the above command.
        </li>
       </ul>
      </li>
      <li>
       copy File.txt
       <ul>
        <li>
         Issues an error message, as File.txt cannot be copied over itself.
        </li>
       </ul>
      </li>
      <li>
       copy File1.txt File2.txt
       <ul>
        <li>
         Copies File1.txt to File2.txt, overwriting File2.txt if confirmed by the user or if run from a batch script.
        </li>
       </ul>
      </li>
      <li>
       copy File.txt "My Directory"
       <ul>
        <li>
         Copies File.txt into "My Directory" directory, assuming "My Directory" exists.
        </li>
       </ul>
      </li>
      <li>
       copy Dir1 Dir2
       <ul>
        <li>
         Copies all files directly located in directory Dir1 into Dir2, assuming Dir1 and Dir2 are directories. Does not copy files located in nested directories of Dir1.
        </li>
       </ul>
      </li>
      <li>
       copy *.txt *.bak
       <ul>
        <li>
         For each *.txt file in the current folder, makes a copy ending with "bak" rather than "txt".
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Links:
     </p>
     <ul>
      <li>
       <a class="external text" href="http://ss64.com/nt/copy.html" rel="nofollow">
        copy at ss64.com
       </a>
      </li>
      <li>
       <a class="external text" href="http://www.microsoft.com/resources/documentation/windows/xp/all/proddocs/en-us/copy.mspx" rel="nofollow">
        copy at Microsoft
       </a>
      </li>

     <h3>
      <span class="mw-headline" id="DEL">
       DEL
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=36" title="Edit section: DEL">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Deletes files.
      <i>
       Use with caution, especially in combination with wildcards.
      </i>
      Only deletes files, not directories, for which see
      <a href="#RD">
       RD
      </a>
      . For more, type "del /?".
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       del File.txt
      </li>
      <li>
       del /s *.txt
       <ul>
        <li>
         Deletes the files recursively including nested directories, but keeps the directories; mercilessly deletes all matching files without asking for confirmation.
        </li>
       </ul>
      </li>
      <li>
       del /p /s *.txt
       <ul>
        <li>
         As above, but asks for confirmation before every single file.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Links:
     </p>
     <ul>
      <li>
       <a class="external text" href="http://ss64.com/nt/del.html" rel="nofollow">
        del at ss64.com
       </a>
      </li>
      <li>
       <a class="external text" href="http://www.microsoft.com/resources/documentation/windows/xp/all/proddocs/en-us/del.mspx?mfr=true" rel="nofollow">
        del at Microsoft
       </a>
      </li>
     </ul>

     </h3>
     <p>
      Lists the contents of a directory. Offers a range of options. Type "dir /?" for more help.
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       dir
       <ul>
        <li>
         Lists the files and folders in the current folder, excluding hidden files and system files; uses a different manner of listing if DIRCMD variable is non-empty and contains switches for dir.
        </li>
       </ul>
      </li>
      <li>
       dir D:
      </li>
      <li>
       dir /b C:\Users
      </li>
      <li>
       dir /s
       <ul>
        <li>
         Lists the contents of the directory and all subdirectories recursively.
        </li>
       </ul>
      </li>
      <li>
       dir /s /b
       <ul>
        <li>
         Lists the contents of the directory and all subdirectories recursively, one file per line, displaying complete path for each listed file or directory.
        </li>
       </ul>
      </li>
      <li>
       dir *.txt
       <ul>
        <li>
         Lists all files with .txt extension.
        </li>
       </ul>
      </li>
      <li>
       dir /a
       <ul>
        <li>
         Includes hidden files and system files in the listing.
        </li>
       </ul>
      </li>
      <li>
       dir /ah
       <ul>
        <li>
         Lists hidden files only.
        </li>
       </ul>
      </li>
      <li>
       dir /ad
       <ul>
        <li>
         Lists directories only. Other letters after /A include S, I, R, A and L.
        </li>
       </ul>
      </li>
      <li>
       dir /ahd
       <ul>
        <li>
         Lists hidden directories only.
        </li>
       </ul>
      </li>
      <li>
       dir /a-d
       <ul>
        <li>
         Lists files only, omitting directories.
        </li>
       </ul>
      </li>
      <li>
       dir /a-d-h
       <ul>
        <li>
         Lists non-hidden files only, omitting directories.
        </li>
       </ul>
      </li>
      <li>
       dir /od
       <ul>
        <li>
         Orders the files and folders by the date of last modification. Other letters after /O include N (by name), E (by extension), S (by size), and G (folders first)
        </li>
       </ul>
      </li>
      <li>
       dir /o-s
       <ul>
        <li>
         Orders the files by the size descending; the impact on folder order is unclear.
        </li>
       </ul>
      </li>
      <li>
       dir /-c /o-s /a-d
       <ul>
        <li>
         Lists files ordered by size descending, omitting the thousands separator via /-C, excluding folders.
        </li>
       </ul>
      </li>
      <li>
       dir /s /b /od
       <ul>
        <li>
         Lists the contents of the directory and all subdirectories recursively, ordering the files in each directory by the date of last modification. The ordering only happens per directory; the complete set of files so found is not ordered as a whole.
        </li>
       </ul>
      </li>
     </ul>

      Links:

        dir at ss64.com
       </a>

     <h3>
      <span class="mw-headline" id="FOR">
       FOR
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=44" title="Edit section: FOR">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Iterates over a series of values, executing a command.
     </p>
     <p>
      In the following examples,%i is to be used from the command line while%%i is to be used from a batch.
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       for%%i in (1,2,3) do echo%%i
       <ul>
        <li>
         In a batch, echoes 1, 2, and 3. In a batch, the command must use a double percent sign.
        </li>
        <li>
         The remaining examples are intended to be directly pasted into a command line, so they use a single percent sign and include "@" to prevent repetitive display.
        </li>
       </ul>
      </li>
      <li>
       for%i in (1,2,3) do @echo%i
       <ul>
        <li>
         From a command line, echoes 1, 2, and 3.
        </li>
        <li>
         The for command tries to interpret the items as file names and as patterns of file names containing wildcards.
        </li>
        <li>
         It does not complain if the items do not match existing file names, though.
        </li>
       </ul>
      </li>
      <li>
       for%i in (1,2,a*d*c*e*t) do @echo%i
       <ul>
        <li>
         Unless you happen to have a file matching the third pattern, echoes 1 and 2, discarding the third item.
        </li>
       </ul>
      </li>
      <li>
       for%i in (1 2,3;4) do @echo%i
       <ul>
        <li>
         Echoes 1, 2, 3, and 4. Yes, a mixture of item separators is used.
        </li>
       </ul>
      </li>
      <li>
       for%i in (*.txt) do @echo%i
       <ul>
        <li>
         Echoes file names of files located in the current folder and having the .txt extension.
        </li>
       </ul>
      </li>
      <li>
       for%i in ("C:\Windows\system32\*.exe") do @echo%i
       <ul>
        <li>
         Echoes file names matching the pattern.
        </li>
       </ul>
      </li>
      <li>
       for /r%i in (*.txt) do @echo%i
       <ul>
        <li>
         Echoes file names with full paths, of files having the extension .txt located anywhere in the current folder including nested folders.
        </li>
       </ul>
      </li>
      <li>
       for /d%i in (*) do @echo%i
       <ul>
        <li>
         Echoes the names of all folders in the current folder.
        </li>
       </ul>
      </li>
      <li>
       for /r /d%i in (*) do @echo%i
       <ul>
        <li>
         Echoes the names including full paths of all folders in the current folder, including nested folders.
        </li>
       </ul>
      </li>
      <li>
       for /r%i in (*) do @if%~zi geq 1000000 echo%~zi%i
       <ul>
        <li>
         For each file in the current folder and its subfolders that has the size greater than or equal to 1,000,000 bytes, outputs the file size in bytes and the full path of the file. For the syntax in%~zi, see
         <a href="#Percent_tilde">
          #Percent tilde
         </a>
         .
        </li>
       </ul>
      </li>
      <li>
       for /l%i in (1,1,10) do @echo%i
       <ul>
        <li>
         Echoes the numbers from 1 to 10.
        </li>
       </ul>
      </li>
      <li>
       for /f "tokens=*"%i in (list.txt) do @echo%i
       <ul>
        <li>
         For each line in a file, echoes the line.
        </li>
       </ul>
      </li>
      <li>
       for /f "tokens=*"%i in (list1.txt list2.txt) do @echo%i
       <ul>
        <li>
         For each line in the files, echoes the line.
        </li>
       </ul>
      </li>
      <li>
       for /f "tokens=*"%i in (*.txt) do @echo%i
       <ul>
        <li>
         Does nothing. Does not accept wildcards to match file names.
        </li>
       </ul>
      </li>
      <li>
       for /f "tokens=1-3 delims=:"%a in ("First:Second::Third") do @echo%c-%b-%a
       <ul>
        <li>
         Parses a string into tokens delimited by ":".
        </li>
        <li>
         The quotation marks indicate the string is not a file name.
        </li>
        <li>
         The second and third tokens are stored in%b and%c even though%b and%c are not expressly mentioned in the part of the command before "do".
        </li>
        <li>
         The two consecutive colons are treated as one separator;%c is not "" but rather "Third".
        </li>
        <li>
         Does some of the job of the cut command from other operating systems.
        </li>
       </ul>
      </li>
      <li>
       for /f "tokens=1-3* delims=:"%a in ("First:Second::Third:Fourth:Fifth") do @echo%c-%b-%a:%d
       <ul>
        <li>
         As above, just that the 4th and 5th items get captured in%d as "Fourth:Fifth", including the separator.
        </li>
       </ul>
      </li>
      <li>
       for /f "tokens=1-3* delims=:,"%a in ("First,Second,:Third:Fourth:Fifth") do @echo%c-%b-%a:%d
       <ul>
        <li>
         Multiple delimiters are possible.
        </li>
       </ul>
      </li>
      <li>
       for /f "tokens=1-3"%a in ("First Second Third,item") do @echo%c-%b-%a
       <ul>
        <li>
         The default delimiters are space and tab. Thus, they differ from the separators used to separate arguments passed to a batch.
        </li>
       </ul>
      </li>
      <li>
       for /f "tokens=*"%i in ('cd') do @echo%i
       <ul>
        <li>
         For each line of the result of a command, echoes the line.
        </li>
       </ul>
      </li>
      <li>
       for /f "tokens=*"%i in ('dir /b /a-d-h') do @echo%~nxai
       <ul>
        <li>
         For each non-hidden file in the current folder, displays the file attributes followed by the file name. In the string "%~nxai", uses the syntax described at
         <a href="#Percent_tilde">
          #Percent tilde
         </a>
         .
        </li>
       </ul>
      </li>
      <li>
       for /f "usebackq tokens=*"%i in (`dir /b /a-d-h`) do @echo%~nxai
       <ul>
        <li>
         As above, but using the backquote character (`) around the command to be executed.
        </li>
       </ul>
      </li>
      <li>
       for /f "tokens=*"%i in ('tasklist ^| sort ^&amp; echo End') do @echo%i
       <ul>
        <li>
         Pipes and ampersands in the command to be executed must be escaped using caret (^).
        </li>
       </ul>
      </li>
      <li>
       (for%i in (1,2,3) do @echo%i) &gt; anyoldtemp.txt
       <ul>
        <li>
         To redirect the entire result of a for loop, place the entire loop inside brackets before redirecting. Otherwise, the redirection will tie to the body of the loop, so each new iteration of the body of the loop will override the results of the previous iterations.
        </li>
       </ul>
      </li>
      <li>
       for%i in (1,2,3) do @echo%i &gt; anyoldtemp.txt
       <ul>
        <li>
         An example related to the one above. It shows the consequence of failing to put the loop inside brackets.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      <i>
       Continue:
      </i>
      To jump to the next iteration of the loop and thus emulate the continue statement known from many languages, you can use goto provided you put the loop body in a subroutine, as shown in the following:
     </p>
     <div class="mw-highlight">
      <pre>
       <span class="k">
        for
       </span>
       <span class="nv">
        %%i
       </span>
       in (a b c)
       <span class="k">
        do
       </span>
       <span class="k">
        call
       </span>
       <span class="nl">
        :for_body
       </span>
       <span class="nv">
        %%i
       </span>
       exit
       <span class="n">
        /b
       </span>
       <span class="nl">
        :for_body
       </span>
       <span class="k">
        echo
       </span>
       <span class="m">
        1
       </span>
       <span class="nv">
        %1
       </span>
       <span class="k">
        goto
       </span>
       <span class="nl">
        :cont
       </span>
       <span class="k">
        echo
       </span>
       <span class="m">
        2
       </span>
       <span class="nv">
        %1
       </span>
       <span class="nl">
        :cont
       </span>
       exit
       <span class="n">
        /b
       </span>
      </pre>
     </div>
     <p>
      If you use goto directly inside the for loop, the use of goto breaks the loop bookkeeping. The following fails:
     </p>
     <div class="mw-highlight">
      <pre>
       <span class="k">
        for
       </span>
       <span class="nv">
        %%i
       </span>
       in (a b c)
       <span class="k">
        do
       </span>
       (
       <span class="k">
        echo
       </span>
       <span class="m">
        1
       </span>
       <span class="nv">
        %%i
       </span>
       <span class="k">
        goto
       </span>
       <span class="nl">
        :cont
       </span>
       <span class="k">
        echo
       </span>
       <span class="m">
        2
       </span>
       <span class="nv">
        %%i
       </span>
       <span class="nl">
        :cont
       </span>
       <span class="k">
        echo
       </span>
       <span class="m">
        3
       </span>
       <span class="nv">
        %%i
       </span>
       )
      </pre>
     </div>
     <p>
      Links:
     </p>
     <ul>
      <li>
       <a class="external text" href="http://ss64.com/nt/for.html" rel="nofollow">
        for at ss64.com
       </a>

     <h3>
      <span class="mw-headline" id="FTYPE">
       FTYPE
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=45" title="Edit section: FTYPE">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Displays or sets the command to be executed for a file type. See also
      <a href="#ASSOC">
       ASSOC
      </a>
      .
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       ftype
       <ul>

         Lists all associations of commands to be executed with file types, as, for example, 'Perl="C:\Perl\bin\perl.exe" "%1"%*'

      </li>
      <li>
       ftype | find "Excel.Sheet"
       <ul>
        <li>
         Lists only associations whose display line contains "Excel.Sheet"

     <h3>
      <span class="mw-headline" id="IF">
       IF
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=47" title="Edit section: IF">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Conditionally executes a command. Documentation is available by entering IF /? to CMD prompt.
     </p>
     <p>
      Available elementary tests:
     </p>
     <ul>
      <li>
       exist &lt;filename&gt;
      </li>
      <li>
       &lt;string&gt;==&lt;string&gt;
      </li>
      <li>
       &lt;expression1&gt; equ &lt;expression2&gt; -- equals
      </li>
      <li>
       &lt;expression1&gt; neq &lt;expression2&gt; -- not equal
      </li>
      <li>
       &lt;expression1&gt; lss &lt;expression2&gt; -- less than
      </li>
      <li>
       &lt;expression1&gt; leq &lt;expression2&gt; -- less than or equal
      </li>
      <li>
       &lt;expression1&gt; gtr &lt;expression2&gt; -- greater then
      </li>
      <li>
       &lt;expression1&gt; geq &lt;expression2&gt; -- greater than or equal
      </li>
      <li>
       defined &lt;variable&gt;
      </li>
      <li>
       errorlevel &lt;number&gt;
      </li>
      <li>
       cmdextversion &lt;number&gt;
      </li>
     </ul>
     <p>
      To each elementary test, "not" can be applied. Apparently there are no operators like AND, OR, etc. to combine elementary tests.
     </p>
     <p>
      The /I switch makes the == and equ comparisons ignore case.
     </p>
     <p>
      An example:
     </p>
     <div class="mw-highlight">
      <pre>
       <span class="k">
        if
       </span>
       not
       <span class="k">
        exist
       </span>
       <span class="nv">
        %targetpath%
       </span>
       (
       <span class="k">
        echo
       </span>
       Target path not found.
  exit
       <span class="n">
        /b
       </span>
       )
      </pre>
     </div>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       if not 1 equ 0 echo Not equal
      </li>
      <li>
       if 1 equ 0 echo A &amp; echo B
       <ul>
        <li>
         Does nothing; both echo commands are subject to the condition.
        </li>
       </ul>
      </li>
      <li>
       if not 1 equ 0 goto:mylabel
      </li>
      <li>
       if not a geq b echo Not greater
      </li>
      <li>
       if b geq a echo Greater
      </li>
      <li>
       if b geq A echo Greater in a case-insensitive comparison
      </li>
      <li>
       if B geq a echo Greater in a case-insensitive comparison
      </li>
      <li>
       if 0 equ 00 echo Numerical equality
      </li>
      <li>
       if not 0==00 echo String inequality
      </li>
      <li>
       if 01 geq 1 echo Numerical comparison
      </li>
      <li>
       if not "01" geq "1" echo String comparison
      </li>
      <li>
       if 1 equ 0 (echo Equal) else echo Unequal
       <ul>
        <li>
         Notice the brackets around the positive then-part to make it work.
        </li>
       </ul>
      </li>
      <li>
       if not a==A echo Case-sensitive inequality
      </li>
      <li>
       if /i a==A echo Case-insensitive equality
      </li>
      <li>
       if /i==/i echo This does not work
      </li>
      <li>
       if "/i"=="/i" echo Equal, using quotation marks to prevent the literal meaning of /i
      </li>
     </ul>
     <p>
      Links:
     </p>

       <a class="external text" href="http://ss64.com/nt/if.html" rel="nofollow">
        if at ss64.com
       </a>
      </li>
      <li>
       <a class="external text" href="https://www.microsoft.com/resources/documentation/windows/xp/all/proddocs/en-us/if.mspx" rel="nofollow">
        if at Microsoft
       </a>
      </li>

     </p>
     <ul>
      <li>

        </li>
       </ul>
      </li>

       move File.txt Dir
       <ul>

        </li>
       </ul>
      </li>
      <li>
       move F:\File.txt
       <ul>
        <li>
         Moves the file to the current directory.
        </li>
       </ul>
      </li>
      <li>
       move F:\*.txt
       <ul>
        <li>

     <h3>
      <span class="mw-headline" id="SET">
       SET
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=61" title="Edit section: SET">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Displays or sets environment variables. With /P switch, it asks the user for input, storing the result in the variable. With /A switch, it performs simple arithmetic calculations, storing the result in the variable. With string assignments, there must be no spaces before and after the equality sign; thus, "set name = Peter" does not work, while "set name=Peter" does.
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       set
       <ul>
        <li>
         Displays a list of environment variables
        </li>
       </ul>
      </li>
      <li>
       set HOME
       <ul>
        <li>
         Displays the values of the environment variables whose names start with "HOME"
        </li>
       </ul>
      </li>
      <li>
       set MYNUMBER=56
      </li>
      <li>
       set HOME=%HOME%;C:\Program Files\My Bin Folder
      </li>
      <li>
       set /P user_input=Enter an integer:
      </li>
      <li>
       set /A result = 4 * ( 6 / 3 )
       <ul>
        <li>
         Sets the result variable with the result of a calculation. See also
         <a href="#Calculation">
          #Calculation
         </a>
         .
        </li>
       </ul>
      </li>
     </ul>

     <h3>
      <span class="mw-headline" id="SETLOCAL">
       SETLOCAL
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=62" title="Edit section: SETLOCAL">

       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      When used in a batch file, makes all further changes to environment variables local to the current batch file. When used outside of a batch file, does nothing. Can be ended using
      <a href="#ENDLOCAL">
       ENDLOCAL
      </a>
      . Exiting a batch file automatically calls "end local". Can be used to create subprograms: see
      <a href="#Functions">
       Functions
      </a>
      .
     </p>
     <p>
      Furthermore, can be used to
      <i>
       enable delayed expansion
      </i>
      like this: "setlocal EnableDelayedExpansion". Delayed expansion consists in the names of variables enclosed in exclamation marks being replaced with their values only after the execution reaches the location of their use rather than at an earlier point.
     </p>
     <p>

     </p>
     <div class="mw-highlight">
      <pre>
       <span class="p">
        @
       </span>
       <span class="k">
        echo
       </span>
       <span class="k">
        off
       </span>
       <span class="k">
        call
       </span>
       <span class="nl">
        :myhead
       </span>
       <span class="m">
        2
       </span>
       File.txt
exit
       <span class="n">
        /b
       </span>
       <span class="c">
        :: Function myhead
       </span>
       <span class="c">
        :: ===============
       </span>
       <span class="c">
        :: %1 - lines count, %2 - file name
       </span>
       <span class="nl">
        :myhead
       </span>
       <span class="k">
        setlocal
       </span>
       EnableDelayedExpansion
       <span class="k">
        set
       </span>
       <span class="nv">
        counter
       </span>
       <span class="o">
        =
       </span>
       <span class="m">
        1
       </span>
       <span class="k">
        for
       </span>
       <span class="n">
        /f
       </span>
       <span class="s2">
        "tokens=*"
       </span>
       <span class="nv">
        %%i
       </span>
       in (
       <span class="nv">
        %2
       </span>
       )
       <span class="k">
        do
       </span>
       (
       <span class="k">
        echo
       </span>
       <span class="nv">
        %%i
       </span>
       <span class="k">
        set
       </span>
       <span class="n">
        /a
       </span>
       counter
       <span class="o">
        =
       </span>
       !counter!+
       <span class="m">
        1
       </span>
       <span class="k">
        if
       </span>
       !counter!
       <span class="o">
        gtr
       </span>
       <span class="nv">
        %1
       </span>
       exit
       <span class="n">
        /b
       </span>
       )
exit
       <span class="n">
        /b
       </span>
      </pre>

     <h3>
      <span class="mw-headline" id="START">
       START
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=64" title="Edit section: START">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Starts a program in new window, or opens a document. Uses an unclear algorithm to determine whether the first passed argument is a window title or a program to be executed; hypothesis: it uses the presence of quotes around the first argument as a hint that it is a window title.
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       start notepad.exe &amp; echo "Done."
       <ul>
        <li>
         Starts notepad.exe, proceeding to the next command without waiting for finishing the started one. Keywords: asynchronous.
        </li>
       </ul>
      </li>
      <li>
       start "notepad.exe"
       <ul>
        <li>
         Launches a new console window with notepad.exe being its title, apparently an undesired outcome.
        </li>
       </ul>
      </li>
      <li>
       start "" "C:\Program Files\Internet Explorer\iexplore.exe"
       <ul>
        <li>
         Starts Internet Explorer. The empty "" passed as the first argument is the window title of a console that actually does not get opened, or at least not visibly so.
        </li>
       </ul>
      </li>
      <li>
       start "C:\Program Files\Internet Explorer\iexplore.exe"
       <ul>
        <li>
         Launches a new console window with "C:\Program Files\Internet Explorer\iexplore.exe" being its title, apparently an undesired outcome.
        </li>
       </ul>
      </li>
      <li>
       start /wait notepad.exe &amp; echo "Done."
       <ul>
        <li>
         Starts notepad.exe, waiting for it to end before proceeding.
        </li>
       </ul>
      </li>
      <li>
       start /low notepad.exe &amp; echo "Done."
       <ul>
        <li>
         As above, but starting the program with a low priority.
        </li>
       </ul>
      </li>
      <li>
       start "" MyFile.xls
       <ul>
        <li>
         Opens the document in the program assigned to open it.
        </li>
       </ul>
      </li>
      <li>
       start
       <ul>
        <li>
         Starts a new console (command-line window) in the same current folder.
        </li>
       </ul>
      </li>
      <li>
       start .
       <ul>
        <li>
         Opens the current folder in Windows Explorer.
        </li>
       </ul>
      </li>
      <li>
       start ..
       <ul>
        <li>
         Opens the parent folder in Windows Explorer.
        </li>
       </ul>
      </li>
      <li>
       start "" "mailto:"
       <ul>
        <li>
         Starts the application for writing a new email.
        </li>
       </ul>
      </li>
      <li>
       start /b TODO:example-application-where-this-is-useful
       <ul>
        <li>
         Starts the application without opening a new console window, redirecting the output to the console from which the start command was called.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Links:
     </p>

       <a class="external text" href="http://ss64.com/nt/start.html" rel="nofollow">
        start at ss64.com
       </a>

      <li>

     <p>
      External commands available to Windows command interpreter are separate executable program files, supplied with the operating system by Microsoft, or bundled as standard with the third-party command interpreters.      <h3>
      <span class="mw-headline" id="ATTRIB">
       ATTRIB
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=73" title="Edit section: ATTRIB">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Displays or sets file attributes. With no arguments, it displays the attributes of all files in the current directory. With no attribute modification instructions, it displays the attributes of the files and directories that match the given search wildcard specifications. Similar to
      <a href="/wiki/Guide_to_Unix/Commands/File_System_Utilities#chmod" title="Guide to Unix/Commands/File System Utilities">
       chmod
      </a>
      of other operating systems.
     </p>
     <p>
      Modification instructions:
     </p>
     <ul>
      <li>
       To add an attribute, attach a '+' in front of its letter.
      </li>
      <li>
       To remove an attribute, attach a '-' in front of its letter
      </li>
      <li>
       Attributes:
       <ul>
        <li>
         A - Archived
        </li>
        <li>
         H - Hidden
        </li>
        <li>
         S - System
        </li>
        <li>
         R - Read-only
        </li>
        <li>
         ...and possibly others.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       attrib
       <ul>
        <li>
         Displays the attributes of all files in the current directory.
        </li>
       </ul>
      </li>
      <li>
       attrib File.txt
       <ul>
        <li>
         Displays the attributes of the file.
        </li>
       </ul>
      </li>
      <li>
       attrib +r File.txt
       <ul>
        <li>
         Adds the "Read-only" attribute to the file.
        </li>
       </ul>
      </li>
      <li>
       attrib -a File.txt
       <ul>
        <li>
         Removes the "Archived" attribute from the file.
        </li>
       </ul>
      </li>
      <li>
       attrib -a +r File.txt
       <ul>
        <li>
         Removes the "Archived" attribute and adds the "Read-only" attribute to the file.
        </li>
       </ul>
      </li>
      <li>
       attrib +r *.txt
       <ul>
        <li>
         Acts on a set of files.
        </li>
       </ul>
      </li>
      <li>
       attrib /S +r *.txt
       <ul>
        <li>
         Acts recursively in subdirectories.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      For more, type "attrib /?".
     </p>
     <p>
      Links:
     </p>
     <ul>
      <li>
       <a class="external text" href="http://ss64.com/nt/attrib.html" rel="nofollow">
        attrib at ss64.com
       </a>
      </li>
      <li>

       </span>
      </span>

     <p>
      Lets the user choose one of multiple options by pressing a single key, and sets the error level as per the chosen option. Absent in Windows 2000 and Windows XP, it was reintroduced in Windows Vista, and has remained in Windows 7 and 8.
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       choice /m "Do you agree"
       <ul>
        <li>
         Presents the user with a yes/no question, setting the error level to 1 for yes and to 2 for no. If the user presses Control + C, the error level is 0.
        </li>
       </ul>
      </li>
      <li>
       choice /c rgb /m "Which color do you prefer"
       <ul>
        <li>
         Presents the user with a question, and indicates the letters for the user. Responds to user pressing r, g or b, setting the error level to 1, 2 or 3.
        </li>
       </ul>
      </li>
     </ul>

     <h3>
      <span class="mw-headline" id="DOSKEY">
       DOSKEY
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=89" title="Edit section: DOSKEY">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Above all, creates macros known from other operating systems as aliases. Moreover, provides functions related to command history, and enhanced command-line editing. Macros are an alternative to very short batch scripts.
     </p>
     <p>
      Macro-related examples:
     </p>
     <ul>
      <li>
       doskey da=dir /s /b
       <ul>
        <li>
         Creates a single macro called "da"
        </li>
       </ul>
      </li>
      <li>
       doskey np=notepad $1
       <ul>
        <li>
         Creates a single macro that passes its first argument to notepad.
        </li>
       </ul>
      </li>
      <li>
       doskey /macrofile=doskeymacros.txt
       <ul>
        <li>
         Loads macro definitions from a file.
        </li>
       </ul>
      </li>
      <li>
       doskey /macros
       <ul>
        <li>
         Lists all defined macros with their definitions.
        </li>
       </ul>
      </li>
      <li>
       doskey /macros | find "da"
       <ul>
        <li>
         Lists all macro definitions that contain "da" as a substring; see also
         <a href="#FIND">
          FIND
         </a>
         .
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Command history-related examples:
     </p>
     <ul>
      <li>
       doskey /history
       <ul>
        <li>
         Lists the complete command history.
        </li>
       </ul>
      </li>
      <li>
       doskey /history | find "dir"
       <ul>
        <li>
         Lists each line of command history that contains "dir" as a substring
        </li>
       </ul>
      </li>
      <li>
       doskey /listsize=100
       <ul>
        <li>
         Sets the size of command history to 100.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      To get help on doskey from command line, type "doskey /?".
     </p>
     <p>
      Links:
     </p>
     <ul>
      <li>
       <a class="external text" href="http://ss64.com/nt/doskey.html" rel="nofollow">
        doskey at ss64.com
       </a>
      </li>
      <li>
       <a class="external text" href="http://www.microsoft.com/resources/documentation/windows/xp/all/proddocs/en-us/doskey.mspx" rel="nofollow">

     <p>
      Searches for a string in files or input, outputting matching lines. Unlike
      <a href="#FINDSTR">
       FINDSTR
      </a>
      , it cannot search folders recursively, cannot search for a regular expression, requires quotation marks around the sought string, and treats space literally rather than as a logical or.
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       find "(object" *.txt
      </li>
      <li>
       dir /S /B | find "receipt"
      </li>
      <li>
       dir /S /B | find /I /V "receipt"
       <ul>
        <li>
         Prints all non-matching lines in the output of the dir command, ignoring letter case.
        </li>
       </ul>
      </li>
      <li>
       find /C "inlined" *.h
       <ul>
        <li>
         Instead of outputting the matching lines, outputs their count. If more than one file is searched, outputs one count number per file preceded with a series of dashes followed by the file name; does not output the total number of matching lines in all files.
        </li>
       </ul>
      </li>
      <li>
       find /C /V "" &lt; file.txt
       <ul>
        <li>
         Outputs the number of lines AKA line count in "file.txt". Does the job of "wc -l" of other operating systems. Works by treating "" as a string not found on the lines. The use of redirection prevents the file name from being output before the number of lines.
        </li>
       </ul>
      </li>
      <li>
       type file.txt | find /C /V ""
       <ul>
        <li>
         Like the above, with a different syntax.
        </li>
       </ul>
      </li>
      <li>
       type *.txt 2&gt;NUL | find /C /V ""
       <ul>
        <li>
         Outputs the sum of line counts of the files ending in ".txt" in the current folder. The "2&gt;NUL" is a redirection of standard error that removes the names of files followed by empty lines from the output.
        </li>
       </ul>
      </li>
      <li>
       find "Schnheit" *.txt
       <ul>
        <li>
         If run from a batch file saved in unicode UTF-8 encoding, searches for the search term "Schnheit" in UTF-8 encoded *.txt files. For this to work, the batch file must not contain the byte order mark written by Notepad when saving in UTF-8. Notepad++ is an example of a program that lets you write UTF-8 encoded plain text files without byte order mark. While this works with find command, it does not work with
         <a href="#FINDSTR">
          #FINDSTR
         </a>
         .
        </li>
       </ul>
      </li>
      <li>
       find "Copyright" C:\Windows\system32\a*.exe
       <ul>
        <li>
         Works with binary files no less than text files.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Links:
     </p>
     <ul>
      <li>
       <a class="external text" href="http://ss64.com/nt/find.html" rel="nofollow">
        find at ss64.com
       </a>
      </li>
      <li>
       <a class="external text" href="http://www.microsoft.com/resources/documentation/windows/xp/all/proddocs/en-us/find.mspx" rel="nofollow">
        find at Microsoft
       </a>
      </li>
     </ul>
     <h3>
      <span class="mw-headline" id="FINDSTR">
       FINDSTR
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=93" title="Edit section: FINDSTR">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Searches for regular expressions or text strings in files. Does some of the job of "grep" command known from other operating systems, but is much more limited in the regular expressions it supports.
     </p>
     <p>
      Treats space in a regular expression as a disjunction AKA logical or unless prevented with /c option.
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       findstr /s "[0-9][0-9].*[0-9][0-9]" *.h *.cpp
       <ul>
        <li>
         Searches recursively all files whose name ends with dot h or dot cpp, priting only lines that contain two consecutive decimal digits followed by anything followed by two consecutive decimal digits.
        </li>
       </ul>
      </li>
      <li>
       findstr "a.*b a.*c" File.txt
       <ul>
        <li>
         Outputs all lines in File.txt that match any of the
         <i>
          two
         </i>
         regular expressions separated by the space. Thus, the effect is one of
         <i>
          logical or
         </i>
         on regular expressions.
        </li>
       </ul>
      </li>
      <li>
       echo world | findstr "hello wo.ld"
       <ul>
        <li>
         Does not match. Since the 1st item before the space does not look like a regex, findstr treats the whole search term as a plain search term.
        </li>
       </ul>
      </li>
      <li>
       echo world | findstr /r "hello wo.ld"
       <ul>
        <li>
         Matches. The use of /r forces regex treatment.
        </li>
       </ul>
      </li>
      <li>
       findstr /r /c:"ID: *[0-9]*" File.txt
       <ul>
        <li>
         Outputs all lines in File.txt that match the single regular expression containing a space. The use of /c prevents the space from being treated as a logical or. The use of /r switches the regular expression treatment on, which was disabled by default by the use of /c. To test this, try the following:
         <ul>
          <li>
           echo ID: 12|findstr /r /c:"ID: *[0-9]*$"
           <ul>
            <li>
             Matches.
            </li>
           </ul>
          </li>
          <li>
           echo ID: 12|findstr /c:"ID: *[0-9]*$"
           <ul>
            <li>
             Does not match, as the search string is not interpreted as a regular expression.
            </li>
           </ul>
          </li>
          <li>
           echo ID: abc|findstr "ID: *[0-9]*$"
           <ul>
            <li>
             Matches despite the output of echo failing to match the complete regular expression: the search is interpreted as one for lines matching "ID:"
             <i>
              or
             </i>
             "*[0-9]*$".
            </li>
           </ul>
          </li>
         </ul>
        </li>
       </ul>
      </li>
      <li>
       findstr /ric:"id: *[0-9]*" File.txt
       <ul>
        <li>
         Does the same as the previous example, but in a case-insensitive manner.
        </li>
        <li>
         While findstr enables this sort of accumulation of switches behind a single "/", this is not possible with any command. For instance, "dir /bs" does not work, while "dir /b /s" does.
        </li>
        <li>
         To test this, try the following:
         <ul>
          <li>
           echo ID: 12|findstr /ric:"id: *[0-9]*$"
          </li>
          <li>
           echo ID: ab|findstr /ric:"id: *[0-9]*$"
          </li>
         </ul>
        </li>
       </ul>
      </li>
      <li>
       findstr /msric:"id: *[0-9]*" *.txt
       <ul>
        <li>
         Like above, but recursively for all files per /s, displaying only matching files rather than matching lines per /m.
        </li>
       </ul>
      </li>
      <li>
       echo hel lo | findstr /c:"hel lo" /c:world
       <ul>
        <li>
         /c switch can be used multiple times to create logical or.
        </li>
       </ul>
      </li>
      <li>
       echo \hello\ | findstr "\hello\"
       <ul>
        <li>
         Does not match. Backslash before quotation marks and multiple other characters acts as an escape; thus, \" matches ".
        </li>
       </ul>
      </li>
      <li>
       echo \hello\ | findstr "\\hello\\"
       <ul>
        <li>
         Matches. Double backslash passed to findstr stands for a single backslash.
        </li>
       </ul>
      </li>
      <li>
       echo \hello\ | findstr \hello\
       <ul>
        <li>
         Matches. None of the single backslashes passed to findstr is followed by a character on which the backslash acts as an escape.
        </li>
       </ul>
      </li>
      <li>
       echo ^"hey | findstr \^"hey | more
       <ul>
        <li>
         To search for a quote (quotation mark), you need to escape it two times: once for the shell using caret (^), and once for findstr using backslash (\).
        </li>
       </ul>
      </li>
      <li>
       echo ^"hey | findstr ^"\^"hey there^" | more
       <ul>
        <li>
         To search for a quote and have the search term enclosed in quotes as well, the enclosing quotes need to be escaped for the shell using caret (^).
        </li>
       </ul>
      </li>
      <li>
       echo //comment line | findstr \//
       <ul>
        <li>
         If forward slash (/) is the 1st character in the search term, it needs to be escaped with a backslash (\). The escaping is needed even if the search term is enclosed in quotes.
        </li>
       </ul>
      </li>
      <li>
       findstr /xlg:File1.txt File2.txt
       <ul>
        <li>
         Outputs set intersection: lines present in both files.
        </li>
       </ul>
      </li>
      <li>
       findstr /xlvg:File2.txt File1.txt
       <ul>
        <li>
         Outputs set difference: File1.txt - File2.txt.
        </li>
       </ul>
      </li>
      <li>
       findstr /m Microsoft C:\Windows\system32\*.com
       <ul>
        <li>
         Works with binary files no less than text files.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Limitations of the regular expressions of "findstr", as compared to "grep":
     </p>
     <ul>
      <li>
       No support of groups -- "\(", "\)".
      </li>
      <li>
       No support of greedy iterators -- "*?".
      </li>
      <li>
       No support of "zero or one of the previous" -- "?".
      </li>
      <li>
       And more.
      </li>
     </ul>
     <p>
      Other limitations: There is a variety of limitations and strange behaviors as documented at
      <a class="external text" href="http://stackoverflow.com/questions/8844868/what-are-the-undocumented-features-and-limitations-of-the-windows-findstr-comman" rel="nofollow">
       What are the undocumented features and limitations of the Windows FINDSTR command?
      </a>
      .
     </p>
     <p>
      Also consider typing "findstr /?".
     </p>
     <p>
      Links:
     </p>
     <ul>
      <li>
       <a class="external text" href="http://ss64.com/nt/findstr.html" rel="nofollow">
        findstr at ss64.com
       </a>
      </li>
      <li>
       <a class="external text" href="http://www.microsoft.com/resources/documentation/windows/xp/all/proddocs/en-us/findstr.mspx" rel="nofollow">
        findstr at Microsoft
       </a>
      </li>
      <li>
       <a class="external text" href="http://stackoverflow.com/questions/8844868/what-are-the-undocumented-features-and-limitations-of-the-windows-findstr-comman" rel="nofollow">
        What are the undocumented features and limitations of the Windows FINDSTR command?
       </a>
       at StackOverflow

     <h3>
      <span class="mw-headline" id="FORFILES">
       FORFILES
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=94" title="Edit section: FORFILES">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Finds files by their modification date and file name pattern, and executes a command for each found file. Is very limited, especially compared to the find command of other operating systems. Available since Windows Vista. For more, type "forfiles /?".
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       forfiles /s /d 06/10/2015 /c "cmd /c echo @fdate @path"
       <ul>
        <li>
         For each file in the current folder and its subfolders modified on 10 June 2015 or later, outputs the file modification date and full file path. The date format after /d is locale specific. Thus, allows to find most recently modified files. Keywords: most recently changed files.
        </li>
       </ul>
      </li>
      <li>
       forfiles /m *.txt /s /d 06/10/2015 /c "cmd /c echo @fdate @path"
       <ul>
        <li>
         As above, but only for files ending in .txt.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Links:
     </p>
     <ul>
      <li>
       <a class="external text" href="http://ss64.com/nt/forfiles.html" rel="nofollow">

       </a>
      </li>
      <li>
       <a class="external text" href="https://technet.microsoft.com/en-us/library/cc753551.aspx" rel="nofollow">

     <h3>
      <span class="mw-headline" id="MORE">
       MORE
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=104" title="Edit section: MORE">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Displays the contents of a file or files, one screen at a time. When redirected to a file, performs some conversions, also depending on the used switches.
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       more Test.txt
      </li>
      <li>
       more *.txt
      </li>
      <li>
       grep -i sought.*string Source.txt | more /p &gt;Out.txt
       <ul>
        <li>
         Taking the output of a non-Windows grep command that produces line breaks consisting solely of LF character without CR character, converts LF line breaks to CR-LF line breaks. CR-LF newlines are also known as DOS line breaks, Windows line breaks, DOS newlines, Windows newlines, and CR/LF line endings,as opposed to LF line breaks used by some other operating systems.
        </li>
        <li>
         In some setups, seems to output gibberish if the input contains LF line breaks and tab characters at the same time.
        </li>
        <li>
         In some setups, for the conversion, /p may be unneeded. Thus, "more" would convert the line breaks even without /p.
        </li>
       </ul>
      </li>
      <li>
       more /t4 Source.txt &gt;Target.txt
       <ul>
        <li>
         Converts tab characters to 4 spaces.
        </li>
        <li>
         In some setups, tab conversion takes place automatically, even without the /t switch. If so, it is per default to 8 spaces.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Switch /e:
     </p>
     <ul>
      <li>
       The online documentation for "more" in Windows XP and Windows Vista does not mention the switch.
      </li>
      <li>
       The switch /e is mentioned in "more /?" at least in Windows XP and Windows Vista.
      </li>
      <li>
       Per "more /?", the switch is supposed to enable extended features listed at the end of "more /?" help such as showing the current row on pressing "=". However, in Windows XP and Windows Vista, that seems to be enabled by default even without /e.
      </li>
      <li>
       Hypothesis: In Windows XP and Windows Vista, /e does not do anything; it is present for compatibility reasons.
      </li>
     </ul>

      Links:

       TASKLIST
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=119" title="Edit section: TASKLIST">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Lists tasks, including task name and process id (PID).
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       tasklist | sort
      </li>
      <li>
       tasklist | find "AcroRd"
      </li>
      <li>
       tasklist | find /C "chrome.exe"
       <ul>
        <li>
         Displays the number of tasks named "chrome.exe", belonging to Google Chrome browser.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Links:
     </p>
     <ul>
      <li>
       <a class="external text" href="http://ss64.com/nt/tasklist.html" rel="nofollow">
        tasklist at ss64.com
       </a>
      </li>
      <li>
       <a class="external text" href="http://www.microsoft.com/resources/documentation/windows/xp/all/proddocs/en-us/tasklist.mspx" rel="nofollow">
        tasklist at Microsoft
       </a>
      </li>
     </ul>
     <h3>
      <span class="mw-headline" id="TIMEOUT">
       TIMEOUT
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=120" title="Edit section: TIMEOUT">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Waits a specified number of seconds, displaying the number of remaining seconds as time passes, allowing the user to interrupt the waiting by pressing a key. Also known as delay or sleep. Available in Windows Vista and later.
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       timeout /t 5
       <ul>
        <li>
         Waits for five seconds, allowing the user to cancel the waiting by pressing a key.
        </li>
       </ul>
      </li>
      <li>
       timeout /t 5 /nobreak
       <ul>
        <li>
         Waits for five seconds, ignoring user input other than Control + C.
        </li>
       </ul>
      </li>
      <li>
       timeout /t 5 /nobreak &gt;nul
       <ul>
        <li>
         As above, but with no output.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Workaround in Windows XP:
     </p>
     <ul>
      <li>
       ping -n 6 127.0.0.1 &gt;nul
       <ul>
        <li>
         Waits for five seconds; the number after -n is the number of seconds to wait plus 1.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Perl-based workaround in Windows XP, requiring Perl installed:
     </p>
     <ul>
      <li>
       perl -e "sleep 5"
       <ul>
        <li>
         Waits for 5 seconds.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Links:
     </p>
     <ul>
      <li>
       <a class="external text" href="http://ss64.com/nt/timeout.html" rel="nofollow">
        timeout at ss64.com
       </a>
      </li>
      <li>
       <a class="external text" href="http://technet.microsoft.com/en-us/library/cc754891%28v=ws.10%29.aspx" rel="nofollow">
        timeout at Microsoft
       </a>
      </li>
      <li>
       <a class="external text" href="http://stackoverflow.com/questions/735285/how-to-wait-in-a-batch-script" rel="nofollow">
        How to wait in a batch script?
       </a>
       at stackoverflow.com
      </li>
      <li>
       <a class="external text" href="http://stackoverflow.com/questions/166044/sleeping-in-a-batch-file" rel="nofollow">
        Sleeping in a batch file
       </a>
       at stackoverflow.com
      </li>
     </ul>
     <h3>
      <span class="mw-headline" id="TREE">
       TREE
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=121" title="Edit section: TREE">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Displays a tree of all subdirectories of the current directory to any level of recursion or depth. If used with /F switch, displays not only subdirectories but also files.
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       tree
      </li>
      <li>
       tree /f
       <ul>
        <li>
         Includes files in the listing, in addition to directories.
        </li>
       </ul>
      </li>
      <li>
       tree /f /a
       <ul>
        <li>
         As above, but uses 7-bit ASCII characters including "+", "-" and \" to draw the tree.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      A snippet of a tree using 8-bit ASCII characters:
     </p>
     <pre>
      winevt
   Logs
   TraceFormat
winrm
     </pre>
     <p>
      A snippet of a tree using 7-bit ASCII characters:
     </p>
     <pre>
      +---winevt
|   +---Logs
|   \---TraceFormat
+---winrm
     </pre>
     <p>
      Links:
     </p>
     <ul>
      <li>
       <a class="external text" href="http://www.microsoft.com/resources/documentation/windows/xp/all/proddocs/en-us/tree.mspx" rel="nofollow">
        tree at Microsoft
       </a>
      </li>
     </ul>
     <h3>
      <span class="mw-headline" id="WHERE">
       WHERE
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=122" title="Edit section: WHERE">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Displays the location of a file, searching in the current directory and in the PATH by default. Does some of the job of "which" command of some other operating systems.
     </p>
     <p>
      Available on Windows 2003, Windows Vista, Windows 7, and later; not available on Windows XP. An alternative to be used with Windows XP is in the examples below.
     </p>
     <p>
      Does not find internal commands, as there are no dot exe files for them to match.
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       where find
       <ul>
        <li>
         Outputs the location of the find command, possibly "C:\Windows\System32\find.exe".
        </li>
       </ul>
      </li>
      <li>
       for%i in (find.exe) do @echo%~$PATH:i
       <ul>
        <li>
         Outputs the location of "find.exe" on Windows XP. The name has to include ".exe", unlike with the where command.
        </li>
       </ul>
      </li>
      <li>
       where /r . Tasks*
       <ul>
        <li>
         Searches for files whose name matches "Task*" recursively from the current folder. Similar to "dir /b /s Tasks*"
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Links:
     </p>
     <ul>
      <li>
       <a class="external text" href="http://ss64.com/nt/where.html" rel="nofollow">
        where at ss64.com
       </a>
      </li>
      <li>
       <a class="external text" href="http://stackoverflow.com/questions/304319/is-there-an-equivalent-of-which-on-windows" rel="nofollow">
        Is there an equivalent of 'which' on windows?
       </a>
      </li>
     </ul>

     <h3>
      <span class="mw-headline" id="XCOPY">
       XCOPY
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Windows_Batch_Scripting&amp;action=edit&amp;section=124" title="Edit section: XCOPY">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Copies files and directories in a more advanced way than
      <a href="#COPY">
       COPY
      </a>
      , deprecated in Windows Vista and later. Type xcopy /? to learn more, including countless options.
     </p>
     <p>
      Examples:
     </p>
     <ul>
      <li>
       xcopy C:\Windows\system
       <ul>
        <li>
         Copies all files, but not files in nested folders, from the source folder ("C:\Windows\system") to the current folder.
        </li>
       </ul>
      </li>
      <li>
       xcopy /s /i C:\Windows\system C:\Windows-2\system
       <ul>
        <li>
         Copies all files and folders to any nesting depth (via "/s") from the source folder ("C:\Windows\system") to "C:\Windows-2\system", creating "Windows-2\system" if it does not exist (via "/i").
        </li>
       </ul>
      </li>
      <li>
       xcopy /s /i /d:09-01-2014 C:\Windows\system C:\Windows-2\system
       <ul>
        <li>
         As above, but copies only files changed on 1 September 2014 or later. Notice the use of the month-first convention even if you are on a non-US locale of Windows.
        </li>
       </ul>
      </li>
      <li>
       xcopy /L /s /i /d:09-01-2014 C:\Windows\system C:\Windows-2\system
       <ul>
        <li>
         As above, but in a test mode via /L (list-only, output-only, display-only). Thus, does not do any actual copying, merely lists what would be copied.
        </li>
       </ul>
      </li>
     </ul>
     <p>
      Links:
     </p>
     <ul>
      <li>
       <a class="external text" href="http://ss64.com/nt/xcopy.html" rel="nofollow">
        xcopy at ss64.com
       </a>
      </li>
      <li>
       <a class="external text" href="http://www.microsoft.com/resources/documentation/windows/xp/all/proddocs/en-us/xcopy.mspx" rel="nofollow">
        xcopy at Microsoft
       </a>
      </li>
     </ul>
