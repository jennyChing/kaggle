         </span>
        </p>
        <p>
         The following is a breakdown of how I developed the game using
         <span class="italic">
          cocos2d-x
         </span>
         . I used v2.2.x, but most of this should be applicable for v3.x. I wanted to write this as a lesson to any budding (or even experienced) game developers to show them how to start and what might be the major pitfalls along the way. It's a comprehensive guide but by no means covers every step of the design and development process.
        </p>

         <span class="italic">

        <p>
         <span class="bold">
          Understanding
          <span class="italic">
           cocos2d-x
          </span>
         </span>
        </p>
        <p>
         The
         <span class="italic">
          cocos2d-x
         </span>
         system is fairly complex, yet thankfully most of it should be running in the background without you needing to delve into it. A lot of the structure is taken from iOS development, so thankfully I could get my head around it quickly. If you come from an Android or
         <a class="" href="http://dev.windows.com/en-us?WT.mc_id=Blog_Developer_Vue+Sit_Awareness_Discover_Sep+14_WindowsPhone8_View+Internal+Website" id="" onclick="dcsSetVar('WT.z_ea_name','The definitive guide to developing with Cocos2d-x, by Cowboy Games - Body - Windows Phone 8 - Local','WT.z_ea_actionoffer','Vue-Sit','WT.z_ea_msproduct','Unknown','WT.z_ea_group','Articles','WT.z_ea_funnel','Awareness','WT.z_ea_targetcampaign','000XXX00000');MscomCustomEvent('ms.ea_offer','sit','ms.ea_action','vue','ms.ea_name','The definitive guide to developing with Cocos2d-x, by Cowboy Games - Body - Windows Phone 8 - Unknown - Awareness - Local','ms.cpgnguid','000XXX00000','ms.lang','en','ms.loc','gb');  " target="_blank" title="Windows Phone 8">
          Windows Phone 8
         </a>
         background, it probably won't take you too long either.
        </p>
        <p>
         To start an application running you'll come across the following classes:
        </p>
        <p>
         <span class="bold">
          CCDirector
         </span>
         - this is the big boss of your application. The director is responsible for passing along render calls, user interaction events, and much more. Most of the time you will only be using the director to present different scenes in your game.
        </p>
        <p>
         <span class="bold">
          CCEGLView
         </span>
         - the view is responsible for rendering the game on to the device screen. Other than the initial setup you probably won't ever need to worry about it.
        </p>
        <p>
         <span class="bold">
          CCFileUtils
         </span>
         - every operating system has its own way of storing and managing files. This is what you use if you want to perform any kind of file handling operations, although more often than not this is transparent to you. When the app starts you provide it with the
         <span class="italic">
          search paths
         </span>
         , which I'll explain shortly.
        </p>
        <p>
         <span class="bold">
          CCScene
         </span>
         - this is the base node that the director will display. I'll describe it in more detail in
         <span class="italic">
          node hierarchy
          <span class="bold">
           .
          </span>
         </span>
        </p>
        <p>
         <span class="bold">
          Design Resolution and Policy
         </span>
        </p>
        <p>
         When you start a
         <span class="italic">
          cocos2d-x
         </span>
         game, one of the first calls you need to make are:
        </p>
        <pre class="xml" id="" name="code" style="">
         CCEGLView::sharedOpenGLView()-&gt;setDesignResolutionSize(designResolutionSize.width, designResolutionSize.height, designPolicy);
        </pre>
        <p>
         Ready Steady Play was designed for an iPad retina screen, which means that all images and sizing were based on a
         <span class="italic">
          design resolution
         </span>
         of 1536x2048 pixels. On its own that doesn't really matter, because the important part is the
         <span class="italic">
          design policy
         </span>
         . The policy dictates how the game is displayed on screens of varying aspect ratios. The following are the definitions of the different design policies, taken from
         <span class="bold">
          CCEGLViewProtocol.h:
         </span>
        </p>
        <pre class="xml" id="" name="code" style="">
         enum ResolutionPolicy
        {
        // The entire application is visible in the specified area without trying to
        // preserve the original aspect ratio.
        // Distortion can occur, and the application may appear stretched or
        // compressed.
        kResolutionExactFit,
        // The entire application fills the specified area, without distortion but
        // possibly with some cropping, while maintaining the original aspect ratio of
        // the application.
        kResolutionNoBorder,
        // The entire application is visible in the specified area without distortion
        // while maintaining the original aspect ratio of the application. Borders can
        // appear on two sides of the application.
        kResolutionShowAll,
        // The application takes the height of the design resolution size and modifies
        // the width of the internal canvas so that it fits the aspect ratio of the
        // device no distortion will occur however you must make sure your application
        // works on different aspect ratios
        kResolutionFixedHeight,
        // The application takes the width of the design resolution size and modifies
        // the height of the internal canvas so that it fits the aspect ratio of the
        // device no distortion will occur however you must make sure your application
        // works on different aspect ratios
        kResolutionFixedWidth,
        kResolutionUnKnown,
        };
        </pre>
        <p>
         When designing for mobile phones of varying sizes, the first three options are not ideal. You don't want your game being squashed (
         <span class="italic">
          Exact Fit
         </span>
         ), cropped (
         <span class="italic">
          No Border
         </span>
         ) or having black borders (
         <span class="italic">
          Show All
         </span>
         ). We opted for
         <span class="italic">
          Fixed Height
         </span>
         as it is far less drastic having a wider screen than a shorter one, especially considering that we use both portrait and landscape modes. Take the second game in Ready Steady Play, Hobby
         <img alt="The definitive guide to developing with Cocos2d-x, by Cowboy Games" class="" src="file:///C:\Users\v-jodubi\AppData\Local\Temp\msohtmlclip1\01\clip_image001.png?v=0.0" style=""/>
         <img alt="The definitive guide to developing with Cocos2d-x, by Cowboy Games" class="" src="file:///C:\Users\v-jodubi\AppData\Local\Temp\msohtmlclip1\01\clip_image002.png?v=0.0" style=""/>
         Hurdles, as an example when played on an iPhone 5 screen:
        </p>
        <p>
         <img alt="The definitive guide to developing with Cocos2d-x, by Cowboy Games" class="" src="/en-gb/developers/images/articles/content/497-01.png?v=0.0" style=""/>
        </p>
        <p>
         Fixed Height
        </p>
        <p>
         <img alt="The definitive guide to developing with Cocos2d-x, by Cowboy Games" class="" src="/en-gb/developers/images/articles/content/497-02.png?v=0.0" style=""/>
        </p>
        <p>
         Fixed Width
        </p>
        <p>
         In
         <span class="italic">
          Fixed Width
         </span>
         mode it becomes almost impossible to fit anything on the screen, while in
         <span class="italic">
          Fixed Height
         </span>
         the game is slightly zoomed out. When compared between devices it might look a bit strange, but most players will just use one device and they won't know the difference.
        </p>
        <p>
         <span class="bold">
          Content Scale Factor
         </span>
        </p>
        <p>
         Once the design resolution and policy are set up, you need to tell
         <span class="italic">
          cocos2d-x
         </span>
         which image resources you intend to use, which will define the content scale factor. The standard setup for
         <span class="italic">
          cocos2d-x
         </span>
         is to have 3 different sizes for images: HDR (High Definition Retina), HD and SD (Standard Definition). As mentioned before we designed Ready Steady Play for a retina iPad screen, so our size definitions are:
        </p>
        <p>
         <span class="bold">
          HDR
         </span>
         : 1536x2048
        </p>
        <p>
         <span class="bold">
          HD
         </span>
         : 768x1024
        </p>
        <p>
         <span class="bold">
          SD
         </span>
         : 384x512
        </p>
        <p>
         When the app starts we choose the appropriate images resource folder, based on screen size, and we calculate the content scale factor by calling:
        </p>
        <pre class="xml" id="" name="code" style="">
         director-&gt;setContentScaleFactor(resource.size.width/designResolutionSize.width);
        </pre>
        <p>
         <span class="bold">
          Search Paths
         </span>
        </p>
        <p>
         The search paths are used by
         <span class="bold">
          CCFileUtils
         </span>
         to find the required resources, particularly images and sound files. The
         <span class="bold">
          Resources
         </span>
         folder (described earlier) should have one folder for each of your resource definitions (HDR/HD/SD), where each one has identical images resized accordingly. E.g. If we have an image whose path is
         <span class="bold">
          Resources/HDR/Button.png
         </span>
         whose size is 100x100 pixels, then there will be an image at
         <span class="bold">
          Resources/HD/Button.png
         </span>
         sized 50x50 and another at
         <span class="bold">
          Resources/SD/Button.png
         </span>
         sized 25x25.
        </p>
        <p>
         When the application begins you provide
         <span class="bold">
          CCFileUtils
         </span>
         with the paths to search in the
         <span class="bold">
          Resources
         </span>
         folder. Using the above example, if you provide it with
         <span class="bold">
          "HD"
         </span>
         and then make a call to:
        </p>
        <pre class="xml" id="" name="code" style="">
         CCTextureCache::sharedTextureCache()-&gt;addImage("Button.png")
        </pre>
        <p>
         ...and you will get back a texture using the image at
         <span class="bold">
          Resources/HD/Button.png
         </span>
         .
        </p>
        <p>
         <span class="bold">
          Node hierarchy
         </span>
        </p>
        <p>
         If you don't understand tree structures then you need to do some homework as I'm not going to go into much detail here. The general idea is that every element that is displayed on the screen is of type
         <span class="bold">
          CCNode
         </span>
         , and multiple nodes can be grouped together by them as 'children' of a parent node.
        </p>
        <p>
         Every
         <span class="bold">
          CCScene
         </span>
         object should have a
         <span class="bold">
          CCLayer
         </span>
         child, and all other nodes in the scene will be added on top of the layer. This again inherits from iOS by using the Model-View-Controller (MVC) design paradigm, where the scene is the
         <span class="italic">
          controller
         </span>
         that is responsible for communicating between the
         <span class="italic">
          models
         </span>
         (game logic) and
         <span class="italic">
          views
         </span>
         (layer and its child nodes).
        </p>
        <p>
         As mentioned before,
         <span class="bold">
          CCDirector
         </span>
         will present one
         <span class="bold">
          CCScene
         </span>
         object at a time. A scene represents different screens that will appear during the lifetime of your application. The scene navigation for Ready Steady Play looks like this:
        </p>
        <p>
         <img alt="The definitive guide to developing with Cocos2d-x, by Cowboy Games" class="" src="/en-gb/developers/images/articles/content/497-03.jpg?v=0.0" style=""/>
        </p>
        <p>
         <span class="bold">
          Sprite sheets
         </span>
        </p>
        <p>
         The basic idea of a sprite sheet is that, if you have an animation of 50 images, you don't want to load 50 files from disk, as disk reading/writing are
         <span class="italic">
          very
         </span>
         slow operations. Sprite sheets allow you to load one large
         <span class="bold">
          <span class="italic">
           texture
          </span>
         </span>
         (image) containing all the frames in the animation, along with a corresponding
         <span class="bold">
          <span class="italic">
           data
          </span>
         </span>
         file that defines the positions of each frame.
        </p>
        <p>
         Most of the animations in Ready Steady Play are done using sprite sheets. Each animation is created in
         <a class="" href="http://www.adobe.com/uk/products/aftereffects.html" id="" onclick="dcsSetVar('WT.z_ea_name','The definitive guide to developing with Cocos2d-x, by Cowboy Games - Body - After Effects - Local','WT.z_ea_actionoffer','Vue-Sit','WT.z_ea_msproduct','Unknown','WT.z_ea_group','Articles','WT.z_ea_funnel','Awareness','WT.z_ea_targetcampaign','000XXX00000');MscomCustomEvent('ms.ea_offer','sit','ms.ea_action','vue','ms.ea_name','The definitive guide to developing with Cocos2d-x, by Cowboy Games - Body - After Effects - Unknown - Awareness - Local','ms.cpgnguid','000XXX00000','ms.lang','en','ms.loc','gb');  " target="_blank" title="After Effects">
          After Effects
         </a>
         , exported as a sequence of PNG images at 25 frames per second, imported into a life-saving tool called
         <a class="" href="https://www.codeandweb.com/texturepacker" id="" onclick="dcsSetVar('WT.z_ea_name','The definitive guide to developing with Cocos2d-x, by Cowboy Games - Body - TexturePacker - Local','WT.z_ea_actionoffer','Vue-Sit','WT.z_ea_msproduct','Unknown','WT.z_ea_group','Articles','WT.z_ea_funnel','Awareness','WT.z_ea_targetcampaign','000XXX00000');MscomCustomEvent('ms.ea_offer','sit','ms.ea_action','vue','ms.ea_name','The definitive guide to developing with Cocos2d-x, by Cowboy Games - Body - TexturePacker - Unknown - Awareness - Local','ms.cpgnguid','000XXX00000','ms.lang','en','ms.loc','gb');  " target="_blank" title="TexturePacker">
          TexturePacker
         </a>
         , which subsequently exports it back out as corresponding
         <span class="bold">
          <span class="italic">
           data
          </span>
         </span>
         (
         <span class="bold">
          .plist
         </span>
         ) and
         <span class="bold">
          <span class="italic">
           texture
          </span>
         </span>
         (
         <span class="bold">
          .pvr.ccz

         ) files.

        <p>
         TexturePacker has predefined settings for
         <span class="italic">
          cocos2d
         </span>
         projects, where it will automatically export the sprite sheets in
         <span class="bold">
          HDR
         </span>
         ,
         <span class="bold">

          SD
         </span>
         sizes to correlate with the folders in your
         <span class="bold">

         folder. TexturePacker has great documentation, so I won't go into more detail here.

        <p>
         <span class="bold">
          Porting to iOS
         </span>
        </p>
        <p>
         From the early stages (about a month into development) I made sure that the game ran on both iOS and Android. Having the 2 projects running side-by-side meant that the code was being written with cross-platform compatibility in mind. For example I originally used arc4random for generating random numbers, until I tried to compile for Android and realised that I had to use drand48 instead.
        </p>
        <p>
         Certain parts of the program needed to be written specifically for different operating systems. The simplest example is localisation. While
         <span class="italic">
          cocos2d-x
         </span>
         will tell you the language the device is using, it doesn't provide any other localisation tools like number or date formatting. To handle localisation I created 3 files to define the
         <span class="bold">
          Localization
         </span>
         class:
        </p>
        <p>
         <span class="bold">
          Localization.h
         </span>
        </p>
        <p>
         <span class="bold">
          Localization.cpp
         </span>
        </p>
        <p>
         <span class="bold">
          Localization.mm
         </span>
        </p>
        <p>
         If you've ever written any C++ code then you probably know that
         <span class="bold">
          .h
         </span>
         is the interface file and
         <span class="bold">
          .cpp
         </span>
         is the C++ implementation file, but you may be wondering, "what is the
         <span class="bold">
          .mm
         </span>
         file?" It's actually an Objective-C++ implementation file. One of the most frustrating challenges of writing cross-platform code is cross-language interoperability. Communicating between C++ and Objective-C code is thankfully very easy. The iOS version of Ready Steady Play does not even compile
         <span class="bold">
          Localization.cpp
         </span>
         to define the
         <span class="bold">
          Localization
         </span>
         class, it only needs to use
         <span class="bold">
          Localization.mm
         </span>
         . The following is a modified excerpt from that file:
        </p>
        <pre class="xml" id="" name="code" style="">
         #include "Localization.h"
        #import &lt;Foundation/Foundation.h&gt;
        std::string Localization::localizedString(const char* key)
        {
        return [[[NSBundle mainBundle] localizedStringForKey:[NSString stringWithCString:key
        encoding:NSUTF8StringEncoding]
        value:@
        table:nil] UTF8String];
        }
        std::string Localization::formattedInteger(long number)
        {
        static NSNumberFormatter *formatter;
        static dispatch_once_t onceToken;
        dispatch_once(&amp;onceToken, ^{
        formatter = [[NSNumberFormatter alloc] init];
        [formatter setNumberStyle:NSNumberFormatterDecimalStyle];
        [formatter setMaximumFractionDigits:0];
        [formatter setMinimumIntegerDigits:1];
        [formatter setLocale:[NSLocale autoupdatingCurrentLocale]];
        });
        return [[formatter stringFromNumber:[NSNumber numberWithLong:number]] UTF8String];
        }
        </pre>
        <p>
         It might seem strange to see C++ and Objective-C intertwined like that, but it's all perfectly legal. The one pitfall that might catch you is that the
         <span class="bold">
          .h
         </span>
         file can't mix the two languages. It has to be either a C++ header
         <span class="italic">
          or
         </span>
         an Objective-C header.
        </p>
        <p>
         <span class="bold">
          Porting to Android
         </span>
        </p>
        <p>
         The first major headache I encountered with the Android port was the build script. When you (finally) set up the project in Eclipse and try to build it, the first thing it does is try to run
         <span class="bold">
          proj.android/build_native.sh
         </span>
         . The script uses the Android NDK to build the project, which references the Makefile
         <span class="bold">
          proj.android/jni/Android.mk
         </span>
         . This needs to be customised to reference all the source files that need to be compiled for the Android project. Initially it has the template source files, but adding files one-by-one is a massive headache. I opted for the
         <span class="italic">
          wildcard
         </span>
         approach, so I use this:
        </p>
        <p>
         LOCAL_SRC_FILES:= $(wildcard $(LOCAL_PATH)/../../Classes/*.cpp)
        </p>
        <p>
         LOCAL_SRC_FILES:= $(LOCAL_SRC_FILES:$(LOCAL_PATH)/%=%)
        </p>
        <p>
         Unfortunately this means that all my source files are thrown into a single folder, but since Xcode organises everything for me that wasn't a problem.
        </p>
        <p>
         The Makefile also needs to know of any external headers being referenced. E.g. I use
         <span class="italic">
          rapidjson
         </span>
         in parts of my code, so I had to add the following line:
        </p>
        <p>
         LOCAL_C_INCLUDES += $(LOCAL_PATH)/../../cocos2d-x/extensions/CocoStudio/Json/rapidjson
        </p>
        <p>
         The second major headache when porting to Android is JNI. As mentioned in the iOS section, interoperating between languages is tricky. How are you meant to convert a C string (const char*) to a Java string (java.lang.String)? This is where JNI comes in to play. JNI allows you to make calls from C++ to Java, but not the other way around, which means you may have to do polling to retrieve values asynchronously.
        </p>
        <p>
         Generally I tried to delve as little as possible into JNI, so to keep the JNI code simple I call static methods from one big Java file. It is by no means the recommended way to do it, but the less JNI code you have to write the better, believe me.
        </p>
        <p>
         The general data types that you'll want to use are fairly straightforward, such as jint, jlong, jboolean, jsize, etc. However the trickiest one of all is jstring. Consider the
         <span class="bold">
          Localization
         </span>
         class explained earlier.
        </p>
        <pre class="xml" id="" name="code" style="">
         #include "platform/android/jni/JniHelper.h"
        #include "JNIString.h"
        USING_NS_CC;
        using namespace std;
        string Localization::localizedString(const char* key)
        {
        JniMethodInfo t;
        string ret("");
        if (JniHelper::getStaticMethodInfo(
        t,
        "io/hoverstud/readysteadyplay/ReadySteadyPlay",
        "localizedStringJNI",
        "(Ljava/lang/String;)Ljava/lang/String;""))
        {
        jstring jKey = t.env-&gt;NewStringUTF(key);
        jstring jstr = (jstring)t.env-&gt;CallStaticObjectMethod(t.classID, t.methodID, jKey);
        jstringToString(t.env, jstr, &amp;ret);
        t.env-&gt;DeleteLocalRef(t.classID);
        t.env-&gt;DeleteLocalRef(jKey);
        t.env-&gt;DeleteLocalRef(jstr);
        }
        return ret;
        }
        </pre>
        <p>
         As you can see, it is easy to get a jstring object from a C string or as a return value, however converting a jstring back into a C string is not so easy. At first you might be tempted to use JniHelper::jstring2string, which calls the JNI method GetStringUTFChars. You must never use this method! What you really want is something that returns an array of
         <span class="bold">
          <span class="italic">
           UTF-8
          </span>
         </span>
         characters, yet for some ungodly reason GetStringUTFChars returns an array of
         <span class="bold">
          <span class="italic">
           modified UTF-8
          </span>
         </span>
         characters, which appear as jumbled letters when displayed in a label.
        </p>
        <p>
         Instead, I used my own jstringTostring method to call GetStringChars (not GetStringUTFChars), which returns an array of
         <span class="italic">
          UTF-16
         </span>
         characters that I pass through to an open source
         <a class="" href="http://www.codeproject.com/Articles/14637/UTF-With-C-in-a-Portable-Way" id="" onclick="dcsSetVar('WT.z_ea_name','The definitive guide to developing with Cocos2d-x, by Cowboy Games - Body - UTF-8 converter from Nemanja Trifunovic - Local','WT.z_ea_actionoffer','Vue-Sit','WT.z_ea_msproduct','Unknown','WT.z_ea_group','Articles','WT.z_ea_funnel','Awareness','WT.z_ea_targetcampaign','000XXX00000');MscomCustomEvent('ms.ea_offer','sit','ms.ea_action','vue','ms.ea_name','The definitive guide to developing with Cocos2d-x, by Cowboy Games - Body - UTF-8 converter from Nemanja Trifunovic - Unknown - Awareness - Local','ms.cpgnguid','000XXX00000','ms.lang','en','ms.loc','gb');  " target="_blank" title="UTF-8 converter from Nemanja Trifunovic">
          UTF-8 converter from Nemanja Trifunovic
         </a>
         to convert it to a valid C string.
        </p>
        <p>
         <span class="bold">
          Porting to Windows Phone 8
         </span>
        </p>
        <p>
         Support for Windows Phone 8 was introduced in
         <span class="italic">
          cocos2d-x
         </span>
         v2.2.3, but Ready Steady Play was built using v2.2. Therefore, the first stage of porting to WP8 was upgrading
         <span class="italic">
          cocos2d-x
         </span>
         . The problem was that I forked
         <span class="italic">
          cocos2d-x
         </span>
         so that we could adapt the code to suit our game, so I couldn't simply swap to the latest git tag; I had to manually merge the code. This proved to be a painfully slow process, since I didn't want to either
         <span class="bold">
          <span class="italic">
           git merge
          </span>
         </span>
         or
         <span class="bold">
          <span class="italic">
           rebase
          </span>
         </span>
         the code and risk losing my changes. For any of you working off v2.2.3 or later, be very thankful.
        </p>
        <p>
         The second stage was getting the project set up. This was when I discovered the project creation script
         <span class="bold">
          cocos2d-x/tools/project-creator/create_project.py
         </span>
         . This script creates a project in
         <span class="bold">
          cocos2d-x/projects
         </span>
         , which is not where I wanted it to be. Unfortunately, when I moved the
         <span class="bold">
          proj.wp8-xaml
         </span>
         to the root folder, all the search paths were out of sync. I've never worked with a Visual Studio project, especially one as interweaved as
         <span class="italic">
          cocos2d-x
         </span>
         , so finding exactly which search paths needed to be updated and how was a laborious task, to say the least.
        </p>
        <p>
         After combing through the project for a few days I eventually got the game running. Our test device in the office is a Nokia Lumia 520, which is recommended as the lowest-end device for testing. The biggest challenge with this device is the memory limit of 150MB. I hope you don't think that that's a lot, because it's not.
        </p>
        <p>
         Remember how I mentioned that
         <span class="italic">
          cocos2d-x
         </span>
         images have HDR, HD and SD versions? Forget about it. WP8 gets SD (sprite sheets) only. Are you planning on using music and sound with stereo channels? Forget about it. Mono-only for WP8 (and Android too, if you want the game to run smoothly).
        </p>
        <p>
         Thankfully WP8 allows you to check the device's memory capability. You simply need to call
        </p>
        <pre class="xml" id="" name="code" style="">
         ((long)DeviceExtendedProperties.GetValue("ApplicationWorkingSetLimit") &lt;= 188743680L)
        </pre>
        <p>
         to check whether or not it is a low-end device, in which case you can limit the resources.
        </p>
        <p>
         Once the game was running without expending memory, the next stage was to get the localisation working, which meant once again getting into cross-language interoperability. WP8 XAML projects use "Managed C++", where you can use C#-esque objects in your C++ code. Let's go back to our good friend
         <span class="bold">
          Localization
         </span>
         .
        </p>
        <pre class="xml" id="" name="code" style="">
         #include "LocalizationComponent.h"
        #include "CCWinRTUtils.h"
        static inline std::string winRTStringToString(Platform::String^ str)
        {
        std::string ret(cocos2d::CCUnicodeToUtf8(str-&gt;Data()));
        // Replace \r\n with \n
        size_t pos;
        while ((pos = ret.find('\r\n')) != std::string::npos)
        ret.replace(pos, 2, '\n');
        return ret;
        }
        std::string Localization::localizedString(const char* key)
        {
        // Convert the key to a wstring
        std::wstring keyString = cocos2d::CCUtf8ToUnicode(key);
        // Convert the key to a Platform::String
        Platform::String^ str = ref new Platform::String(keyString.data(), keyString.length());
        // Get localized string
        str = PhoneDirect3DXamlAppComponent::LocalizationComponent::Instance
        -&gt;GetLocalizedString(str);
        return winRTStringToString(str);
        }
        std::string Localization::formattedInteger(long number)
        {
        // Get localized string
        Platform::String^ str = PhoneDirect3DXamlAppComponent::LocalizationComponent::Instance
        -&gt;GetFormattedNumber(number);
        return winRTStringToString(str);
        }
        </pre>
        <p>
         There are a couple of important things to note here:
        </p>
        <p>
         <span class="bold">
          CCWinRTUtils.h
         </span>
         provides you with the basic methods you need for converting between C# strings and C strings, which is great. Unfortunately they add a '\r' before a '\n', so you need to remove that, otherwise little white boxes appear before a new line in your
         <span class="bold">
          CCLabel
         </span>
         s.
        </p>
        <p>
         Like with Objective-C++ vs C++, the header files cannot be a mix of the two languages. This means you need to use an intermediary Managed C++ class to communicate between C++ and C#, which in the above example is PhoneDirect3DXamlAppComponent::LocalizationComponent.
        </p>
        <p>
         The last hurdle was an unexpected one. I started preparing the screenshots for submission to the Windows Phone Store, which is time-consuming but wonderfully easy thanks to the WP8 simulator, and everything seemed fine until I got to Asian languages. Suddenly all my labels were printing out boxes instead of characters.
        </p>
        <p>
         I thought at first there was a problem with my conversion of C# strings to C strings, but that wasn't the case. Eventually I dug into
         <span class="bold">
          platform/winrt/CCFreeTypeFont.cpp
         </span>
         and discovered the problem. You see, when iOS and Android try to draw a glyph - what we normally call a 'character' in an alphabet - that can't be found in the supplied font, they fall back to a system font that contains the glyph. The WP8 OS normally does this too, but
         <span class="italic">
          cocos2d-x
         </span>
         creates labels for WP8 using
         <span class="bold">
          CCFreeTypeFont
         </span>
         , which doesn't perform the fallback.
        </p>
        <p>
         At first I thought to add a font file that contained all the required characters, but such a font file is approximately 22MB (there are a
         <span class="italic">
          lot
         </span>
         of Unicode characters out there) and not something that should be added manually.
        </p>
        <p>
         After plenty of research, I discovered the following to solve my problem:
        </p>
        <ol>
         <li>
          <span class="bold">
           CCFreeTypeFont
          </span>
          has a method called loadSystemFont, which returns nullptr for WP8 by default but can actually be used. I found the solution on
          <a class="" href="http://www.cocos2d-x.org/forums/6/topics/37224" id="" onclick="dcsSetVar('WT.z_ea_name','The definitive guide to developing with Cocos2d-x, by Cowboy Games - Body - this cocos2d-x forum topic - Local','WT.z_ea_actionoffer','Vue-Sit','WT.z_ea_msproduct','Unknown','WT.z_ea_group','Articles','WT.z_ea_funnel','Awareness','WT.z_ea_targetcampaign','000XXX00000');MscomCustomEvent('ms.ea_offer','sit','ms.ea_action','vue','ms.ea_name','The definitive guide to developing with Cocos2d-x, by Cowboy Games - Body - this cocos2d-x forum topic - Unknown - Awareness - Local','ms.cpgnguid','000XXX00000','ms.lang','en','ms.loc','gb');  " target="_blank" title="this cocos2d-x forum topic">
           this cocos2d-x forum topic
          </a>
          .
         </li>
         <li>
          The available system fonts for WP8 can be found
          <a class="" href="http://msdn.microsoft.com/en-US/library/windows/apps/hh202920(v=vs.105).aspx#BKMK_SupportedUIFontsinWindowsPhone" id="" onclick="dcsSetVar('WT.z_ea_name','The definitive guide to developing with Cocos2d-x, by Cowboy Games - Body - here - Local','WT.z_ea_actionoffer','Vue-Sit','WT.z_ea_msproduct','Unknown','WT.z_ea_group','Articles','WT.z_ea_funnel','Awareness','WT.z_ea_targetcampaign','000XXX00000');MscomCustomEvent('ms.ea_offer','sit','ms.ea_action','vue','ms.ea_name','The definitive guide to developing with Cocos2d-x, by Cowboy Games - Body - here - Unknown - Awareness - Local','ms.cpgnguid','000XXX00000','ms.lang','en','ms.loc','gb');  " target="_blank" title="here">
           here
          </a>
          . Passing the right font name to your code is not straightforward. E.g. If you want to load the Yu Gothic font (for Japanese) then you pass to loadSystemFont("YuGothic"), but if you want to load Microsoft Mhei (for Traditional Chinese) you need to pass in "MSMhei".
         </li>
         <li>
          The
          <span class="italic">
           FreeType
          </span>
          library informs the code when a glyph is not found. At this point of the code I set up the fallback mechanism to go through the pre-loaded system fonts until it finds one with the required glyph.
         </li>
        </ol>
        <p>
         <span class="bold">
          Publishing
         </span>
        </p>
        <p>
         To date we have released Ready Steady Play (at a premium price of 1.49GBP / $1.99USD) to iOS, Android and Windows Phone 8, with at least some kind of a feature on the Apple, Google Play and WP8 stores.
        </p>
        <p>
         <span class="bold">
          iOS App Store
         </span>
        </p>
        <p>
         For the iOS App Store we were listed in the
         <span class="bold">
          <span class="italic">
           Best New Games
          </span>
         </span>
         section, which is manually curated in each country for iPhone and iPad separately. E.g. in China we were listed for both iPhone and iPad, but in the US only for iPad. This makes a significant difference, but we were pretty thrilled either way as it resulted in a lot of downloads.
        </p>
        <p>
         <span class="bold">
          Google Play
         </span>
        </p>
        <p>
         A friend of mine, who is a big fan of the game and has been featured himself by Apple and Google, e-mailed his contacts to consider us for a feature. Shortly after we got an e-mail from the Google Play editorial team asking us to make some changes to fix the game. Once they were happy with the update they featured Ready Steady Play in their
         <span class="bold">
          <span class="italic">
           New + Updated Games
          </span>
         </span>
         section.
        </p>
        <p>
         <span class="bold">
          Windows Phone Store
         </span>
        </p>
        <p>
         Microsoft have been phenomenal in helping to get our game published to their store and featured. We didn't initially intend to port the game for Windows Phone, but a fortuitous meeting with their technical evangelist, Lee Stott, put us on the path. We were showing off Ready Steady Play at the
         <a class="" href="http://www.developconference.com/" id="" onclick="dcsSetVar('WT.z_ea_name','The definitive guide to developing with Cocos2d-x, by Cowboy Games - Body - Develop games conference - Local','WT.z_ea_actionoffer','Vue-Sit','WT.z_ea_msproduct','Unknown','WT.z_ea_group','Articles','WT.z_ea_funnel','Awareness','WT.z_ea_targetcampaign','000XXX00000');MscomCustomEvent('ms.ea_offer','sit','ms.ea_action','vue','ms.ea_name','The definitive guide to developing with Cocos2d-x, by Cowboy Games - Body - Develop games conference - Unknown - Awareness - Local','ms.cpgnguid','000XXX00000','ms.lang','en','ms.loc','gb');  " target="_blank" title="Develop games conference">
          Develop games conference
         </a>
         in Brighton when we met him. When we told him that the game was made using
         <span class="italic">
          cocos2d-x
         </span>
         he gave us the links that we needed to port the game over, and provided technical assistance throughout the procedure.
        </p>
        <p>
         In addition to the technical support, he also put us in touch with Microsoft's marketing department, who gave us invaluable advice and also offered to nominate the game once it was released.
        </p>
        <p>
         The Windows Phone Store features are not like the others. They do daily listings and for set groups of countries. How they choose these countries is seemingly random, but the results are very rewarding.
        </p>
        <p>
         <span class="bold">
          Summary
         </span>
        </p>
        <p>
         It took about 7 months to create the first release of Ready Steady Play.
         <span class="italic">
          cocos2d-x
         </span>
         has proven to be a great tool to use for developing a 2D mobile game, and it would take a lot to convince me to use something else. Being an open source project means that developers can cater it to their needs, and the project has hundreds of contributors helping to make it better with each iteration
        </p>
        <p>
         I really hope that this article helps anybody interested in game development to dive right in to using
         <span class="italic">
          cocos2d-x
         </span>
         . Remember that you're not alone in using this tool or any of the others out there (and there are many), with each having its own fast-growing community ready and willing to help you along the way.
        </p>
