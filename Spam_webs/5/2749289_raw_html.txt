     <p>
      An
      <b>
       interpreted language
      </b>
      is a
      <a href="/wiki/Programming_language" title="Programming language">
       programming language
      </a>
      for which most of its implementations execute instructions directly, without previously
      <a href="/wiki/Compiler" title="Compiler">
       compiling
      </a>
      a
      <a href="/wiki/Computer_program" title="Computer program">
       program
      </a>
      into
      <a class="mw-redirect" href="/wiki/Machine_language" title="Machine language">
       machine-language
      </a>
      instructions. The
      <a href="/wiki/Interpreter_(computing)" title="Interpreter (computing)">
       interpreter
      </a>
      executes the program directly, translating each statement into a sequence of one or more
      <a class="mw-redirect" href="/wiki/Subroutines" title="Subroutines">
       subroutines
      </a>
      already compiled into machine code.
     </p>
     <p>
      The terms
      <i>
       interpreted language
      </i>
      and
      <i>
       <a href="/wiki/Compiled_language" title="Compiled language">
        compiled language
       </a>
      </i>
      are not well defined because, in theory, any programming language can be either interpreted or compiled. In modern programing language implementation it is increasingly popular for a platform to provide both options.
     </p>
     <p>
      Interpreted languages can also be contrasted with
      <a class="mw-redirect" href="/wiki/Machine_language" title="Machine language">
       machine languages
      </a>
      . Functionally, both execution and interpretation mean the same thing  fetching the next instruction/statement from the program and executing it. Although interpreted
      <a href="/wiki/Bytecode" title="Bytecode">
       bytecode
      </a>
      is additionally identical to machine code in form and has an
      <a href="/wiki/Assembly_language" title="Assembly language">
       assembler
      </a>
      representation, the term "interpreted" is practically reserved for "software processed" languages (by
      <a href="/wiki/Virtual_machine" title="Virtual machine">
       virtual machine
      </a>
      or
      <a href="/wiki/Emulator" title="Emulator">
       emulator
      </a>
      ) on top of the native (i.e. hardware) processor.
     </p>
     <p>
      In principle, programs in many languages may be
      <a href="/wiki/Compiler" title="Compiler">
       compiled
      </a>
      or interpreted, emulated or executed natively, so this designation is applied solely based on common implementation practice, rather than representing an essential property of a language. Akin to
      <a href="/wiki/Microcode" title="Microcode">
       processor microcoding
      </a>
      , many interpreters internally rely on
      <a href="/wiki/Just-in-time_compilation" title="Just-in-time compilation">
       just-in-time compilation
      </a>
      .
     </p>
     <p>
      Avoiding compilation, interpreted programs are easier to evolve during both development
      <a href="#Advantages_of_interpreting_a_language">
       and execution (where they can morph themselves)
      </a>
      . On the other hand, since compilation implies translation into more machine-friendly format, interpreted programs run more slowly and less efficiently (i.e. waste considerably more energy). This is especially true for higher-level
      <a href="/wiki/Scripting_language" title="Scripting language">
       scripting languages
      </a>
      , whose statements are complex to analyze compared to machine instruction.
     </p>
     <p>
      Many languages have been implemented using both compilers and interpreters, including
      <a href="/wiki/BASIC" title="BASIC">
       BASIC
      </a>
      ,
      <a href="/wiki/C_(programming_language)" title="C (programming language)">
       C
      </a>
      ,
      <a href="/wiki/Lisp_(programming_language)" title="Lisp (programming language)">
       Lisp
      </a>
      ,
      <a href="/wiki/Pascal_(programming_language)" title="Pascal (programming language)">
       Pascal
      </a>
      , and
      <a href="/wiki/Python_(programming_language)" title="Python (programming language)">
       Python
      </a>
      .
      <a href="/wiki/Java_(programming_language)" title="Java (programming language)">
       Java
      </a>
      and
      <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">
       C#
      </a>
      are compiled into bytecode, the
      <a href="/wiki/Interpreted_language#Languages_usually_compiled_to_a_bytecode" title="Interpreted language">
       virtual machine-friendly interpreted language
      </a>
      . Lisp implementations can freely mix interpreted and compiled code.
     </p>

     <h2>
      <span class="mw-headline" id="Historical_background_of_interpreted.2Fcompiled">
       Historical background of interpreted/compiled
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Interpreted_language&amp;action=edit&amp;section=1" title="Edit section: Historical background of interpreted/compiled">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      In the early days of computing, language design was heavily influenced by the decision to use compiling or interpreting as a mode of execution. For example, some compiled languages require that programs must explicitly state the data-type of a
      <a class="mw-redirect" href="/wiki/Variable_(programming)" title="Variable (programming)">
       variable
      </a>
      at the time it is declared or first used while some interpreted languages take advantage of the dynamic aspects of interpreting to make such declarations unnecessary. For example,
      <a href="/wiki/Smalltalk" title="Smalltalk">
       Smalltalk
      </a>
      (1980), which was designed to be interpreted at run-time, allows generic objects to dynamically interact with each other.
     </p>
     <p>
      Initially, interpreted languages were compiled line-by-line; that is, each line was compiled as it was about to be executed, and if a loop or subroutine caused certain lines to be executed multiple times, they would be recompiled every time. This has become much less common. Most so-called interpreted languages use an
      <a class="mw-redirect" href="/wiki/Intermediate_representation" title="Intermediate representation">
       intermediate representation
      </a>
      , which combines compiling and interpreting. In this case, a compiler may output some form of
      <a href="/wiki/Bytecode" title="Bytecode">
       bytecode
      </a>
      or
      <a href="/wiki/Threaded_code" title="Threaded code">
       threaded code
      </a>
      , which is then executed by a bytecode interpreter.
     </p>
     <p>
      Examples include:
     </p>
     <ul>
      <li>
       <a href="/wiki/Java_(programming_language)" title="Java (programming language)">
        Java
       </a>
      </li>
      <li>
       <a href="/wiki/Python_(programming_language)" title="Python (programming language)">
        Python
       </a>
      </li>
      <li>
       <a href="/wiki/Ruby_(programming_language)" title="Ruby (programming language)">
        Ruby
       </a>
       (similarly, it uses an
       <a href="/wiki/Abstract_syntax_tree" title="Abstract syntax tree">
        abstract syntax tree
       </a>
       as intermediate representation)
      </li>
      <li>
       <a href="/wiki/Forth_(programming_language)" title="Forth (programming language)">
        Forth
       </a>
      </li>
     </ul>
     <p>
      The intermediate representation can be compiled once and for all (as in Java), each time before execution (as in Perl or Ruby), or each time a change in the source is detected before execution (as in Python).
     </p>
     <h2>
      <span class="mw-headline" id="Advantages_of_interpreting_a_language">
       Advantages of interpreting a language
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Interpreted_language&amp;action=edit&amp;section=2" title="Edit section: Advantages of interpreting a language">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      Interpreting a language gives implementations some additional flexibility over compiled implementations. Features that are often easier to implement in interpreters than in compilers include (but are not limited to):
     </p>
     <ul>
      <li>
       <a class="mw-redirect" href="/wiki/Platform_independent" title="Platform independent">
        platform independence
       </a>
       (Java's byte code, for example)
      </li>
      <li>
       <a class="mw-redirect" href="/wiki/Reflection_(computer_science)" title="Reflection (computer science)">
        reflection
       </a>
       and reflective use of the evaluator (e.g. a first-order
       <a href="/wiki/Eval" title="Eval">
        eval
       </a>
       function)
      </li>
      <li>
       <a class="mw-redirect" href="/wiki/Dynamic_typing" title="Dynamic typing">
        dynamic typing
       </a>
      </li>
      <li>
       smaller executable program size (since implementations have flexibility to choose the instruction code)
      </li>
      <li>
       <a class="mw-redirect" href="/wiki/Scope_(programming)#Dynamic_scoping" title="Scope (programming)">
        dynamic scoping
       </a>
      </li>
     </ul>
     <h2>
      <span class="mw-headline" id="Disadvantages_of_interpreted_languages">
       Disadvantages of interpreted languages
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Interpreted_language&amp;action=edit&amp;section=3" title="Edit section: Disadvantages of interpreted languages">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      The main disadvantage of interpreting is a much slower speed of program execution compared to direct
      <a href="/wiki/Machine_code" title="Machine code">
       machine code
      </a>
      execution on the host
      <a class="mw-redirect" href="/wiki/CPU" title="CPU">
       CPU
      </a>
      . A technique used to improve performance is
      <a href="/wiki/Just-in-time_compilation" title="Just-in-time compilation">
       just-in-time compilation
      </a>
      which converts frequently executed sequences of interpreted instruction to host machine code.
     </p>

     <h3>
      <span class="mw-headline" id="Languages_usually_compiled_to_a_bytecode">
       Languages usually compiled to a bytecode
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Interpreted_language&amp;action=edit&amp;section=5" title="Edit section: Languages usually compiled to a bytecode">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Many interpreted languages are first compiled to
      <a href="/wiki/Bytecode" title="Bytecode">
       bytecode
      </a>
      , which is normally interpreted by
      <a href="/wiki/Virtual_machine" title="Virtual machine">
       virtual machine
      </a>
      exploiting some
      <a href="/wiki/Just-in-time_compilation" title="Just-in-time compilation">
       just-in-time compilation
      </a>
      of bytecode to
      <a class="mw-redirect" href="/wiki/Native_code" title="Native code">
       native code
      </a>
      . However, sometimes, bytecode can also be compiled to a native binary using an
      <a class="mw-redirect" href="/wiki/AOT_compiler" title="AOT compiler">
       AOT compiler
      </a>
      ) or executed natively, by hardware processor.
     </p>
     <ul>
      <li>
       <a href="/wiki/Java_(programming_language)" title="Java (programming language)">
        Java
       </a>
       (is compiled into Java bytecode to be interpreted by
       <a class="mw-redirect" href="/wiki/JVM" title="JVM">
        JVM
       </a>
       )
       <ul>
        <li>
         <a href="/wiki/Groovy_(programming_language)" title="Groovy (programming language)">
          Groovy
         </a>
        </li>
        <li>
         <a href="/wiki/Join_Java" title="Join Java">
          Join Java
         </a>
        </li>
        <li>
         <a class="mw-redirect" href="/wiki/ColdFusion" title="ColdFusion">
          ColdFusion
         </a>

        <li>
         <a href="/wiki/Scala_(programming_language)" title="Scala (programming language)">
          Scala
         </a>
        </li>
       </ul>
      </li>
      <li>
       <a href="/wiki/Lua_(programming_language)" title="Lua (programming language)">
        Lua
       </a>

        .NET Framework
       </a>
       languages (translated to bytecode, called

        CIL
       </a>
       ).
