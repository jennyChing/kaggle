       <header>
        The ~200 Line Linux Kernel Patch That Does Wonders
       </header>
       <div class="author">
        Written by
        <a href="http://www.michaellarabel.com/">
         Michael Larabel
        </a>
        in
        <a href="?page=category&amp;item=Software">
         Software
        </a>
        on 16 November 2010.
        <strong>
         Page 2 of 2
        </strong>
        .
        <a href="/forums/node/22011">
         224 Comments
        </a>
       </div>
       <div class="content">
        <p>
         Below is the video of the Linux desktop when running the Linux Git kernel and the patch in question was applied but the auto-group scheduler was disabled via its sysfs interface.
        </p>
        <p align="center">
         <object height="340" width="560">
          <param name="movie" value="http://www.youtube.com/v/uk70SeGA7pg?fs=1&amp;hl=en_US&amp;rel=0&amp;color1=0x3a3a3a&amp;color2=0x999999">
          </param>
          <param name="allowFullScreen" value="true">
          </param>
          <param name="allowscriptaccess" value="always">
          </param>
          <embed allowfullscreen="true" allowscriptaccess="always" height="340" src="http://www.youtube.com/v/uk70SeGA7pg?fs=1&amp;hl=en_US&amp;rel=0&amp;color1=0x3a3a3a&amp;color2=0x999999" type="application/x-shockwave-flash" width="560">
          </embed>
         </object>
        </p>
        <p>
         As you can see, the experience when compiling the Linux kernel with so many jobs is rather troubling to the Linux desktop experience. At no point in the video was the 1080p sample video paused, but that was just where the current mainline Linux kernel is at with 2.6.37. There was also some stuttering with glxgears and some responsiveness elsewhere. This is even with all of the Linux 2.6.37 kernel improvements up to today. If recording a video of an older kernel release, the experience is even more horrific! Now let's see what happens when enabling the patch's new scheduler code.
        </p>
        <p align="center">
         <object height="340" width="560">
          <param name="movie" value="http://www.youtube.com/v/prxInRdaNfc?fs=1&amp;hl=en_US&amp;rel=0&amp;color1=0x3a3a3a&amp;color2=0x999999">
          </param>
          <param name="allowFullScreen" value="true">
          </param>
          <param name="allowscriptaccess" value="always">
          </param>
          <embed allowfullscreen="true" allowscriptaccess="always" height="340" src="http://www.youtube.com/v/prxInRdaNfc?fs=1&amp;hl=en_US&amp;rel=0&amp;color1=0x3a3a3a&amp;color2=0x999999" type="application/x-shockwave-flash" width="560">
          </embed>
         </object>
        </p>
        <p>
         It is truly a night and day difference. The 1080p Ogg video now played smoothly a majority of the time when still compiling the Linux kernel with 64 jobs. Glxgears was also better and the window movements and desktop interactivity was far better. When compiling the Linux kernel with 128 jobs or other workloads that apply even greater strain, the results are even more dramatic, but it is not great for a video demonstration; the first video recorded under greater strained made the "before look" appear as like a still photograph.
        </p>
        <p>
         This patch truly does wonders to the Linux kernel in improving the desktop responsiveness / interactivity. This patch with improving his web-browsing experience and more, impresses even Linus Torvalds. The merge window is now closed for the
         <a href="http://www.phoronix.com/scan.php?page=search&amp;q=Linux+2.6.37">
          Linux 2.6.37 kernel
         </a>
         , but this should be an exciting improvement that should be found in the
         <a href="http://www.phoronix.com/scan.php?page=search&amp;q=Linux+2.6.38">
          Linux 2.6.38 kernel
         </a>
         and at least keep the people happy waiting around for
         <a href="http://www.phoronix.com/scan.php?page=news_item&amp;px=ODY5OQ">
          Reiser4 / Open-Source VIA Graphics / Radeon HD 6000 Series DRM
         </a>
         to arrive in the mainline kernel.
        </p>
        <p>
         In related work while waiting for your kernel to build, you may be interested in reading the recent
         <a href="http://www.phoronix.com/vr.php?view=15409">
          five years of Linux kernel benchmarks
         </a>
         and
         <a href="http://www.phoronix.com/vr.php?view=15422">
          GCC vs. LLVM compiler benchmarks
         </a>
         with GCC 4.2/4.3/4.4/4.5/4.6, LLVM-GCC, Clang on LLVM 2.8, and DragonEgg on LLVM 2.8 with GCC 4.5.1.
        </p>
